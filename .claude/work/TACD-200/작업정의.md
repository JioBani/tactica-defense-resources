# TACD-200: [전장 - 소환수] - 배치/이동/스왑

## 작업 내용

소환수(Defender)의 배치/이동/스왑 시스템을 구현한다. 대기석(WaitingArea)과 전장(BattleArea) 간 드래그&드롭 기반의 배치 조작을 완성하고, 배치 제한 규칙을 적용한다.

### 하위 일감

| Jira | 제목 | 설명 |
|------|------|------|
| TACD-203 | 배치/회수 | 대기석 → 전장 배치, 전장 → 대기석 회수 |
| TACD-204 | 이동/스왑 | 전장 내 빈 칸 이동, 전장 내 소환수 간 스왑 |
| TACD-205 | 배치 제한 | 배치 상한 초과 시 불가, 전투 중 배치/이동 불가 |

### 발견된 문제점

#### 1. Defender.cs 이벤트 구독 해제 버그
```csharp
// 현재 (잘못됨)
private void OnDisable()
{
    draggable.OnDragEnd -= NotifyDragStart;  // ❌ OnDragStart여야 함
    draggable.OnDragEnd += NotifyDragEnd;    // ❌ -= 여야 함
}
```

#### 2. DefenderSideSell 배치 상한 체크 로직 결함
- **문제**: `IsDefenderLimitExceeded()`가 모든 드롭에서 실행됨
- **영향**: 전장→전장 이동이나 스왑 시에도 상한 체크에 걸림 (실제로는 인원 변동 없음)
- **케이스 분석**:
  - 대기석→빈 전장 셀: 인원 +1 → 상한 체크 **필요** ✅
  - 대기석→점유된 전장 셀 (스왑): 인원 변동 없음 → 상한 체크 **불필요** ❌ 차단됨
  - 전장→전장 빈 셀: 인원 변동 없음 → 상한 체크 **불필요** ❌ 차단됨
  - 전장→전장 점유 셀 (스왑): 인원 변동 없음 → 상한 체크 **불필요** ❌ 차단됨

#### 3. WaitingArea 페이즈 제한 없음
- **문제**: WaitingArea.CanAccept()가 항상 true 반환
- **영향**: 전투 중에도 전장→대기석 회수가 가능 (기획: 전투 중 배치/이동 불가)

#### 4. DefenderSideSell 네이밍/위치 문제
- "Sell" 폴더에 배치 로직이 있어 역할이 모호
- 현재 이름 `DefenderSideSell`이 실제 역할(전장 셀)과 맞지 않음

## 완료 정의

### TACD-203 (배치/회수)
- [ ] 대기석 소환수를 빈 전장 셀로 드래그하면 배치된다
- [ ] 대기석 소환수를 점유된 전장 셀로 드래그하면, 기존 소환수와 스왑된다 (기존→대기석, 신규→전장)
- [ ] 전장 소환수를 대기석으로 드래그하면 회수된다
- [ ] 배치/회수 시 Placement 상태가 정확히 갱신된다
- [ ] Defender.cs의 이벤트 구독 해제 버그가 수정된다

### TACD-204 (이동/스왑)
- [ ] 전장 소환수를 전장 내 빈 셀로 드래그하면 이동한다
- [ ] 전장 소환수를 전장 내 다른 소환수 위로 드래그하면 스왑된다
- [ ] 이동/스왑 시 배치 상한 체크에 걸리지 않는다

### TACD-205 (배치 제한)
- [ ] 배치 상한 도달 시, 대기석→빈 전장 셀 배치가 차단된다
- [ ] 배치 상한 도달 시에도, 스왑/이동은 정상 동작한다
- [ ] 전투 중(Combat 페이즈) 모든 배치/이동/스왑이 차단된다 (전장↔전장, 전장↔대기석 모두)
