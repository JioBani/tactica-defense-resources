# 구현 이슈 분해

에픽 티켓 번호: $ARGUMENTS

## 참조 문서

먼저 `Claude/work/decompose/$ARGUMENTS/` 디렉토리에 기존 기록 파일이 있는지 확인한다.
- 파일이 있으면: 기록 파일을 읽고, 마지막 진행 상태부터 이어서 수행한다.
- 파일이 없으면: 아래 참조 문서를 읽고 처음부터 수행한다.

### 문서 읽기 규칙

- 현재 컨텍스트에서 이미 읽은 문서는 다시 읽지 않는다.
- 컨텍스트 컴팩트 발생 시에만 필요한 문서를 다시 읽는다.
- 다시 읽을 때는 필요한 구간만 offset/limit으로 읽는다.

아래 문서를 **필요한 시점에** 읽는다. 선행 로딩은 Phase A에 필요한 문서만 수행한다.

### Phase A 시작 시 읽는 문서
1. `Document/프로젝트 관리/구현 이슈 분해 방법론.md` - 분해 방법론 (워크플로우)

### Phase B 시작 시 읽는 문서
2. `Document/프로젝트 관리/업무 분류 체계.md` - 업무 분류 정의 (5-4단계에서 사용)

### Jira 등록 직전에 읽는 문서
3. `Document/프로젝트 관리/구현 이슈 작성 요령.md` - 이슈 작성 형식 (4단계, 8단계에서 사용)
4. `.claude/docs/jira.md` - Jira 등록 지침 (4단계, 8단계에서 사용)

## 작업 파일

단계별 결과를 아래 파일에 기록하며 진행한다:

```
Claude/work/decompose/{에픽 티켓 번호}/
├── 분해결과.md                    # Phase A: Epic → Task 결과
├── {Task 티켓 번호}.md            # Phase B: Task Sub-task 분해 (Jira 등록 후 생성)
├── {Task 티켓 번호}.md
├── ...
└── 교차검증.md                    # 교차 검증: 플랫 리스트 + 대조 결과
```

### 기록 규칙

- **단계 완료 시마다** 결과를 파일에 기록한다
- 사용자 검증 후 수정된 내용은 해당 단계에 덮어쓴다
- 컨텍스트 컴팩트 발생 시, 해당 파일을 읽어 이전 결과를 복원한다

### 분해결과.md 형식 (Phase A)

```markdown
# 구현 이슈 분해: {에픽 티켓 번호} - {에픽 제목}

> 현재 Phase: A / B
> 현재 작업: {상태 설명}

---

## 1단계: 기획서 파악 ✅

**참조 기획서**: {경로}
**핵심 요약**: ...

## 2단계: 에픽 경계 확인 ✅

**대상 에픽**: {티켓 번호} - {제목}
**인접 에픽**: ...
**경계 판단**: ...

## 3단계: Task 도출 ✅

### 상위 동작 트리
(트리 구조)

### 상호작용 분석 (상위)
(대상 분류별 열거 + 주요 연결 지점 개요)

### Task 목록
(경계 판단 결과)

## 4단계: Task Jira 등록 ✅

| # | Task | Jira | Sub-task 분해 |
|---|------|------|--------------|
| 1 | {기능명} | TACD-XXX | ✅ / 🔄 / - |
| 2 | {기능명} | TACD-XXX | - |
```

### Task별 파일 형식 (Phase B)

파일명: `{Task 티켓 번호}.md` (4단계에서 Jira 등록 후 생성)

```markdown
# {Task 티켓 번호}: {Task 제목}

> 상태: 진행중 / 사용자 검증 대기 / 완료

---

## 5단계: Sub-task 도출 {✅/🔄}

### 5-1. 동작 트리 상세화
(말단 노드까지 분해한 트리)

### 5-2. 상호작용 분석 (상세)
(대상별 × 내용별 매트릭스)

### 5-3. 보조 관점 체크
(데이터/속성, UI/정보 표시, 기반 작업)

### 5-4. 업무 분류 및 분류
(전체 항목 + 분류 결과)

## 6단계: 검증 {✅/🔄}

## 7단계: 이슈 목록 정리 {✅/🔄}

(구현 이슈 분해 방법론.md 7단계 형식으로 작성)

## 8단계: Sub-task Jira 등록 {✅/🔄}

(등록된 이슈 번호 기록)
```

### 교차검증.md 형식

```markdown
# 교차 검증: {에픽 티켓 번호} - {에픽 제목}

## 플랫 리스트

| # | 항목 | 출처 (동작/상호작용/보조관점) | 업무 분류 |
|---|------|---------------------------|----------|
| 1 | ... | 동작 | 구현 |
| 2 | ... | 상호작용 | 구현 |

## 대조 결과

### 매핑된 항목
| # | 플랫 리스트 항목 | 매핑 대상 (Task > Sub-task) |
|---|-----------------|---------------------------|
| 1 | ... | T1 > 1-1 |

### 누락 후보 — Task 수준
(기존 Task 어디에도 속하지 않는 항목 → Phase A에서 Task 자체를 놓친 것)
| # | 플랫 리스트 항목 | 판정 (추가/기각) | 사유 |
|---|-----------------|----------------|------|
| 1 | ... | 추가 / 기각 | ... |

### 누락 후보 — Sub-task 수준
(기존 Task에 속할 수 있지만 Sub-task로 도출되지 않은 항목)
| # | 플랫 리스트 항목 | 대상 Task | 판정 (추가/기각) | 사유 |
|---|-----------------|----------|----------------|------|
| 1 | ... | T1 | 추가 / 기각 | ... |
```

## 워크플로우 요약

### Phase A: Epic → Task
1+2단계: 기획서 파악 + 에픽 경계 확인을 **병렬로** 진행한다 (기획서 Read와 Jira 에픽 조회는 독립적)
3단계: Task 도출 → 상위 동작 트리 + 상호작용 분석(상위) + 경계 판단 → **사용자 검증**
4단계: Task Jira 등록

### Phase B: Task별 Sub-task 도출 (병렬) + 교차 검증 (병렬)

Phase B 시작 시, 모든 Task의 5단계 도출과 교차 검증을 **서브에이전트로 병렬 실행**한다.

```
Phase B 시작
  ├─ 서브에이전트 1: Task 1 (5단계 도출)  ─┐
  ├─ 서브에이전트 2: Task 2 (5단계 도출)  ─┤
  ├─ 서브에이전트 3: Task 3 (5단계 도출)  ─┤→ 결과 수집
  ├─ ...                                 ─┤
  └─ 서브에이전트 N: 교차 검증            ─┘
                                           ↓
                                    메인: 6~7단계 검증/정리 (사용자 검증)
                                           ↓
                                    메인: 8단계 Jira 등록
```

**서브에이전트** (Task별 병렬):
5단계: Sub-task 도출 → 동작 트리 상세화 + 상호작용 상세 + 보조 관점 체크 + 업무 분류

**메인 에이전트** (서브에이전트 결과 수집 후 순차):
6단계: 검증 → 누락/중복 확인 → **사용자 검증**
7단계: 이슈 목록 정리 → 형식에 맞춰 정리하여 보여주기 → **사용자 검증**
8단계: Sub-task Jira 등록

**교차 검증** (병렬 서브에이전트):
- 모든 분해 축(동작 트리 + 상호작용 + 보조 관점)을 한번에 적용 → 플랫 리스트 도출
- 6~7단계 검증 완료 후, 플랫 리스트 vs Task/Sub-task 구조 대조 → 누락 후보 도출
- **사용자 검증**: 누락 후보를 추가할지 기각할지 결정

### 서브에이전트 실행 규칙

#### Task별 서브에이전트

Phase B 시작 시, 각 Task에 대해 Agent 도구로 **병렬 실행**한다. 각 서브에이전트에 아래를 전달한다:

- **기획서**: 원문 파일 경로를 전달하여 서브에이전트가 **직접 Read** (요약 전달 금지 — 뉘앙스 손실 방지)
- **에픽 경계 확인 결과**: Phase A 2단계 결과를 텍스트로 전달
- **해당 Task 정보**: Task 이름, 범위, 상위 동작 트리에서의 위치
- **방법론**: 5단계 수행 지시 + 상호작용 분해 축 정의 + 보조 관점 정의를 텍스트로 전달
- **산출물 원칙**:
  - 코드베이스 탐색은 맥락 파악을 위해 **허용**한다.
  - 단, 산출물(동작 트리, 상호작용 매트릭스, Sub-task 항목)에는 **기능/동작 수준**으로만 작성한다.
  - 클래스명, 메서드명, 코드 패턴, 파일 경로, 구현 용어(이벤트 발행, DTO, 콜백 등) 를 산출물에 포함하지 않는다.
  - 플레이어/기획자가 읽어도 이해할 수 있는 **게임 동작** 수준으로 작성한다.
  - 나쁜 예: "LifeCrystalAttacker가 DefenseLine 태그를 감지하여 OnRoundLoseEventDto를 발행한다"
  - 나쁜 예: "침략자가 라인 끝에 도달하면 전투 패배 이벤트가 발생한다" (이벤트는 구현 용어)
  - 좋은 예: "침략자가 라인 끝에 도달하면 전투에서 패배한다"
  - **예외**: 작업 자체가 기술적 설계인 경우(데이터 스키마, 프레임워크, 데이터 구조 등)는 기술 용어가 곧 작업 내용이므로 허용한다.
    - 예: "소환술사 데이터 스키마 제작", "상태 프레임워크 구현", "시너지 데이터 구조 설계"

서브에이전트는 결과를 `Claude/work/decompose/{에픽 티켓 번호}/{Task 티켓 번호}.md`에 기록한다.

#### 교차 검증 서브에이전트

Task별 서브에이전트와 동시에 Agent 도구로 **병렬 실행**한다.

- **기획서**: 원문 파일 경로를 전달하여 서브에이전트가 **직접 Read**
- **에픽 경계 확인 결과**: Phase A 2단계 결과를 텍스트로 전달
- **전달하지 않는 정보**: Phase A의 Task 목록, Phase B의 Sub-task 결과 (독립성 보장)
- **산출물 원칙**: Task별 서브에이전트와 동일 (코드 탐색 허용, 산출물은 기능/동작 수준만)

서브에이전트는 결과를 `Claude/work/decompose/{에픽 티켓 번호}/교차검증.md`에 기록한다.

## 주의사항

- 각 검증 단계에서 반드시 사용자 확인을 받은 후 다음 단계로 진행한다.
- 사용자가 수정을 요청하면 해당 단계를 수정한 뒤 다시 검증을 받는다.
- 기획서는 `Document/기획서/` 폴더에서 찾는다.
- 이슈 작성 시 `구현 이슈 작성 요령.md`의 형식을 따른다.
- Phase B 진행 시, `분해결과.md`와 해당 Task 파일만 읽으면 된다.
- **Jira 등록 전 코드 레벨 필터링**: 서브에이전트 산출물에 클래스명, 메서드명, 파일 경로 등 코드 레벨 내용이 남아 있으면, Jira 등록 전에 기능/동작 수준으로 변환한다.
