# UR 추출 방법론 (User Requirement Elicitation Methodology)

> 기획서(산문·의도 중심)로부터 **사용자 요구사항(UR)** 을 **누락 없이** 길어내고, 결정 안 된 지점을 **미결로 식별**하기 위한 방법론이다.
> 한 대상(세부 기능 또는 기능)을 입력으로, 요구사항 명세서에 들어갈 UR 집합 + 미결 대장을 산출한다.
> 규칙 출처: [[기획 운영 컨벤션]] §2(게임 기능 계층·완전성)·§3(도출 체인). 정본 도메인 자료: `docs/기획서`.

---

## 0. 왜 이 방법론이 필요한가 (전제)

이 방법론으로 뽑은 UR은 그대로 작업 요구사항이 된다. 그래서:

- **UR 누락 = 구현 누락.** 추출의 *완전성*이 생명이다.
- 기획서엔 `잠정`·`추후`·`약 N라운드마다`·`낮은 확률` 같은 **미결**이 가득하다. 방법론은 이를 *없앨* 수 없다 — **명시적으로 식별·수집**해 빠뜨리지도, 환각으로 메우지도 않게 해야 한다.

그래서 이 방법론이 보장하는 두 가지:

1. **누락 0** — 산문에서 모든 UR을 빠짐없이 길어낸다. "다 뽑았다"를 *구성적으로 증명*하는 장치를 둔다(§6 역방향 소스-절 커버리지). 그냥 읽고 나열하면 반드시 샌다.
2. **미결 식별** — 결정 안 된 지점을 UR로 위장(환각)하지 않고, OQ(Open Question)로 못박아 추적한다. 종료 조건은 "미결 0"이 **아니라** "미결 식별·수집 완료"다.

> **완전성의 척도 = 작업 요구사항 도출 무중단.** UR을 보고 작업 요구사항(구현/디자인 작업)을 뽑을 때 *"이게 어떻게 동작해야 하지?"* 에서 멈추는 지점이 **0**이어야 한다. 작업 정의 시 남아도 되는 "열린 것"은 **순수 기술적 고민(구현 방식·아키텍처·성능)·순수 디자인적 고민(비주얼·UX 선택)** 뿐이다. 그 외에 작업을 막는 구간 = "요구사항이 동작을 안 알려줘서 막힘" = 상류에서 잡혔어야 할 **미결/도약/누락**이다. 이 "작업 도출 무중단(기술·디자인 재량 제외)" 이 §3 판정·§9 자기 점검의 최종 기준이다.

> 이 방법론은 기존 `UR 상세 분해 방법론`(분해 3축 + 교차검증 + 선택지 테스트)을 출발점으로, 요구공학 표준 기법(IEEE 830 / ISO 29148, Frame Semantics, Requirements Smells, VORD, RTM backward traceability)으로 보강하고, 새 컨벤션(UR-NNN 전역 평면 · 명세서 .md 반영 · 미결 수집)에 접합한 것이다. 근거 대응표는 [부록 A](#부록-a--리서치-근거-대응표).

## 1. UR 표현 수준 (반드시 지킴)

산출은 **사용자(플레이어/운영자) 시점**의 요구사항이다. "왜/무엇"만 다루고 "어떻게(수단)"는 다루지 않는다.

- **도메인 언어** — 사용자가 기능을 인식하는 어휘(소환수·성급·합성·승급·정비 페이즈·마나…). 시스템 인코딩 식별자(클래스/DB/API 필드/자산 ID)·코드 식별자(변수·메서드)는 등장하지 않는다.
- **동작 항목("~한다")** — 항목은 대상이 실제로 수행하는 *행동*이어야 한다. "스탯 시스템"·"데미지 계산"·"~이벤트 발생" 같은 시스템·메커니즘·구현 표현은 동작 항목으로 쓰지 않는다.
  - ✅ "유닛이 다운된다", "침략자가 라인 끝에 도달하면 소환술사와 교전한다"
  - ❌ "스탯 시스템", "전투 패배 이벤트가 발생한다"
- **검증 가능** — 각 UR은 충족/미충족을 판정할 수 있는 완료 조건이어야 한다. 추상 표현은 검증 가능한 행동/결과로 다시 쓴다.
- **단일성(한 문장 한 가지)** — 한 UR은 하나의 검증 가능한 사실만. 접속사로 두 요구를 묶지 않는다.
- **"하지 않는 것"은 항목으로 두지 않는다** — 사용자에게 노출되는 결과만.
- **현재 동작은 요구사항이 아니다** — 베이스라인 묘사일 뿐.
- **이해관계자 가독성** — 명세서는 프로젝트를 깊이 모르는 이해관계자(기획자 등)도 읽는다. 도메인 용어·구체(위치·수치·형태·타이밍)는 유지하되 **문장은 쉽게**(한 문장 한 가지, 능동·구체, 불필요한 한자어·추상어 배제). 단 쉬운 말이 구체나 검증 가능성을 흐리면 정확성이 우선이다.
- **UR 적격 필터(작업 도출 가능성) 〔F-008〕** — *이 항목으로 구현 또는 디자인 작업 요구사항을 도출할 수 있는가?* 순수 **컨셉·아트**(예: "소환수는 고양이다")·**밸런스 수치**·**콘텐츠 인스턴스**(개별 캐릭터 분포 등)는 UR이 아니다 → §6 비요구사항으로 태깅(컨셉/아트 또는 콘텐츠·밸런스 위임). UR의 *구체*가 밸런스·콘텐츠로 위임되면 본문에 **"(상세는 밸런스/콘텐츠 기획)"** 을 명시한다(작업자가 "여긴 콘텐츠가 채운다"를 알게).
- **과잉 표현 금지 〔F-009〕** — 근본 사실을 *최소 형태*로 쓴다. (a) **"없는 것"**(레어도 구분 없음 등)은 UR이 아니다 — 적지 않는다. (b) 한 UR에 **두 사실을 묶지 않는다**(단일성 재확인 — 묶으면 작업 도출이 막힌다). (c) 기획서의 **표·데이터로 표현되는 것을 문장으로 장황하게 풀지 않는다**(예: 역할군 분포는 "분포 표 중 하나를 따른다"로 족하다 — 표는 데이터로 cross-ref). ※ 이것은 *장황·중복서술* 금지이지, 인접 기능에 걸친 요구사항의 교차 표현(seam/cross-ref, §1.5)을 막는 게 아니다 — 그건 누락 방지를 위해 허용된다.
- **자기설명적 표기 — 약칭·식별자 단독 금지 〔F-013〕** — 이 문서는 *사람(기획자)이 읽고 피드백하는* 산출물이다. **사람이 위에서 아래로 한 번 읽어 이해**되어야 한다(다른 문서·번호를 해독하거나 다른 줄로 점프하지 않고). 식별자(기능/세부기능 id·OQ 번호·기획서 절 번호)는 *문장에 포인터로 덧붙이되, 문장의 의미를 대체하지 않는다*. 함축은 작성자의 머릿속에서만 풀리고 독자에겐 안 풀린다.
  - **출처는 문서 제목과 함께** — 문서 번호만으론 무엇을 가리키는지 모른다. `07 §4.5.4` ❌ → `「07 게임 시스템 기획서」 §4.5.4` ✅. (정본 문서: 00 기획서 · 04 세계관 기획서 · 07 게임 시스템 기획서 · 08-1 컨텐츠 레벨디자인 · 08-2 육성 스케일링 · 09 의사결정 기록.)
  - **seam/cross-ref는 완전한 문장으로**(§1.5) — 타입 접두·계층 id는 *기계 태그*로 남기되, **그 줄을 사람이 읽어 무슨 일이 어디서 일어나고 무엇이 저쪽 소관인지** 알 수 있어야 한다. `seam → 전장/라운드: 다음 정비 진입·라운드 진행` ❌(라벨 조각 — 무슨 뜻인지 사람은 모름) → "전투에서 적이 전멸하면 이 라운드의 전투가 끝나고 다음 정비 페이즈로 넘어간다 — 라운드 진행·페이즈 전환의 상세 규정은 `전장/라운드` 소관이다(seam)." ✅
  - **OQ 번호 인라인 참조엔 한 줄 설명을 붙인다** — 번호는 포인터일 뿐, *무엇이 열려 있는지*는 문장이 말해야 한다. `(표시 화면 내 위치·형태는 OQ-WV-01)` ❌ → "(이 정보를 화면 어디에 어떤 형태로 표시할지는 아직 미정 — OQ-WV-01)" ✅.
  - 줄임표·기호 나열(`A·B·C`)로 의미를 압축하지 않는다 — 사람이 각 항목을 풀어 읽을 수 있게 문장/절로 쓴다. (단 §1 표·데이터 cross-ref 처럼 *원본이 표인* 경우는 예외.)

> 코드 grounding이 필요하면 직접 read 하지 말고 코드 탐색에 **도메인 모드**로 위임한다. 산출에 코드 용어가 유입되지 않게 한다.

## 1.5 기능 귀속 — UR을 어느 세부기능에 부착하는가 〔보강〕

**용어 — "노드" 폐기 〔F-015〕.** 이 방법론은 "노드"라는 뭉뚱그린 말을 쓰지 않는다. 게임 기능 계층은 **콘텐츠 > 기능 > 세부기능** 이며, **UR은 *항상* 세부기능에 부착한다**(기능·콘텐츠에 UR이 직접 붙는 일은 없다 — 요구사항이 있으면 세부기능을 만들어 거기 붙인다). 분석 *대상*은 기능 또는 세부기능일 수 있고(대상이 기능이면 그 세부기능들을 열거해 각각에 부착), 참조(귀속·seam·cross-ref)는 대상 기능 또는 세부기능을 가리킨다. (CLI 식별자 `ur nodes`·`--node`·`nodes.jsonl` 는 도구 명칭이라 그대로 둔다.)

**조직화는 분석 결과지 미결이 아니다 〔F-016〕.** UR을 어느 세부기능에 붙일지(귀속), 어떻게 묶을지(그룹), 세부기능을 새로 만들지·이름을 바꿀지·경계를 어디로 둘지 — 이 **요구사항 *조직화*는 분석가가 사실로부터 *추론해 결정하고 근거와 함께 제시*하는 산출물**이다. **OQ(미결)로 올려 사람에게 떠넘기지 않는다.** 미결(OQ)은 *기획적(게임 설계) 미결* — 게임의 사실·룰이 안 정해진 것 — 만을 가리킨다(§8-2). "세부기능을 새로 만들까요?"·"이 세부기능을 폐지할까요?" 류는 미결이 아니라 분석가가 답을 내야 할 결과다. 사람은 그 결과를 *교정*할 수 있으나 그건 게이트가 아니다. (조직화 결과는 초안에 "구조 제안"으로 근거와 함께 적어 제시한다.)

**1차 기준 = 도메인 적용 범위.** UR은 그 사실이 *실제로 성립하는 범위*의 세부기능에 부착한다.

- 여러 종류에 **공통 성립** → 공통(유닛) 세부기능. 예: "모든 유닛은 동일 능력치 체계를 공유한다" → `전장/유닛/능력치`.
- **특정 종류에만 성립**하는 고유값·고유 행동 → 그 종류 기능의 세부기능. 예: "소환술사는 성급이 없다" → `전장/소환술사/*`.

**보조 신호 — 범위 판단을 덮어쓰지 못한다.**

- *기획서 절 위치*: 어느 상위 절 아래 서술됐는지는 단서일 뿐이다. **공통 절(예: §1 유닛 능력치) 안에 특정 종류 특례가 중첩 서술된 경우 특히 주의** — 절 위치에 끌려 종류 특례를 공통(유닛) 세부기능에 남기지 않는다.
- *더 구체적인 전용 세부기능의 존재*: 전용 세부기능이 있으면 거기로 끌리되, 그 역시 보조 신호일 뿐 범위 판단을 대신하지 않는다.

**이음새(seam) 항목.** "A는 공통 체계를 공유한다"류 공통↔종류 경계 사실은 한쪽 세부기능에만 **1회** 부착하고 반대쪽은 cross-ref(중복 금지).

**외부귀속 시 실행 seam 보존 〔F-006〕.** 어떤 사건을 다른 기능/세부기능으로 외부귀속할 때, *현재 세부기능의 실행 흐름이 그 사건을 지나가면* 끊긴 지점에 **seam UR**을 남긴다 — "여기서 X가 일어난다(상세 규정은 Y에)". cross-ref 메모만으로는 *현재 세부기능만 보고 작업할 때* 실행이 이어지지 않는다. 예: 유닛 라이프사이클에서 "라운드 종료 시 아군 유닛이 회복된다(회복 상세는 소환수·소환술사 쪽)"는 유닛 세부기능에도 seam UR로 남겨야 라운드 종료 전이가 닫힌다. (외부귀속 ≠ 실행 흐름에서 삭제.)

**참조 표기 컨벤션(typed·계층 타깃) 〔F-006〕.** 다른 기능/세부기능을 가리키는 모든 참조는 **타입 접두 + 불변 기능/세부기능 id**로 적는다(드래프트-로컬 id 금지 — 그건 발번 전이라 불안정):
- `seam → 전장/소환터미널` (실행 핸드오프: "여기서 일어나고 상세는 저기")
- `cross-ref: 전장/유닛/스킬` (정본은 저쪽, 여기선 중복 UR화 안 함)
- `외부귀속: 전장/시너지` (도메인 소관이 저쪽)
- `함의: <같은 사실의 다른 표현>`
이 타입 접두 + id는 **기계 태그**다(추후 _db 스키마 relations 필드로 승격할 때 마이그레이션을 기계적으로 만든다). **태그만 달랑 두지 말고 반드시 사람이 읽어 완결되는 문장과 함께 쓴다 〔F-013〕** — 무슨 일이 여기서 일어나고 무엇이 저쪽 소관인지 그 줄만 읽고 알 수 있어야 한다. `seam → 전장/라운드: 다음 정비 진입·라운드 진행`(라벨 조각, 독자 해독 불가) ❌ → "전투에서 적이 전멸하면 이 라운드가 끝나고 다음 정비 페이즈로 넘어간다 — 라운드 진행·페이즈 전환의 상세는 `전장/라운드` 소관(seam)" ✅. (지금은 _db 스키마 변경 없이 *표기 컨벤션*만 — 기록된 UR이 없어 링크 대상이 아직 없으므로.)

**누락 방지 > 중복 회피 〔F-001 단서〕.** "1회 부착·중복 금지"는 *같은 사실의 정본을 한 곳에* 두라는 것이지, 인접 기능에 걸친 요구사항을 양쪽에 표현하는 것(seam/cross-ref)까지 막는 게 아니다. **요구사항이 반드시 유일할 필요는 없다 — 누락을 막는 게 최우선이다.** 한 세부기능만 보고 작업할 때 흐름이 끊기면, 중복을 감수하고 교차 표현(seam)을 남긴다.

**한 절 = 공통 + 특례 혼재.** 한 기획서 절이 공통 사실과 종류 특례를 함께 서술하면 **문장 단위로 쪼개** 공통은 공통(유닛) 세부기능, 특례는 종류별 세부기능으로 분리한다.

**속성 vs 메커니즘 분리 · cross-cutting 시스템 정본 인지 〔F-012〕.** 한 개념이 *A의 속성*이면서 동시에 *시스템 B의 메커니즘*일 수 있다. 이때 **보유/소유 사실**은 A에, **작동 메커니즘**은 B(해당 시스템 기능)에 둔다.
- 예: "소환술사가 심상 효과를 보유한다"(보유 → `전장/소환술사`) vs "심상이 임계치 2/4/6으로 발동하고 카운트로 적용된다"(메커니즘 → `전장/시너지`). 후자를 소환술사에 정본 UR화하면 *시너지의 핵심 요구사항을 엉뚱한 곳에 둔 오귀속*이다(건강한 seam 중복과 다르다 — 줄여야 한다).
- **판별 전 전체 기능 계층을 인지하라** — `node tools/ur/ur.mjs nodes` 로 어떤 시스템/기능이 존재하는지(특히 cross-cutting: 시너지·전술지원 등) 먼저 훑는다. "이 메커니즘의 정본 소관 시스템이 따로 있는가?"를 묻고, 있으면 그 시스템 기능에 두고 로컬엔 seam/cross-ref만 남긴다.

**인접 기능/세부기능의 완료 정의 참고 〔F-012〕.** 분석·귀속 시 인접/관련 기능·세부기능에 **이미 기록된 UR(완료 정의)이 있으면** 참고한다 — `ur list --node <id>` 또는 `ur query`. (a) 같은 사실을 중복 *정본화*하지 말고 cross-ref, (b) 경계를 그 정의에 맞춰 오귀속을 막는다. 아직 _db에 기록된 UR이 없으면(분석 초기) 기능 계층 + 기획서 시스템 구조로 *소관만* 판별한다. 드래프트 단계 형제 산출물은 *같은 멤버의 연속 작업일 때만* 리드로 참고(비교 런에선 금지). ※ 그래도 **누락 방지 > 중복** — 소관이 애매하면 일단 seam으로 남기고(중복은 사람이 검토로 줄일 수 있으나 누락은 알아채기 어렵다), 오귀속만 인접 참고로 줄인다.

> 안티패턴(실제 발생): 소환술사 고유 특례(성급 없음·기준값·기본 사거리)가 §1.1.1(유닛 능력치 하위)에 서술됐다는 이유로 `전장/유닛/능력치`에 남는 것. 절 위치가 도메인 범위 판단을 덮어쓴 오귀속이다.

## 2. 분해 4축 — 항목을 망라 도출한다

네 축을 조합해 후보 항목을 빠짐없이 끌어낸다. 각 후보 항목에는 **태그**를 단다:

- 💡 **신규 도출** — 기획서 규칙에서 *유일하게* 따라오는 함의
- ✅ **기존 충족** — 기획서에 이미 정의·명시됨
- ❓ **기획 누락** — 기획서에 답 없음 → 미결(OQ)

### 축 1. 동작 트리 — 대상이 내부적으로 무엇을 하는가

- 대상의 **라이프사이클(생성 → 행동 → 소멸)** 에서 출발해 동작 흐름을 트리로, **말단 항목까지** 상세화한다.
- 시작 질문: "이 대상은 게임에서 실제로 무엇을 하는가?"
- 기획서 목차를 그대로 트리로 변환하지 않는다 — 축은 **동작 흐름**이다.
- **대칭성 체크** — 기획서에서 병렬/대칭으로 기술된 개념(A와 B / N개 종류 / 3가지 모드 / "~ 등")이 트리에 **모두** 대응되는지 확인한다. "대기·공격·다운 *등*" 같은 미열거를 닫는 장치.

### 축 2. 상호작용 — 대상이 외부와 무엇을 주고받는가

5개 대상 분류를 빠짐없이 열거하고 각각 분석한다(도메인 언어로).

| 대상 분류 | 탐색 질문 |
|---|---|
| 인접 기능 | 이 기능과 직접 맞물리는 다른 기능은? |
| 상위 흐름 | 이 기능은 언제 시작/종료되고, 누가 그걸 결정하는가? |
| 공유 시스템 | 이 기능이 공통으로 쓰는 것(자원·수정자·상태 등)은? |
| 데이터 | 이 기능의 규칙/수치를 정하는 설정·정의는? |
| 플레이어 | 플레이어의 어떤 행동이 이 기능을 변화시키는가? |

각 대상에 대해 **트리거(시작/중단) / 데이터 흐름(필요·산출) / 효과(상태 변경)** 를 ←받는·→보내는 방향으로 본다. 표현은 도메인 언어("이벤트 발행"❌ → "~을 알린다"✅).

### 축 3. 보조 관점 — 두 축에서 직접 안 나오는 것

| 관점 | 질문 |
|---|---|
| 데이터/속성 | 이 동작에 필요한 속성/설정값은? |
| UI/정보 표시 | 표시 가능한 정보를 먼저 항목화한 뒤, 각 항목에 §3 의미역 게이트를 적용한다(특히 WHERE-2·WHEN). |
| 기반 작업 | 동작과 무관하게 선행되어야 할 것은? |

### 축 4. 예외·경계 — 정상 경로 밖에서 무엇이 일어나는가 〔보강〕

IEEE 830 / ISO 29148은 완전성을 *"무효 입력에 대한 응답을 포함한"* 모든 요구사항으로 정의한다. 동작 트리(축 1)는 정상 경로 중심이라 이 영역이 구조적으로 덜 탐색된다. 다음을 빠짐없이 묻는다:

| 경계 종류 | 탐색 질문 (예) |
|---|---|
| **실패/거부 경로** | 조건을 못 채우면? (마나 부족·대기석 만석·합성 재료 부족 시 동작은?) |
| **무효/금지 입력** | 허용 안 되는 조작을 하면? (전투 중 배치 시도, 잠금 중 재스캔) |
| **자원 고갈/상한** | 자원이 0이거나 상한에 닿으면? (배치 상한 도달, 패배 횟수 소진) |
| **경계값** | 최소/최대/0/동시 발생 시? (0마나, 첫/마지막 라운드, 동일 위치 충돌) |
| **빈/없음 상태** | 대상이 0개일 때? (배치 0, 풀에 후보 없음) |

> 각 경계에서 기획서에 답이 없으면 → ❓(OQ). "실패하면 어떻게 되는가"가 명시되지 않은 것은 매우 흔한 누락 지점이다.

> **각 예외·거부는 *두 가지*를 닫아야 한다 〔F-017〕.** ① **시스템 상태 결과** — 무엇이 바뀌나/안 바뀌나(조작이 무시되나, 자원이 그대로인가). ② **사용자 피드백(거부·실패 표현)** — *플레이어가 그 거부를 어떻게 아는가*: 무시하고 아무 반응 없음인가 / 안내를 표시하나(무엇을: "마나 부족"·"배치 상한 도달") / 대안을 유도하나(교체·상한 증가). 이건 §3 UI 의미역(WHAT·WHERE·HOW·WHEN)으로 닫는다.
> - **거부의 표현이 비면 작업 도출이 막힌다(§0)** — "합성 실패"만 적고 *플레이어가 왜 안 됐는지 어떻게 아는지*를 비우면 UI 작업을 정의할 수 없다. 모든 사용자 동작 트리의 *거부 가지*까지 표현이 채워져야 한다.
> - 트리거 예: **2성이 아닌 것을 합성 시도 · 마나 부족 상태로 소환 시도 · 배치 상한 도달 후 추가 배치 · 잠금 중 재스캔 · 대기석 만석에서 소환**. 각각 "거부됨"이 아니라 *거부를 플레이어에게 어떻게 알리는가*까지 UR(또는 OQ).
> - **요구 수준 = 동작까지, 비주얼은 디자인 재량.** "상한 도달을 알리는 안내를 표시한다"는 UR(닫아야 함). 그 안내의 색·문구 톤·애니메이션은 디자인 재량(§0). *알리는가/무엇을 알리는가*가 미정이면 → OQ `[결정 필요]`.

---

### 마무리. 기능 레벨 실행 스레드 — 세부기능을 엮어 실제로 돌려본다 〔보강·F-005〕

축 1~4는 각 세부기능을 *고립*해서 분해한다. 그러면 **세부기능 사이를 넘나드는 실행 전이**(데미지계산 → HP 감소 → 다운 → 승리 판정)와 **실행 단위 디테일**(첫 공격 시점·틱 타이밍·동시 발생 우선순위)이 구조적으로 빠진다. 실제 검토에서 막혔던 지점이 거의 다 여기였다.

대상이 기능이면(또는 세부기능이라도 다른 세부기능과 맞물리면), 분해 후 **라이프사이클을 세부기능에 걸쳐 한 번 실행**해본다(coarse — 검토자의 틱 단위 시뮬과 달리 "명백한 흐름"만):

1. 정상 한 판을 시작→끝으로 따라가며, 한 세부기능의 결과가 다음 세부기능으로 넘어가는 **전이마다 규정 UR이 있는지** 본다. 없으면 도약 후보 → UR 추가 또는 OQ. (상호작용 축의 "인접 기능"을 **같은 기능 안의 인접 세부기능**에도 적용하는 것이다.)
2. **실행 단위까지 내려간다** — "초당 N회"가 아니라 *첫 타격 시점·다음 공격 예약*, "최근접 타겟"이 아니라 *동거리 동점 시 우선순위*, *동시 발생 순서*. 개념 수준에서 멈추지 않는다.
3. **자가 반박**: ✅/💡로 확정한 항목에 검토자 입장이 되어 "다른 합리적 선택지가 정말 없나?"를 한 번 되묻는다. 있으면 OQ로 내린다(§4 선택지 테스트의 능동 적용).

> 목표는 검토자만큼 빡빡한 게 아니라 **"이 초안으로 한 판이 일단 돌아가는가"** 를 분석가가 먼저 닫는 것이다. 명백한 도약을 분석가가 메우면 검토자는 미세 결함에 집중하고 ②↔③ 루프가 줄어든다.

## 3. 항목별 의미역 6역 게이트 — 각 UR이 닫혔는지 판정 〔보강·일반화〕

Frame Semantics(Fillmore): 모든 동사는 채워져야 할 **의미역**을 가지며, 빠진 역 = 불완전. 기존 방법론은 이를 *UI 표시*에만(WHAT/WHERE/HOW/WHEN) 적용했으나, 이를 **모든 동작 DoD에 일반화**한다.

각 동작 UR 후보에 대해 6역이 기획서에 정해졌는지 점검한다. 빈 역은 ❓(OQ).

| 역 | 질문 | 예 (소환) |
|---|---|---|
| **행위자** | 누가/무엇이 하는가? | 플레이어 |
| **대상** | 무엇에 대해? | 소환 후보 소환수 |
| **수단/자원** | 무엇으로/얼마를 들여? | 성급별 마나(1/3/9) |
| **위치** | 어디에서/어디로? | 대기석에 생성 |
| **조건/타이밍** | 언제/어떤 조건에서? 새 조건마다 정해졌나? | 정비·전투 페이즈 모두 가능, 대기석 빈자리 있을 때만 |
| **결과/상태변경** | 무엇이 바뀌나? 관찰 가능한 결과는? | 마나 차감, 대기석에 소환수 추가 |

**UI 표시 항목**은 이 게이트의 특수 사례로, 다음 축까지 세분해 닫는다:
WHAT(무엇을) · WHERE-1(어느 화면/요소) · **WHERE-2(요소 *안에서* 정확히 어디: 카드 우상단/체력바 왼쪽 끝)** · HOW(아이콘/색/뱃지/툴팁) · WHEN(상시인가, 특정 조건/타이밍에만인가).

> **이 UI 게이트는 *거부·실패 피드백*(축 4·F-017)에도 똑같이 적용한다.** "거부됨"으로 끝내지 말고 그 거부 표현을 WHAT(무엇을 알리나: 사유·상태)·WHERE(어디에)·WHEN(언제·얼마나)으로 닫는다. 단 *알리는가/무엇을 알리는가*(동작)가 요구 수준이고, 비주얼 스타일은 디자인 재량.

**판정 기준(필수):** *"구현자가 이 UR만 보고 추가 상상 없이 구현·배치할 수 있는가?"* — 화면만 정해지고 화면 내 위치(WHERE-2)가 비면 아직 안 닫힌 것이다. "표시된다"·"관찰 가능하게"로 닫지 않는다.
> 더 강하게(§0): *이 UR로 작업 요구사항을 도출할 때, 순수 기술·디자인 재량이 아닌 block이 생기면* 아직 안 닫힌 것이다 → 미결/도약/누락으로 처리한다.

## 4. 임의 확정 차단 — 선택지 테스트 + 스멜 트리거 〔미결 식별의 핵심〕

"당연히 그렇겠지"로 그럴듯하게 빈칸을 메우는 임의 확정이 미결을 *은폐*한다. 두 장치로 발생 시점에 차단한다.

### 4-1. 선택지 테스트 (필수, 차단 장치)

어떤 구체(위치·수치·형태·대상·범위·타이밍)를 UR로 적기 전에 묻는다:

> *"다른 합리적인 기획자가 이 구체를 **다르게 정할 수 있는가?**"*

- **다르게 정할 수 있으면 → ❓(OQ)** 로 내린다.
- 💡(함의)는 선택지가 **유일할 때만** 허용한다.

**✅ 비상속(필수):** 상위 명제의 ✅(예: "역할군은 아이콘으로 구분된다")는 하위 구체(예: "소환 카드 우상단에 아이콘을 표시한다")로 **자동 상속되지 않는다.** 구체화하면 그 수준에서 근거를 다시 확인하고, 없으면 ❓.

### 4-2. 스멜 어휘 OQ 트리거 〔보강〕

Requirements Smells 연구의 결함 어휘는 미명시의 **기계적 신호**다. 소스 절과 초안 UR을 다음 렉시콘으로 스캔하고, 적중하면 OQ 후보로 올린 뒤 선택지 테스트를 적용한다.

| 스멜 종류 | 트리거 어휘 (기획서 실례) |
|---|---|
| 미확정 표지 | `잠정`, `추후`, `검토 중`, `~예정`, `미정` |
| 주관·정도어 | `적절히`, `충분히`, `자연스럽게`, `매끄럽게`, `가파르게` |
| 약한 표현(가능성) | `~할 수 있다`, `~일 수 있다`, `가능성`, `검토한다` |
| 비교·최상급 | `더 강한`, `더 높은`, `가장`, `명백히 어려운` |
| 미완성 조건/수량 | `일정 라운드마다`, `약 N`, `특정 조건`, `여러`, `등`, `낮은 확률` |
| 비검증 표현 | `한눈에`, `손쉽게`, `과도하게` |

> 단, 모든 적중이 OQ는 아니다 — 선택지 테스트를 통과(선택지 유일)하면 💡로 확정. 스멜은 "여기를 의심하라"는 신호일 뿐 자동 판정이 아니다.

## 5. 독립 교차검증 — 놓친 영역을 잡는다

한 대상 분해의 구조적 한계(놓친 영역은 탐색 자체가 안 됨)를 보완한다. (Perspective-Based Reading)

- **독립 검증자**가 본 분해의 구조화 결과(동작 트리·UR·그룹)를 **보지 않고**, 정본 기획서와 대상만으로 처음부터 **플랫 리스트**(계층 없는 항목 나열, 같은 표현 수준)를 도출한다.
- 본 분해 결과와 플랫 리스트를 대조한다:
  - 플랫 항목이 UR에 **모두 매핑**되면 → 누락 없음.
  - **매핑 안 되는 항목** → 누락 후보. UR 추가 또는 기각을 사용자가 판단.
  - **태그 정합 점검(필수):** 검증자가 ❓/미명시로 태깅한 항목을 본 분해가 ✅/확정 UR로 다뤘으면 → **임의 확정 의심 신호** → 재검토 후 근거 없으면 ❓로 전환.

> 검증자에게 전달: 대상(범위)·정본 기획서 경로·플랫 리스트 기록 경로. 각 항목 형식: `[항목 / 도출 축 / ✅·💡·❓ / 한 줄 근거(기획서 위치)]`.

## 6. 역방향 소스-절 커버리지 — 누락 0 의 *증명* 〔보강·핵심〕

§2~5는 "충분히 뽑았다"는 *생성적* 장치다. 이것만으로는 "빠짐없이"를 **증명**하지 못한다. RTM backward traceability를 적용해 구성적으로 닫는다.

1. 대상의 기획서 해당 절(들)을 **한 문장/절 단위로 분해**한다.
2. 각 절을 다음 중 하나로 **반드시** 분류한다:
   - → **≥1 UR로 매핑** (그 절이 낳은 UR 번호를 적는다), 또는
   - → **비요구사항으로 명시 태깅**: `배경/이유`(09 의사결정 기록감) · `러프 의도`(상위 기획) · `메타/아웃게임`(범위 밖) · `컨셉/아트`(컨셉·비주얼 기획 — UR 아님, F-008) · `콘텐츠/밸런스 위임`(수치·인스턴스를 콘텐츠·밸런스 기획으로 위임, F-008) · `미결`(OQ로 이동)
3. **어느 분류에도 안 들어간 절 = 0** 이어야 종료. 남은 절이 있으면 그 절에서 UR이 더 나오거나 OQ가 더 나온다.

> 이 표(소스 절 → UR/태그)가 곧 누락 없음의 증거이자, 이후 [[기획 운영 컨벤션]] §3 드리프트 감사(기획서 변경 → 영향 UR 추적)의 입력이 된다.

## 7. 엔티티 × 동작 매트릭스 — CRUD형 빈칸 점검 〔보강〕

도메인 명사(엔티티)마다 라이프사이클 동작이 다 정의됐는지 격자로 점검한다. "생성은 정의했는데 소멸은 없다"류를 포착.

- **행**: 대상 영역의 도메인 엔티티(예: 소환수·마나·성급·시너지 카운트·배치 슬롯…)
  - **정적 속성뿐 아니라 동적 런타임 상태도 행에 올린다 〔F-007〕**: *전투 중 변하는 상태*(현재 HP·쿨타임 잔량·상태이상 스택·현재 타겟·다운 여부)는 생성(전투 시작 시 초기화)·변경(피격·틱)·소멸(라운드 종료)이 따로 규정돼야 한다. 정적 속성(최대 체력·성급)만 보면 "현재 HP가 어떻게 줄어드는가" 같은 핵심 전투 상태가 통째로 빠진다.
- **열**: 생성 · 조회/표시 · 변경 · 소멸 · (해당 시) 지속/저장
- 각 셀: 해당 동작이 의미 있는가? 있다면 UR이 있는가/OQ인가/무의미(–)인가?

> 의미 있는데 비어 있는 셀 = 누락 후보. 축 1(동작 트리)과 상호 보완.

## 8. 출력과 미결 처리

### 8-1. UR 산출

- 도출·검증된 각 항목을 **검증 가능한 UR 한 문장**으로 표현한다(§1 표현 수준).
- 컨벤션 §2의 **UR-NNN** 식별자를 부여한다(발번은 [부록 B](#부록-b--ur-번호-컨벤션-제안)).
- 사용자 시점에서 한 덩어리로 인식되는 **요구사항 그룹**으로 묶는다(선택). 한 UR은 한 그룹에만.
- **출력 형식 — UR당 블록 리스트(표 금지) 〔F-014〕.** UR 목록을 *표(markdown table)로 쓰지 않는다.* 근거·연결(seam/cross-ref)이 F-013으로 *문장*이 되면서 표 셀에 문단이 들어가 가로로 터지고 줄바꿈이 깨져 오히려 안 읽힌다. 대신 UR 하나 = 블록 하나로:
  - 1행: `**{id}** · {그룹} · {태그} — {검증 가능한 제목 한 문장}`
  - 다음 줄(들여쓰기): `└ 근거·연결: {출처(문서 제목 동반)}. {seam/cross-ref를 사람이 읽을 완전한 문장}.`
  - 세부기능(부착 단위)별로 소제목(`###`)을 두고 그 아래 블록들을 나열한다. 위→아래로 한 번 읽어 흐름이 잡히게.
  - **표는 진짜 표 데이터에만** — 분포·매트릭스·수치 격자(주로 `분해중간물.md`). UR 목록처럼 *문장이 칸을 채우는* 곳엔 쓰지 않는다.

### 8-2. 미결(OQ) 처리 — *수집*이 종료 조건

> 종료 조건은 **미결 0이 아니라 "미결 식별·수집 완료"** 다.

> **OQ 범위 — 기획적 미결만 〔F-016〕.** OQ는 *게임 설계가 안 정해진 사실·룰*만 가리킨다(예: 치명타 100% 초과 처리, 특성 분포 규칙, 그리드 크기). **요구사항 조직화 — 귀속·그룹화·세부기능 신설/리네임/경계(§1.5) — 는 OQ가 아니다.** 그것은 분석가가 추론해 *결정하고 근거와 함께 제시*할 결과다(초안에 "구조 제안"으로 적고, 사람은 교정만). "이 세부기능을 폐지할까요?"·"새 세부기능을 만들까요?"를 OQ로 올리는 것은 범주 오류 — 분석가가 답을 내라.

- 발견된 모든 ❓를 **미결 대장**에 `OQ-{n}`으로 모은다.
- OQ는 강제 즉시 해소하지 않는다. **모아서 작성**하고, 관련 세부기능은 명세서에서 `미정` 상태로 표시한다(컨벤션 §2 완전성 규칙: 각 기능/세부기능 채워짐/미정 표시).

> **OQ 작성 기준 — 사람이 단독으로 읽고 결정할 수 있는 "결정 카드"로.** OQ는 사람(기획자)이 보고 판단하는 산출물이다. 한 줄로 압축하거나 기호로 나열하지 말고 다음을 **완전한 문장**으로 갖춘다:
> - **질문** — 무엇을 정해야 하는가. 명료한 질문형으로(명사구 "…처리" ❌ → "치명타 확률이 100%를 넘으면 어떻게 처리하는가?" ✅).
> - **현황(왜 미결인가)** — 기획서가 무엇을 말하고 무엇을 비워뒀는지. "미명시"로 끝내지 않는다.
> - **선택지** — 각 후보를 *그 선택의 결과·의미와 함께* 한 줄씩. `{a/b/c}` 라벨 나열 금지.
> - **영향** — 이 결정이 무엇을 좌우하는가(어떤 UR·동작·작업이 이것 때문에 막혀 있는지). 사람이 우선순위를 매기는 근거.
> - **유형 〔F-010〕** — `[결정 필요]`(지금 설계 결정이 있어야 작업 막힘이 풀림) / `[콘텐츠 위임]`(개별 캐릭터·스킬 등 콘텐츠 기획이 채움) / `[밸런스 위임]`(수치·곡선을 밸런싱이 정함) / `[추후·로드맵]`(출시 범위 밖). 사람이 "지금 정할 것"과 "어차피 콘텐츠/밸런스로 미룰 것"을 구분하는 트리아지 표지 — 우선순위를 가른다.
> - **출처** — 기획서 절. **문서 번호만 적지 말고 제목과 함께** 〔F-013〕: `§1.3.3` ❌ → `「07 게임 시스템 기획서」 §1.3.3` ✅.
>
> 예) ❌ `OQ-DM-01 · 데미지계산 · 치명타 100% 초과 처리 · 미명시 · 후보:{상한/무시/더블크리} · §1.3.3`
> ✅ **OQ-DM-01 · 치명타 확률이 100%를 넘으면 어떻게 처리하는가?** (전장/유닛/데미지계산)
> ・현황: §1.3.3은 치명타 확률(%)이 발생 확률을 정한다고만 하고, 버프·시너지로 100%를 넘는 경우를 다루지 않는다.
> ・선택지: ①100% 상한 — 초과분 버림(항상 치명타까지만) / ②더블 크리티컬 — 초과분을 2차 치명타 확률로 사용(치명타 위에 치명타) / ③초과분 무시 — 100% 고정, 추가 효과 없음.
> ・영향: 치명타 빌드의 상한·밸런스를 좌우. 데미지계산의 치명타 적용 UR이 이 결정 없이는 경계 동작을 확정 못 함.
> ・출처: 「07 게임 시스템 기획서」 §1.3.3.
- 큰 설계 결정(밸런스·룰)은 추후 합의 시 [[09 의사결정 기록]](D-NNN)으로 확정 → 기획서 .md 반영 → 그 절이 UR을 새로 낳음. 이 방법론은 그 입구(미결 식별)까지 책임진다.

## 부록 B — UR 번호 컨벤션 + 저장 모델 (구현: `ur` CLI)

컨벤션 §2가 UR-NNN의 *성질*(전역 평면·불변·재사용금지·결번허용·추가순)은 정했으나 **발번 메커니즘**이 비어 있었다. 난점: *재사용 금지 + 결번 허용*을 동시에 지키려면 삭제와 무관하게 살아남는 **최고수위(high-water mark)** 가 필요하다(단순 `grep 최대+1`은 최대 번호 UR 삭제 시 재발급되어 깨진다).

**해법 — 구조화 저장(JSONL) 정본 + `ur` CLI 발번.** `docs/요구사항명세서/_db/*.jsonl` 이 정본이고, `전장/*.md` 는 CLI가 생성하는 투영물이다(상세: [`tools/ur/README.md`](../../tools/ur/README.md)).

- **정본 = `_db/ur.jsonl`** (한 줄 = 한 UR, git 평문). `nodes.jsonl`(계층) · `oq.jsonl`(미결) 동반.
- **발번 = `ur add` → `max(id)+1`** (폐기 포함 최고수위). 결정론 CLI가 수행 — LLM이 번호를 직접 세지 않는다.
- **재사용 금지 = `ur deprecate`** — 행을 지우지 않고 `상태=폐기`. 번호 영구 burn. 결번 허용.
- **이동** — `ur update <id> --node` 로 소속만 바꿈, 번호 유지.
- **정합성 감사**(컨벤션 §3 드리프트) — `ur reconcile`: UR의 부착 세부기능 존재·중복 id·최고수위·기능/세부기능 상태 정합. DuckDB로 `SELECT id FROM '명세서' EXCEPT SELECT id FROM '대장'` 류 차집합도 가능.
- **사람·LLM 접근 분리** — 사람은 생성된 `전장/*.md`(git diff)·웹 UI, LLM은 `ur list`/`ur query`(DuckDB SQL). 모든 *쓰기*는 CLI 독점.
- **도구 독립 유지** — _db는 `docs/요구사항명세서/` 안(기획 영역), 평문 git 추적. 지라를 가리키지 않는다.

> 즉 §2의 "각 기능/세부기능 채워짐/미정 표시"는 `nodes.jsonl` 의 `상태` 필드 + `ur add` 시 자동 `미정→채워짐` 전이로 구현된다.
