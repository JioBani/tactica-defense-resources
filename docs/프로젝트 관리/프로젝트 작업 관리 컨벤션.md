# 프로젝트 작업 관리 컨벤션 (Tactica Defense)

> 요구사항을 어떻게 **분석·분류하고, 작업으로 분해해 관리**하는가의 살아있는(living) 컨벤션.
> 게임 기획 내용이 아니라 *작업 분해·관리 방법론*이다.
> **출처 통합**: 「이슈분해-컨벤션」 + 「작업 도메인·마스터데이터 분류 지식」. 결정 배경은 요구공학(UR/SR 분석)과 본 프로젝트 합의에서 유도.
> **현재 도구**: Canopy (Jira 대체). 단 아래 원칙은 도구 무관 — Jira 시절 매핑은 §1 에 참고로 남긴다.

---

## 0. 목적과 핵심 긴장

- **왜 분해하는가**: 할 일을 정리하기 위해서다. 동시에 작업 관리 공간은 나뿐 아니라 **기획자도 보는** 공간이고, **실 산출물의 진행도는 사용자 요구사항(UR) 단위로 추적**되어야 한다.
- **핵심 긴장**: 추적·기획자 가독은 **UR 단위**가 좋은데, 실제 작업은 **작업 요구사항 단위**다. 게다가 UR ↔ 작업 요구사항은 **1:1 이 아니라 N:M** 이다.
- **해법의 골자**: 두 요구를 한 단위에 섞지 않고 **계층의 서로 다른 높이**에 둔다. 트리는 1:N 추적을 싣고, N:M·선행관계는 보조 표현(링크)으로 싣는다.

---

## 1. 요구사항·작업 계층 (Canopy 매핑)

```
컨텐츠 > 기능 > (세부기능 | 마스터데이터) > 작업
```

| 계층 | 담는 것 | 독자 | 역할 |
|---|---|---|---|
| **컨텐츠 / 기능** | 기능 영역 (도메인어) | 기획자 + 나 | 큰 묶음 |
| **세부기능** | **사용자 요구사항(UR)** 을 소유 | **기획자** | **진행도 추적 단위** |
| **마스터데이터** | 구체 인스턴스 (캐릭터·몬스터·스테이지…) | | 채워야 할 요소 (§4) |
| **작업** | **작업 요구사항 그룹** (도메인 태그) | **나** | **실제 작업 단위** |

- **UR 완료율 = 실 산출물 진행도.** 작업이 UR 에 링크되어 UR 단위 추적이 성립한다.
- "하나의 UR 에 여러 작업" = 표준 fan-out. 단 **작업은 UR 의 하위가 아니다 — N:M 링크로 연결**된다(§5).
- *구 Jira 매핑(참고)*: Epic=기능 영역 / Task=UR / Sub-task=작업(SR 그룹). Canopy 는 이 3단 한계를 풀어 컨텐츠·기능을 분리하고 마스터데이터를 추가했다.

## 2. UR / 작업 요구사항 정의와 경계 (요구공학 기반)

### 사용자 요구사항 (UR)
- **본질**: 플레이어/운영자 시점의 요구를 **도메인 언어**로. **"무엇 / 왜"만** 결정한다.
- **표현**: 사용자가 기능을 인식하는 어휘(소환수·성급·합성·승급·정비 페이즈…). "엔진/프레임워크/데이터 구조/알고리즘" 같은 **시스템 framing 금지** — 그건 작업 요구사항으로 내려갈 재료다.
- **단위**: 검증 가능한 항목 + 기능별 그룹. UR 은 **항상 세부기능에 부착**한다(기능·컨텐츠에 직접 붙지 않는다).
- **상태**: 각 UR 은 수동 상태(완료 / 미구현 / 오구현)를 가진다 — 오구현이면 사유를 기록한다.

### 작업 요구사항 (= 작업)
- **본질**: UR(또는 마스터데이터)을 **만족·완성시키려면 무슨 작업을 해야 하는가**를, **어느 도메인의 일인지**(§3)로 나열한 것. "어느 컴포넌트가 / 어느 데이터로 / 어느 시점에 / 어떤 결과를".
- **표현**: 도메인어 + 컴포넌트 식별자 + 외부 표면. 코드 식별자·알고리즘은 DD(상세 설계) 영역.
- **단위**: **한 번에 작업하기 좋은 그룹**으로 묶음. 이 그룹이 곧 **작업(Task)** 단위다.

### 경계 규칙
- **UR ↔ 작업 요구사항 언어 차이는 강제하지 않는다.** 강제되는 경계는 **작업 요구사항 ↔ DD**:
  - **"상태는 작업 요구사항, 수단은 DD"** — *어떤 상태여야 하는가*만 작업 요구사항. 그 상태를 만드는 수단(마이그레이션·분기·클론)은 DD.
  - **"한 단계 추론 거리"** — 사용자 시점 1단계 추론으로 닿으면 작업 요구사항, 구현 메커니즘 이해 전제(2단계+)면 DD.
- **매핑은 그룹 단위, N:M.** UR ↔ 작업 요구사항은 N:M 이라 **항목 단위 매핑 금지** — **UR 그룹 ↔ 작업** 에서만 대응을 기록한다.
- **매핑은 반드시 명시하고 원문으로 쓴다.** 각 작업은 **그것이 어떤 UR 을 만족시키는지**를 명시한다. 기호/ID 가 아니라 **원문**을 풀어 쓴다 — 기호로만 적으면 사람이 매핑을 못 본다.

## 3. 작업 도메인 (어느 직군의 일인가) 〔통합 추가〕

작업 요구사항은 **어느 도메인의 일인지**로 분류한다. 작업(Task)에 도메인 태그가 하나 붙는다.

| 도메인 | 내용 |
|---|---|
| **기획** | 요소의 스탯·스킬·컨셉 설계. (캐릭터 원화·스킬 이펙트의 *컨셉/아트*는 여기 — '디자인'이 아니다.) |
| **디자인** | 요소가 어떻게 표현·배치되는가 — UI·레이아웃·애니메이션·연출 |
| **사운드** | 모든 게임 사운드 |
| **구현** | 실제 개발. **이 도메인의 작업 요구사항 = 시스템 요구사항(SR)** (별칭). |
| **밸런싱** | 수치 밸런싱 (확률·코스트·성장 곡선) |
| **기타** | 위에 안 잡히는 것 |

- **핵심 — UR 하나가 여러 도메인으로 fan-out.** 기존 컨벤션은 UR→**SR(구현)** 만 봤지만, 실제로 하나의 UR 은 기획·디자인·사운드·구현·밸런싱 작업을 낳는다. 즉 **"작업"은 구현만이 아니다.**
  - 예) "소환수 합성" UR → [기획] 결과물 설계 / [디자인] 합성 UI·연출 / [사운드] 효과음 / [구현(SR)] 로직·데이터 / [밸런싱] 확률·코스트.
- **스코프 밖 — 상위 활동(테두리 만들기)**: **시스템 기획**(메커니즘 테두리)·**세계관 기획**(스토리). 요소를 채우는 작업이 아니라 구조 자체를 만드는 활동 — 본 컨벤션의 작업 도메인으로는 다루지 않는다(추후).

## 4. 마스터데이터 (구체 인스턴스) 〔통합 추가〕

- **정의**: 시스템·세계관 기획이 정한 테두리 안을 **채우는 구체 인스턴스**(캐릭터·몬스터·스테이지·카드, 그 스탯·스킬). = 컨텐츠 요소.
  - 요구공학에서 "콘텐츠 인스턴스는 UR 이 아니다 → 콘텐츠/밸런스 위임"이라며 분석 밖으로 민 그것을, **1급 단위로 추적**하는 것이다.
- **트리 위치**: **세부기능과 같은 tier**(둘 다 기능의 자식). 보통 `소환수 마스터데이터 제작` 같은 기능 노드 아래 잎으로 나열.
- **세부기능 vs 마스터데이터**:

| | 세부기능 | 마스터데이터 |
|---|---|---|
| 성격 | 기능적 **동작**("소환수 합성") | 구체 **인스턴스**("캐릭터 아리아") |
| UR | **소유** → UR → 작업 | **없음** → **작업으로 바로** fan-out |
| 진행도 | 하위 UR 완료율 | UR 없어 "—"(작업으로 추적) |

- **마스터데이터도 도메인으로 fan-out** (기획 스탯·디자인 원화·사운드 보이스·구현 등록·밸런싱 수치).

> **핵심 추상**: UR 과 마스터데이터는 둘 다 "**도메인 묶음으로 fan-out 되는 work-bearing unit**". 종류만 다르다(UR=기능적 요구 / 마스터데이터=구체 인스턴스). 작업 분해 시, 어느 근원에서 나왔든 **"완성하려면 어느 도메인의 무슨 작업이 필요한가"** 를 도메인별로 빠짐없이 길어낸다.

## 5. N:M 잔여분과 공유 인프라 (예외)

- 한 작업이 **여러 UR 에 공유**되거나(상태효과 시스템급 전역 인프라), 한 세부기능에 담기엔 너무 큰 경우 — 트리는 부모가 하나뿐이라 못 담는다.
- 해법: 공유/대형 인프라는 **대응 UR 없는 독립 작업**으로 두고, 소비하는 UR 들이 **`blocks`(선제) 링크**로 가리킨다. **절대 중복 작업으로 만들지 않는다**(이중 집계 방지).
- **승격 판정 규칙**: "이 작업을 UR 하나가 소유할 수 있나?" — 예(한 UR 전용·작음)→ 그 세부기능 밑 작업. 아니오(≥2 UR 공유 또는 대형)→ 독립 작업 + `blocks` 링크.
- **빈도**: 드물다. 평소엔 세부기능 밑 작업으로 충분하다.

## 6. 분해 방법론

- **정본 기준**: `docs/기획서` 의 .md 가 source of truth. 구버전(컨셉아트·외부 프롬프트)은 기준으로 쓰지 않는다. 결정 이유는 `09 의사결정 기록`(ADR).
- **두 방향 교차검증**: 하향식(UR/작업 분석 기법) + 상향식(세부→상위)을 독립 수행 후 통합. 수렴하는 델타는 신뢰도가 높다.
- **결정/미결 분리**: 단독 결정 불가하거나 애매한 항목은 본문에 억지로 확정하지 말고 **별도 미결 목록(OQ)** 으로 모아 사용자가 확정한다. 단 *조직화*(어느 세부기능에 붙일지·그룹·경계)는 미결이 아니라 분석가가 근거와 함께 결정할 산출물이다.
- **작업 도출 무중단(전 도메인)**: UR/마스터데이터를 보고 작업을 뽑을 때 *"이게 어떻게 동작해야 하지?"* 에서 막히는 지점이 **0** 이어야 한다(상류 누락 신호). 남아도 되는 "열린 것"은 순수 기술(구현 방식)·순수 디자인(비주얼 선택) 재량뿐. 이 기준을 **구현뿐 아니라 전 도메인**에 적용한다.
- **신뢰성(환각 방지)**: 분해·종합은 **입력 파일을 직접 read** 한 근거로만. 기억·요약으로 매핑을 추측하지 않는다.
- **사람 가독성·원문 우선**: 요구사항·그룹·매핑을 기호/ID 로만 적지 말고 **원문을 풀어 쓴다**. ID 는 추적용 보조 표기.

## 7. 등록(발행) 프로세스

```
UR 분석  →  작업 요구사항 분석(도메인별, 대응 UR)
   │                    │
   └─────────┬──────────┘
             ▼
   점유 제외: 이미 등록된 작업이 점유한
   작업 요구사항은 빼고 (중복 방지)
             ▼
        등록 draft 작성
             ▼
        사용자 검수 (게이트)
             ▼
   승인 후 업로드 (UR·작업·링크)
```

- 분석 단계(UR/작업)는 검토·승인 게이트를 가지며 산출물을 남긴다.
- 등록은 **바로 올리지 않는다** — draft 를 먼저 작성해 검수받고, **명시적 승인 후에만** 등록한다.
- 쓰기는 되돌리기 어려우므로 **승인 게이트 + 멱등성**(재실행 시 중복 작업 생성 금지)을 지킨다.

## 한 줄 요약

**컨텐츠>기능>세부기능·마스터데이터>작업.** UR 은 세부기능 소유(진행도 추적), 작업은 도메인 태그 붙은 실행 단위. **UR↔작업은 N:M 링크**(트리는 1:N 추적, 공유 인프라는 독립 작업+blocks). 작업은 **구현(SR)만이 아니라 기획·디자인·사운드·밸런싱 전 도메인**으로 fan-out. **마스터데이터**는 UR 없이 작업으로 직접 fan-out 되는 구체 인스턴스. 분해는 직접 read·교차검증·미결 분리, 등록은 draft 검수 후.
