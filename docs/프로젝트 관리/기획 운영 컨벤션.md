# 기획 운영 컨벤션 (Planning Operating Convention)

> 기획을 **어떻게 운영하는가**의 규칙을 정의하는 **살아있는 문서**다. 규칙이 바뀌면 본문을 갱신한다.
> (기획서 = *무엇·사실*, [[09 의사결정 기록]] = 게임 디자인 결정의 *이유*. 이 문서 = **기획 영역**의 운영 규칙 + 개발(작업 명세서)과의 접점.)
> 결정에 이른 배경은 본문을 깔끔히 유지하기 위해 맨 끝 **부록**에 짧게 둔다.

---

## 1. 문서 지형 — 기획 영역과 개발과의 경계

**기획은 요구사항 명세서에서 끝난다.** 기획서·요구사항 명세서가 *기획 영역*(이 컨벤션의 관할)이고, 작업 명세서는 그 산출물을 받아 실행하는 *개발 영역*이다. 둘 사이는 **n:m 도출**이라는 핸드오프로 이어진다.

```
─ 기획 영역 (이 컨벤션의 관할) ───────────────
  기획서          why   의도·러프한 방향 (결정 이유는 09)   [러프해도 됨]
     ▼ 구체화
  요구사항 명세서  what  사용자 관점의 모든 요구사항         [SSOT · 기획의 최종 산출물]
──────────────────────────────  ⟂ n:m 도출 (기획 → 개발 핸드오프)
─ 개발 영역 ─────────────────────────────────
  작업 명세서      how   작업 요구사항·작업 단위            [지라 TACD]
```

| 영역 | 문서 | 무엇을 담나 | 위치 |
|---|---|---|---|
| **기획** | 기획서 | 의도·콘셉트·방향 (러프) | `docs/기획서/00·04·07·08` |
| **기획** | 의사결정 기록 | 게임 디자인 결정의 배경·이유 (immutable) | [[09 의사결정 기록]] |
| **기획** | 요구사항 명세서 | 사용자 관점의 **모든** 요구사항 (SSOT) | `docs/요구사항명세서/` |
| **개발** | 작업 명세서 | 작업 요구사항·작업 단위 (요구사항에서 파생) | 지라 (TACD) |

**관할 경계** — 이 컨벤션은 **기획 영역**(기획서·요구사항 명세서)을 규정한다. 작업 명세서는 개발 영역이라 **내부 운영은 개발 쪽 자유**이며, 여기서는 기획과의 **접점(도출 규칙, §3)만** 정의한다. (작업 명세서는 *역할* 이름이고 지라는 그것을 담는 *도구* — 도구는 교체 가능하다.)

**원칙**

- **SSOT** — 사용자가 무엇을 받는가의 유일·완전 원천은 요구사항 명세서다. 지라에만 있고 명세서엔 없는 요구사항은 존재하지 않는다(지라는 파생).
- **도구 독립** — 요구사항 명세서는 지라를 참조하지 않는다. 참조는 **지라 → 명세서** 단방향. 지라가 사라져도 명세서만으로 완결된다.
- **변경 입구 단일화** — 기획 변경은 **요구사항 명세서로만** 들어오고, 작업 명세서는 거기서 파생된다.

## 2. 게임 기능 계층 — 요구사항 명세서의 구조

요구사항 명세서는 **게임 기능 계층**으로 구성한다. 모든 항목을 **사용자 관점**으로 적는다.

```
콘텐츠                       ← 1계층
 └ 기능                      ← 2계층
    └ 세부 기능               ← 3계층
       └ 요구사항 그룹 (선택)   ← 4계층
          └ 사용자 요구사항     ← 잎
```

| 층 | 정의 | 예 |
|---|---|---|
| **콘텐츠** | 사용자가 하나의 독립된 영역으로 인지하는 최상위 묶음 | 전장 · 소환술사 모집 · 소환술사 강화 |
| **기능** | 한 콘텐츠 안에서 사용자가 구분해 인지하는 기능 단위 | (전장 ▸) 소환수 · 소환터미널 · 라운드 |
| **세부 기능** | 한 기능이 제공하는 개별 동작 단위 | (소환수 ▸) 합성 · 배치 · 라이프사이클 |
| **요구사항 그룹** *(선택)* | 한 세부 기능의 요구사항을 한 요소(측면) 단위로 묶어 인식하기 쉽게 한 그룹 | (합성 ▸) 합성 실행 · 합성 효과 · 표시 |
| **사용자 요구사항 (UR)** | 사용자 관점에서 충족해야 하는 검증 가능한 개별 항목. 서술형 한 문장 | 동일 3개→자동 합성 · 승급 시 스탯 강화 |

**요구사항 식별자** — 각 사용자 요구사항은 **`UR-NNN`** 식별자를 가진다.

- **전역 평면**: 위치(콘텐츠/기능)를 키에 인코딩하지 않는다. 번호는 의미를 안 담고, 의미는 **제목**이 담당 → `UR-247 동일 3개 모으면 자동 합성`.
- **불변·전역 유일**: 추가 순서대로 부여, **재사용·재정렬 금지**, 결번 허용. 요구사항이 다른 곳으로 옮겨가도 번호는 유지(매핑이 안 깨지도록). `D-NNN`(의사결정) 규칙과 동일.
- 정규 키 = `UR-NNN`(매핑·검색이 잡는 것). 제목은 자유롭게 바뀌어도 키 매칭은 안 깨진다.

**작성 절차 (bottom-up)** — 세부 기능을 정한다 → 사용자 요구사항을 쭉 나열한다 → 한 요소 단위로 묶을 만하면 요구사항 그룹으로 묶는다(아니면 생략).

**완전성 규칙** — "완전하지만 waterfall 아님":

- **범위 완전성** (콘텐츠·기능 골격) = **선행** 확정. 모든 기능이 계층에 등장.
- **상세 완전성** (각 기능의 요구사항) = **작업 착수 시 JIT**로 채움. 각 노드에 "채워짐 / 미정" 상태를 표시.

**기획서에의 적용** — 기획서(특히 [[07 게임 시스템 기획서]])도 이 계층을 **느슨히** 따른다. 단 러프한 전달이 목적이므로 ① 상위 계층(콘텐츠·기능)까지만 적거나, ② 계층을 자유롭게 써도 된다. 계층을 끝까지 채우고 완전성을 책임지는 건 요구사항 명세서의 역할이다.

## 3. 요구사항 도출 체인 — 요구사항 명세서 → 작업 명세서

사용자 요구사항에서 작업을 도출한다. **도출은 위→아래, 완료·만족은 아래→위.**

```
사용자 요구사항 (UR)         ← 요구사항 명세서의 잎
   ▲   n:m
작업 요구사항                ← UR 충족을 위해 작업이 만족해야 할 것
   ▲   묶음
작업 단위 (지라 Sub-task)     ← 작업 요구사항을 한 번에 작업하기 좋게 묶은 실행 단위
```

- **UR ↔ 작업 요구사항 = n:m** — 인프라성 작업 요구사항 1개가 여러 UR을 동시에 서빙하기도 한다.
- **작업 요구사항 → 작업 단위 = 묶음** — 작업 단위를 완료하면 묶인 작업 요구사항이 만족되고, (n:m으로) 사용자 요구사항이 만족된다.

**지라 매핑** — 지라 한 트리에 요구사항 명세서의 상위 골격과 작업 명세서가 함께 들어간다.

| 지라 issue type | = 우리 계층 | 소속 |
|---|---|---|
| Epic | 기능 | 요구사항 명세서 (미러) |
| Task | 세부 기능 | 요구사항 명세서 (미러) |
| Sub-task | 작업 단위 | 작업 명세서 |

- **콘텐츠(1계층)** 는 Epic 위라 issue type이 없다 → `[전장]`/`[대기]` **라벨**로 표현.
- **요구사항 그룹·사용자 요구사항**은 지라에 없다 → 요구사항 명세서(문서)에만 존재. 즉 Epic/Task는 **상위 골격(기능·세부기능)만** 반영하고, 요구사항 내용은 문서에 산다.
- **issue type 이름 변경**(Epic→기능, Task→세부 기능, Sub-task→작업 단위)은 **정의만 해두고 아직 적용하지 않는다.** (Jira 관리자 설정이며 시점은 추후.)

**UR ↔ 작업 매핑** — 도구 독립 원칙상 매핑은 **지라(작업) 쪽이 `UR-NNN`을 들고** 있는다(참조는 지라 → 명세서 단방향).

- **현재(소규모): 작업 단위(Sub-task) 설명란에 텍스트로 기재.** 설명란에 고정 섹션 `## 충족 요구사항` 을 두고 그 아래 `- UR-247 동일 3개 모으면 자동 합성` 식으로 나열한다(키+제목 병기 → 작업만 봐도 목적 파악).
- **역방향**(작업 → UR): 설명란을 읽는다. **정방향**(UR → 작업): `jira_search` JQL `text ~ "UR-247"` 또는 작업 description grep.
- 규모가 커지면 **라벨**(`= "UR-247"` 정확검색) → 필요 시 **커스텀 필드** 순으로 이관. `UR-NNN` 키가 안정적이라 **이관 비용 0**. (업계 정석은 라벨/필드/전용앱이며 텍스트는 소규모 한정 — 추후 상향.)

**기획 변경 반영 + 정합성 점검** — 명세서가 git에 있으니 변경 감지는 git이 한다.

```
요구사항 명세서 수정 (git commit)
 → git diff 로 추가/수정/삭제된 UR-NNN 추출
 → 각 UR 을 작업 description 에서 grep → 영향받는 작업
      · 삭제 → 그 UR 단 작업 = 고아 (close/재지정)
      · 수정 → 재검토 (stale 가능)
      · 추가 → 미커버 (작업 생성)
 → 인프라성 작업은 관련 UR 이 안 변하면 건드리지 않음
```

- **드리프트 감사**: 명세서의 UR 집합 vs 작업들에 적힌 UR 집합의 **차집합** → 고아/미커버 일괄 점검.

## 4. 용어 정의

**문서(역할)** — §1 참조: 기획서 · 요구사항 명세서 · 작업 명세서.

**요구사항 명세서 내부 (게임 기능 계층)** — §2 표 참조: 콘텐츠 · 기능 · 세부 기능 · 요구사항 그룹 · 사용자 요구사항(UR).

**작업 명세서 내부**

- **작업 요구사항** — UR에서 도출한, 작업이 충족해야 할 요구사항.
- **시스템 요구사항 (SR)** — 작업 요구사항 중 **구현(시스템·코드) 한정**. 현 프로젝트의 작업은 대부분 여기.
- **작업 단위** — 작업 요구사항을 한 번에 작업하기 좋게 묶은 실행 단위 = 지라 Sub-task.

> **충돌 주의** — **사용자 요구사항**(무엇을 받나 · 요구사항 명세서)과 **작업/시스템 요구사항**(무엇을 만드나 · 작업 명세서)은 다른 층이며 n:m으로 연결된다.

**업계 표준 어휘 (참고)**

| 우리 용어 | 업계 어휘 |
|---|---|
| 요구사항 명세서 | Functional / Requirements Specification (SRS), Living Documentation |
| 게임 기능 계층 | System → Feature 구조 (GDD), Story Map (Jeff Patton) |
| 사용자 요구사항 | User Requirement / Acceptance Criteria |
| 시스템 요구사항 | System Requirement |
| "범위 먼저, 상세는 착수 때" | Progressive Elaboration / Rolling-wave Planning |
| 요구사항이 다 채워져 착수 가능한 상태 | Definition of Ready |

## 5. 미결 사항

> 결정되면 본문으로 승격한다.

- ~~**사용자 요구사항 ID 체계**~~ → §2로 해결: 전역 평면 `UR-NNN`, 불변·재사용금지, 제목 병기.
- ~~**n:m 매핑 위치**~~ → §3로 해결: 지라 작업 **설명란 텍스트**(현재 소규모), 추후 라벨/필드 이관.
- **요구사항 입자도** — 사용자 요구사항 하나를 어디서 끊을지. n:m 매핑 품질을 좌우.
- **작업 요구사항의 물리적 위치** — 별도 산출물인지 작업 단위(Sub-task)에 흡수되는지. (현재는 작업 단위 설명란에 흡수)
- **작업 명세서 계층 재설계** — 작업 편의 기준으로 지라 에픽/Task/Sub-task를 어떻게 묶을지. 인프라를 라벨/이슈타입/링크 중 무엇으로 표시할지.
- **기존 지라 retrofit** — 현재 Sub-task의 서술형 요구사항을 걷어내 요구사항 명세서 UR로 이관하고, Sub-task는 작업 단위로 재구성하는 절차.
- **`지라-요구사항-분석-등록` 스킬 재조정** — UR은 요구사항 명세서로, 작업(SR)은 작업 명세서로 가도록.

## 부록 — 설계 배경 (참고)

이 모델이 푸는 문제: 기존엔 지라 Sub-task에 *사용자 요구사항*과 *그것을 만족시키는 작업·인프라*가 한 레벨에 섞여, 기획이 바뀌면 반영 지점이 애매했다. → **요구사항(문제 공간)과 작업(해결 공간)을 분리**하고, 트리가 아닌 **n:m**으로 연결해 해소했다(인프라 1개가 여러 요구사항을 서빙하는 관계는 트리로 못 그린다).

명명 근거: 1계층은 "시스템"의 내부-메커니즘 뉘앙스를 피해 유저 인지 단위인 **콘텐츠**로, 잎은 작업 측 용어인 "완료 정의(DoD)" 대신 문서 목적과 맞는 **요구사항**으로 택했다. 전장이 콘텐츠고 그 안의 소환수·라운드가 기능인 것은 지라가 `[전장]`을 태그로, 소환수를 에픽으로 둔 층위와 같다.
