# 이슈 분해 방법론

> **구현** 에픽 하위의 이슈를 누락/중복 없이 나누고, 기획 누락을 식별하기 위한 방법론
>
> 본 문서는 업무 분류 체계 중 **구현** 단계의 에픽을 대상으로 한다.

---

## 목표

기획서를 바탕으로 에픽 하위 이슈를 분해하여 다음을 도출한다:

1. **해야 할 일** - 구현해야 할 이슈 목록
2. **기획 누락** - 게임플레이 기획에서 빠진 부분

## 조건

- 누락 없이 나누기
- 중복 없이 나누기
- 기획 미비 식별

---

## 전제: 업무 분류 체계

해야 할 일의 성격을 7개로 분류한다. 각 에픽은 이 중 하나에 해당한다.

1. **게임플레이 기획** - 게임 내 규칙/상태/흐름 등 게임 시스템 자체를 설계/명세
2. **콘텐츠 디자인** - 캐릭터/스킬/특성 등 콘텐츠의 구성과 동작 설계
3. **구현** - 코드/데이터/툴로 실제 플레이 가능하게 구현
4. **비주얼** - 시각적 결과물 제작/적용
5. **사운드** - 청각적 결과물(SFX/BGM/믹싱) 제작/적용
6. **스케일링 디자인** - 난이도/성장/보상 곡선 설계, 밸런스 튜닝
7. **테스트** - 기능 검증, 결함 발견/재현/보고

---

## 분해 방법: 동작 트리 + 보조 관점 체크

### 왜 동작 트리인가

- 게임 기능의 본질은 "무엇이 어떻게 동작하는가"이다.
- 데이터, UI, 시스템 연동 등은 동작을 지원하기 위해 존재한다.
- 따라서 **동작이 주축**이 되고, 나머지는 파생된다.

**동작 노드 vs 시스템 노드 구분:**

트리의 노드는 대상이 실제로 수행하는 **행동**이어야 한다. 행동을 지원하는 시스템/메커니즘은 트리의 상위 노드가 아니라, 동작의 하위 또는 기반으로 위치한다.

| 구분 | 노드 형태 | 예시 |
|------|----------|------|
| **동작 노드** (올바름) | "~한다" 형태의 행동 | 대기한다, 공격한다, 피격당한다, 다운된다 |
| **시스템 노드** (잘못됨) | "~시스템", "~계산", "~규칙" | 스탯 시스템, 데미지 계산, 전투 규칙 |

### 왜 보조 관점이 필요한가

- 동작 트리는 **동작/상태만** 식별한다.
- 동작에서 직접 파생되지 않는 것들(데이터 구조, 정보 표시, 선행 작업)은 별도 관점으로 잡아야 한다.

### 보조 관점 (3개)

| 관점 | 역할 | 질문 |
|------|------|------|
| **데이터/속성** | 동작에서 직접 안 보이는 데이터 구조 도출 | 이 기능에 필요한 속성/설정값은? |
| **UI/정보 표시** | 플레이어에게 보여줄 것 도출 | 플레이어에게 어떤 정보를 전달해야 하는가? |
| **기반 작업** | 선행되어야 할 작업 도출 | 동작과 무관하게 먼저 만들어야 할 것은? |

## 워크플로우

분해는 두 단계로 나뉜다:

- **Phase A**: Epic → Task 도출 및 등록
- **Phase B**: Task별 Sub-task 도출 및 등록 (Task마다 반복)

---

### Phase A: Epic → Task

#### 1단계: 기획서 파악

- 해당 에픽 관련 기획서를 읽고 전체 그림을 이해한다.

#### 2단계: 에픽 경계 확인

- Jira 에픽 목록을 확인하여, 이 에픽의 범위와 인접 에픽의 범위를 구분한다.
- 공통 시스템은 공통 에픽에, 고유 동작만 이 에픽에 포함한다.

#### 3단계: Task 도출

##### 3-1. 상위 동작 트리 작성

- 대상의 **라이프사이클(생성 → 행동 → 소멸)**에서 출발하여, 공통 동작을 포함한 1~2단계 깊이의 트리를 작성한다.
- **시작 질문**: "이 대상은 게임에서 실제로 무엇을 하는가?" — 행동의 흐름을 먼저 나열한 뒤 트리로 구성한다.
- **주의**: 기획서의 목차/항목 구조를 그대로 트리로 변환하지 않는다. 기획서는 참조하되, 트리의 축은 대상의 동작 흐름이다.
- 각 노드에 아래 이모지 태그를 붙여 상태를 구분한다:

| 이모지 | 의미 |
|--------|------|
| 💻 | 구현 대상 (이 에픽에서 새로 구현) |
| 🧩 | 공통 에픽 소속 (구현 여부 무관) |
| ✅ | 중복 (다른 에픽에서 이미 구현 완료) |
| ❓ | 기획 누락 (기획서에 명세 없음) |

- **주의: 구현 복잡도와 동작 노드를 혼동하지 않는다.** 동작 트리는 "게임에서 일어나는 행동"을 나열하는 것이지, "코드로 구현할 작업"을 나열하는 것이 아니다. 구현이 단순하더라도(데이터 설정, 필드 하나 읽기 등) 기획서에서 명시된 행동이면 노드에 포함한다. 구현 복잡도는 이후 업무 분류(5-3)에서 판단한다.
- **대칭성 체크**: 트리 작성 후, 기획서에서 병렬/대칭으로 기술된 개념(A와 B, N개 종류 등)이 트리에도 모두 대응되는지 확인한다.
  - 예: "소환술사 효과와 소환수 특성 두 종류" → 두 가지 모두 트리에 있는가?
  - 예: "공격/방어/지원 3가지 모드" → 세 가지 모두 트리에 있는가?

##### 3-2. Task 경계 판단

상위 동작 트리의 노드들을 아래 기준으로 Task 단위로 묶는다:

- **독립성** - 다른 Task와 겹치지 않는가
- **응집성** - 내부 동작들이 하나의 기능으로 묶이는가
- **크기** - 너무 크면 분할한다

**사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 4단계: Task Jira 등록

- 도출된 Task를 해당 에픽의 하위 작업으로 Jira에 등록한다.

---

### Phase B: Task → Sub-task (Task마다 반복)

대상 Task를 선택한 뒤, 아래 단계를 수행한다.

#### 5단계: Sub-task 도출

##### 5-1. 동작 트리 상세화

- Task 내의 동작을 기획서에 근거하여 말단 노드까지 분해한다.
- **기획서에 없는 부분 → 기획 누락으로 기록한다.**

##### 5-2. 보조 관점 체크

각 동작 노드 + Task 전체에 대해 3개 관점으로 질문한다:

- **데이터/속성**: 이 동작에 필요한 속성/설정값은?
- **UI/정보 표시**: 이 동작/상태와 관련하여 표시 가능한 정보를 항목화한다.
  - 단순히 "UI가 없다"가 아니라, **표시 가능한 정보를 먼저 항목화**한 뒤, 각 항목별로 **플레이어에게 표시할 필요가 있는지를 판별**한다.
  - 표시가 필요한 항목에 대해 기획서와 대조하여 누락을 판단한다.
- **기반 작업**: 동작과 무관하게 선행되어야 할 작업은?

**답이 기획서에 없는 항목 → 기획 누락으로 기록한다.**

##### 5-3. 업무 분류 및 분류

5-1의 말단 노드와 5-2에서 도출된 항목 각각에 업무 분류를 부여한다.

도출된 전체 항목을 사용자에게 보여준 뒤, 아래 기준으로 분류한다:

- **구현** → Sub-task 목록에 포함
- **게임플레이 기획** (선행 필요) → 기획 누락으로 분류
- **그 외 업무 분류** (콘텐츠 디자인, 스케일링 디자인, 비주얼, 사운드 등) → 해당 분류의 후속 작업으로 분리

**주의: 항목이 여러 업무 분류에 걸치는 경우**

하나의 항목이 둘 이상의 업무 분류를 포함할 수 있다. 이 경우 항목을 분리한다.

- 예: "성급 표시" → 표시 UI를 만드는 것(구현) + 표시 디자인을 결정하는 것(비주얼)
- 예: "스탯 테이블 설계" → 데이터 구조를 만드는 것(구현) + 수치/곡선을 결정하는 것(스케일링 디자인)

**사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 6단계: 검증

- 기획서를 다시 읽으며 빠진 내용이 없는지 대조한다.
- 이슈 간 중복이 없는지 확인한다.
- **사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 7단계: 이슈 목록 정리

아래 형식으로 정리하여 사용자에게 보여준다:

```
## Task: {Task 이름}

### 도출된 전체 항목

| # | 업무 분류 | 제목 |
|---|----------|------|
| 1 | 구현 | ... |
| 2 | 구현 | ... |
| 3 | 게임플레이 기획 | ... |
| 4 | 스케일링 디자인 | ... |

### → Sub-task (구현)

| # | 유형 | 제목 | 기획 상태 |
|---|------|------|----------|
| 1 | 하위 스토리 | ... | 구현 가능 |
| 2 | 하위 작업 | ... | 기획 필요 |

### → 기획 누락 (게임플레이 기획)

| # | 누락 항목 | 상세 내용 |
|---|----------|----------|
| 1 | ... | ... |

### → 후속 작업 (기타 업무 분류)

| # | 업무 분류 | 제목 |
|---|----------|------|
| 1 | 스케일링 디자인 | ... |
```

**사용자 검증을 받은 후 다음 단계로 진행한다.**

##### 7-1. 기획 누락 항목의 Sub-task 등록 판단

기획 누락 목록을 사용자에게 보여준 뒤, 각 항목에 대해 **기획 결정과 무관하게 구현이 필요한지** 확인한다.

- **구현이 확실한 항목** → Sub-task로 등록하되, Jira 상태를 **"기획 필요"**로 설정한다.
  - 예: "라운드 간 다운 유닛 처리" — 기획 결정이 어떻게 되든 구현은 반드시 필요
- **기획 판단에 따라 구현 여부가 달라지는 항목** → Sub-task로 등록하지 않고 기획 누락으로만 기록한다.
  - 예: "행동 상태 UI 표시 여부" — 표시하지 않기로 결정되면 구현 불필요

**사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 8단계: Sub-task Jira 등록

- 도출된 Sub-task를 해당 Task의 하위 작업으로 Jira에 등록한다.
- 7-1에서 구현이 확실하다고 판단된 기획 누락 항목도 Sub-task로 등록하고, Jira 상태를 **"기획 필요"**로 설정한다.
- Task의 **설명**에 이 Task가 다루는 범위를 간략히 기술한다.
- Task에 **댓글**로 도출 근거(동작 트리, 보조 관점 체크)를 남긴다. (형식은 `구현 이슈 작성 요령.md` 참고)

| 산출물 | 내용 |
|--------|------|
| **동작 트리** | 에픽 고유 동작을 트리 구조로 분해한 결과 |
