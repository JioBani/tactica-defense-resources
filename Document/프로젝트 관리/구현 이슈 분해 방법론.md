# 이슈 분해 방법론

> **구현** 에픽 하위의 이슈를 누락/중복 없이 나누고, 기획 누락을 식별하기 위한 방법론
>
> 본 문서는 업무 분류 체계 중 **구현** 단계의 에픽을 대상으로 한다.

---

## 목표

기획서를 바탕으로 에픽 하위 이슈를 분해하여 다음을 도출한다:

1. **해야 할 일** - 구현해야 할 이슈 목록
2. **기획 누락** - 게임플레이 기획에서 빠진 부분

## 조건

- 누락 없이 나누기
- 중복 없이 나누기
- 기획 미비 식별

---

## 전제: 업무 분류 체계

해야 할 일의 성격을 7개로 분류한다. 각 에픽은 이 중 하나에 해당한다.

1. **게임플레이 기획** - 게임 내 규칙/상태/흐름 등 게임 시스템 자체를 설계/명세
2. **콘텐츠 디자인** - 캐릭터/스킬/특성 등 콘텐츠의 구성과 동작 설계
3. **구현** - 코드/데이터/툴로 실제 플레이 가능하게 구현
4. **비주얼** - 시각적 결과물 제작/적용
5. **사운드** - 청각적 결과물(SFX/BGM/믹싱) 제작/적용
6. **스케일링 디자인** - 난이도/성장/보상 곡선 설계, 밸런스 튜닝
7. **테스트** - 기능 검증, 결함 발견/재현/보고

---

## 분해 방법: 동작 트리 + 상호작용 + 보조 관점 체크

세 가지 분해 축을 조합하여 항목을 도출한다:

1. **동작 트리** — 대상이 내부적으로 무엇을 하는가
2. **상호작용** — 대상이 외부 시스템과 무엇을 주고받는가
3. **보조 관점** — 동작과 상호작용에서 직접 파생되지 않는 항목 보완

### 1. 동작 트리

- 게임 기능의 본질은 "무엇이 어떻게 동작하는가"이다.
- 데이터, UI, 시스템 연동 등은 동작을 지원하기 위해 존재한다.
- 따라서 **동작이 주축**이 되고, 나머지는 파생된다.

**동작 노드 vs 시스템 노드 구분:**

트리의 노드는 대상이 실제로 수행하는 **행동**이어야 한다. 행동을 지원하는 시스템/메커니즘은 트리의 상위 노드가 아니라, 동작의 하위 또는 기반으로 위치한다.

| 구분 | 노드 형태 | 예시 |
|------|----------|------|
| **동작 노드** (올바름) | "~한다" 형태의 행동 | 대기한다, 공격한다, 피격당한다, 다운된다 |
| **시스템 노드** (잘못됨) | "~시스템", "~계산", "~규칙" | 스탯 시스템, 데미지 계산, 전투 규칙 |

### 2. 상호작용

동작 트리는 대상의 **내부 행동**을 잡지만, **외부 시스템과의 연결 지점**은 직접 드러나지 않는다. 상호작용 분석은 이 연결 지점을 체계적으로 도출한다.

#### 2-1. 상호작용 대상 찾기

대상을 분류별로 탐색하여 빠짐없이 열거한다.

| 대상 분류 | 설명 | 탐색 질문 |
|-----------|------|----------|
| **인접 기능** | 동작이 맞물리는 다른 기능 시스템 | 이 기능과 직접 연동되는 다른 에픽/기능은? |
| **상위 흐름** | 이 기능의 시작/종료를 제어하는 시스템 | 이 기능은 언제 활성화되고, 누가 그걸 결정하는가? |
| **공유 시스템** | 여러 기능이 공통으로 사용하는 프레임워크 | 이 기능이 사용하는 공통 시스템(이벤트, 풀링, 수정자 등)은? |
| **데이터** | 동작을 결정하는 설정/정의 데이터 | 이 기능의 규칙/수치를 정의하는 데이터 에셋은? |
| **플레이어** | 플레이어 조작이 이 기능에 미치는 영향 | 플레이어의 어떤 행동이 이 기능을 변화시키는가? |

#### 2-2. 상호작용 내용 찾기

각 대상에 대해 내용을 분류별 × 방향별로 분석한다.

| 내용 분류 | 설명 | ← 받는 | → 보내는 |
|-----------|------|--------|---------|
| **트리거** | 동작의 시작/중단을 촉발하는 이벤트 | 어떤 외부 이벤트가 이 기능의 동작을 촉발? | 이 기능이 어떤 이벤트를 외부에 알리는가? |
| **데이터 흐름** | 동작에 필요한/산출하는 데이터 | 동작에 필요한 외부 데이터는? | 외부가 참조하는 이 기능의 데이터는? |
| **효과** | 상태 변경 | 외부가 이 기능의 상태를 어떻게 바꾸는가? | 이 기능이 외부 상태를 어떻게 바꾸는가? |

### 3. 보조 관점

- 동작 트리와 상호작용으로도 직접 파생되지 않는 것들(데이터 구조, 정보 표시, 선행 작업)은 별도 관점으로 잡는다.

| 관점 | 역할 | 질문 |
|------|------|------|
| **데이터/속성** | 동작에서 직접 안 보이는 데이터 구조 도출 | 이 기능에 필요한 속성/설정값은? |
| **UI/정보 표시** | 플레이어에게 보여줄 것 도출 | 플레이어에게 어떤 정보를 전달해야 하는가? |
| **기반 작업** | 선행되어야 할 작업 도출 | 동작과 무관하게 먼저 만들어야 할 것은? |

## 워크플로우

분해는 두 단계로 나뉜다:

- **Phase A**: Epic → Task 도출 및 등록
- **Phase B**: Task별 Sub-task 도출 및 등록 (Task마다 반복)

---

### Phase A: Epic → Task

#### 1단계: 기획서 파악

- 해당 에픽 관련 기획서를 읽고 전체 그림을 이해한다.

#### 2단계: 에픽 경계 확인

기획서에서 대상 에픽과 연결된 시스템/기능을 추출하여 인접 에픽을 도출한다.

##### 2-1. 기획서 기반 인접 에픽 도출

1. 기획서에서 대상 에픽의 키워드가 등장하는 **모든 문맥**을 수집한다.
2. 각 문맥에서 연결된 **다른 시스템/기능**을 추출한다.
3. 추출된 시스템을 Jira 에픽 목록에 매핑하여 인접 에픽을 도출한다.

##### 2-2. 경계 판단

- 인접 에픽과의 경계를 판단한다.
- 공통 시스템은 공통 에픽에, 고유 동작만 이 에픽에 포함한다.

#### 3단계: Task 도출

##### 3-1. 상위 동작 트리 작성

- 대상의 **라이프사이클(생성 → 행동 → 소멸)**에서 출발하여, 공통 동작을 포함한 1~2단계 깊이의 트리를 작성한다.
- **시작 질문**: "이 대상은 게임에서 실제로 무엇을 하는가?" — 행동의 흐름을 먼저 나열한 뒤 트리로 구성한다.
- **주의**: 기획서의 목차/항목 구조를 그대로 트리로 변환하지 않는다. 기획서는 참조하되, 트리의 축은 대상의 동작 흐름이다.
- 각 노드에 아래 이모지 태그를 붙여 상태를 구분한다:

| 이모지 | 의미 |
|--------|------|
| 💻 | 구현 대상 (이 에픽에서 새로 구현) |
| 🧩 | 공통 에픽 소속 (구현 여부 무관) |
| ✅ | 중복 (다른 에픽에서 이미 구현 완료) |
| ❓ | 기획 누락 (기획서에 명세 없음) |

- **주의: 구현 복잡도와 동작 노드를 혼동하지 않는다.** 동작 트리는 "게임에서 일어나는 행동"을 나열하는 것이지, "코드로 구현할 작업"을 나열하는 것이 아니다. 구현이 단순하더라도(데이터 설정, 필드 하나 읽기 등) 기획서에서 명시된 행동이면 노드에 포함한다. 구현 복잡도는 이후 업무 분류(5-4)에서 판단한다.
- **대칭성 체크**: 트리 작성 후, 기획서에서 병렬/대칭으로 기술된 개념(A와 B, N개 종류 등)이 트리에도 모두 대응되는지 확인한다.
  - 예: "소환술사 효과와 소환수 특성 두 종류" → 두 가지 모두 트리에 있는가?
  - 예: "공격/방어/지원 3가지 모드" → 세 가지 모두 트리에 있는가?

##### 3-2. 상호작용 분석 (상위)

- 2단계에서 확인한 인접 에픽을 포함하여, 5개 대상 분류별로 상호작용 대상을 열거한다.
- 각 대상과의 상호작용 내용(트리거/데이터 흐름/효과)을 개요 수준으로 파악한다.
- 이 단계에서는 상세 매트릭스가 아닌, **주요 연결 지점의 방향과 성격**을 파악하는 것이 목표이다.

##### 3-3. Task 경계 판단

상위 동작 트리의 노드와 상호작용 분석 결과를 아래 기준으로 Task 단위로 묶는다:

- **독립성** - 다른 Task와 겹치지 않는가
- **응집성** - 내부 동작들이 하나의 기능으로 묶이는가
- **크기** - 너무 크면 분할한다

**사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 4단계: Task Jira 등록

- 도출된 Task를 해당 에픽의 하위 작업으로 Jira에 등록한다.

---

### Phase B: Task → Sub-task (Task마다 반복)

대상 Task를 선택한 뒤, 아래 단계를 수행한다.

#### 5단계: Sub-task 도출

##### 5-1. 동작 트리 상세화

- Task 내의 동작을 기획서에 근거하여 말단 노드까지 분해한다.
- **기획서에 없는 부분 → 기획 누락으로 기록한다.**

##### 5-2. 상호작용 분석 (상세)

- 3-2에서 파악한 상호작용 대상에 대해, 이 Task 범위 내에서 **대상별 × 내용별 매트릭스**를 작성한다.
- 각 대상에 대해 트리거(←→), 데이터 흐름(←→), 효과(←→) 6가지 방향을 체크한다.
- 도출된 연결 지점 중 **이 에픽에서 구현해야 할 항목**을 식별한다.

##### 5-3. 보조 관점 체크

각 동작 노드 + Task 전체에 대해 3개 관점으로 질문한다:

- **데이터/속성**: 이 동작에 필요한 속성/설정값은?
- **UI/정보 표시**: 이 동작/상태와 관련하여 표시 가능한 정보를 항목화한다.
  - 단순히 "UI가 없다"가 아니라, **표시 가능한 정보를 먼저 항목화**한 뒤, 각 항목별로 **플레이어에게 표시할 필요가 있는지를 판별**한다.
  - 표시가 필요한 항목에 대해 기획서와 대조하여 누락을 판단한다.
- **기반 작업**: 동작과 무관하게 선행되어야 할 작업은?

**답이 기획서에 없는 항목 → 기획 누락으로 기록한다.**

##### 5-4. 업무 분류 및 분류

5-1의 말단 노드, 5-2의 상호작용 항목, 5-3에서 도출된 항목 각각에 업무 분류를 부여한다.

도출된 전체 항목을 사용자에게 보여준 뒤, 아래 기준으로 분류한다:

- **구현** → Sub-task 목록에 포함
- **게임플레이 기획** (선행 필요) → 기획 누락으로 분류
- **그 외 업무 분류** (콘텐츠 디자인, 스케일링 디자인, 비주얼, 사운드 등) → 해당 분류의 후속 작업으로 분리

**주의: 항목이 여러 업무 분류에 걸치는 경우**

하나의 항목이 둘 이상의 업무 분류를 포함할 수 있다. 이 경우 항목을 분리한다.

- 예: "성급 표시" → 표시 UI를 만드는 것(구현) + 표시 디자인을 결정하는 것(비주얼)
- 예: "스탯 테이블 설계" → 데이터 구조를 만드는 것(구현) + 수치/곡선을 결정하는 것(스케일링 디자인)

**사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 6단계: 검증

- 기획서를 다시 읽으며 빠진 내용이 없는지 대조한다.
- 이슈 간 중복이 없는지 확인한다.
- **사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 7단계: 이슈 목록 정리

아래 형식으로 정리하여 사용자에게 보여준다:

```
## Task: {Task 이름}

### 도출된 전체 항목

| # | 업무 분류 | 제목 |
|---|----------|------|
| 1 | 구현 | ... |
| 2 | 구현 | ... |
| 3 | 게임플레이 기획 | ... |
| 4 | 스케일링 디자인 | ... |

### → Sub-task (구현)

| # | 유형 | 제목 | 기획 상태 |
|---|------|------|----------|
| 1 | 하위 스토리 | ... | 구현 가능 |
| 2 | 하위 작업 | ... | 기획 필요 |

### → 기획 누락 (게임플레이 기획)

| # | 누락 항목 | 상세 내용 |
|---|----------|----------|
| 1 | ... | ... |

### → 후속 작업 (기타 업무 분류)

| # | 업무 분류 | 제목 |
|---|----------|------|
| 1 | 스케일링 디자인 | ... |
```

**사용자 검증을 받은 후 다음 단계로 진행한다.**

##### 7-1. 기획 누락 항목의 Sub-task 등록 판단

기획 누락 목록을 사용자에게 보여준 뒤, 각 항목에 대해 **기획 결정과 무관하게 구현이 필요한지** 확인한다.

- **구현이 확실한 항목** → Sub-task로 등록하되, Jira 상태를 **"기획 필요"**로 설정한다.
  - 예: "라운드 간 다운 유닛 처리" — 기획 결정이 어떻게 되든 구현은 반드시 필요
- **기획 판단에 따라 구현 여부가 달라지는 항목** → Sub-task로 등록하지 않고 기획 누락으로만 기록한다.
  - 예: "행동 상태 UI 표시 여부" — 표시하지 않기로 결정되면 구현 불필요

**사용자 검증을 받은 후 다음 단계로 진행한다.**

#### 8단계: Sub-task Jira 등록

- 도출된 Sub-task를 해당 Task의 하위 작업으로 Jira에 등록한다.
- 7-1에서 구현이 확실하다고 판단된 기획 누락 항목도 Sub-task로 등록하고, Jira 상태를 **"기획 필요"**로 설정한다.
- Task의 **설명**에 이 Task가 다루는 범위를 간략히 기술한다.
- Task에 **댓글**로 도출 근거(동작 트리, 상호작용 분석, 보조 관점 체크)를 남긴다. (형식은 `구현 이슈 작성 요령.md` 참고)

### 교차 검증 (Phase B와 병렬 진행)

Phase B(Task별 Sub-task 도출)의 구조적 한계를 보완하기 위한 검증 단계이다.

#### 목적

Phase A에서 Task를 놓치면, Phase B가 해당 영역을 탐색하지 않아 누락이 전파된다. 교차 검증은 **모든 분해 축을 한번에 적용한 독립 결과**로 이를 검증한다.

#### 방법

Phase B 시작 시, **별도 에이전트**로 아래를 병렬 수행한다:

1. 모든 분해 축(동작 트리 + 상호작용 + 보조 관점)을 **한번에** 적용하여 플랫 리스트를 도출한다.
2. 각 항목은 **기능/동작 수준**으로 작성한다. 코드 구현 단위(클래스명, 메서드, 코드 패턴)로 작성하지 않는다.

#### 검증

Phase B 완료 후, 플랫 리스트를 Task/Sub-task 구조와 대조한다:

- 플랫 리스트의 항목이 Task/Sub-task에 **모두 매핑**되면 → 누락 없음
- **매핑 안 되는 항목**이 있으면 → 누락 수준을 판별한다:
  - **기존 Task에 속할 수 있는 항목** → Sub-task 누락 후보
  - **기존 Task 어디에도 속하지 않는 항목** → Task 누락 후보 (Phase A에서 Task 자체를 놓친 것)

**사용자가 각 누락 후보를 검토하여 추가할지 기각할지 결정한다.**
