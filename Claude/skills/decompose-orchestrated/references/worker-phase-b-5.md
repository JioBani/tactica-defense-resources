# 워커: Sub-task 도출 (5단계, Task별)

이 파일은 서브에이전트가 직접 읽고 수행하는 지시서이다.
하나의 Task에 대한 Sub-task 도출을 수행한다.

## 입력 파라미터

- **에픽 티켓 번호**: 프롬프트에서 전달받음
- **Task 티켓 번호**: 프롬프트에서 전달받음
- **Task 이름**: 프롬프트에서 전달받음
- **Phase A 결과**: 분해결과.md 경로 (기획서 경로, 에픽 경계 포함)
- **작업 파일**: 이 Task의 결과를 기록할 파일 경로

## 사전 읽기

1. **Phase A 결과** (분해결과.md)를 읽어 기획서 경로, 에픽 경계, 상위 동작 트리를 확인한다.
2. **기획서**: 분해결과.md 1단계에 기록된 기획서 경로의 원문을 직접 읽는다. (요약이 아닌 원문)
3. **방법론**: `Document/프로젝트 관리/구현 이슈 분해 방법론.md`의 "5단계: Sub-task 도출" 섹션을 읽는다.
4. **업무 분류 체계**: `Document/프로젝트 관리/업무 분류 체계.md`를 읽는다. (5-4단계에서 사용)

## 수행 절차

방법론 문서의 5단계를 따라 수행한다. 핵심 요약:

### 5-1. 동작 트리 상세화

- 이 Task 내의 동작을 기획서에 근거하여 **말단 노드까지** 분해한다.
- 각 노드에 이모지 태그를 붙인다: 💻 / 🧩 / ✅ / ❓
- 기획서에 없는 부분 → ❓ 기획 누락으로 기록한다.

### 5-2. 상호작용 분석 (상세)

- Phase A의 상호작용 대상에 대해, 이 Task 범위 내에서 **대상별 × 내용별 매트릭스**를 작성한다.
- 각 대상에 대해 트리거(←→), 데이터 흐름(←→), 효과(←→) 6가지 방향을 체크한다.
- 이 에픽에서 구현해야 할 항목을 식별한다.

### 5-3. 보조 관점 체크

각 동작 노드 + Task 전체에 대해 3개 관점으로 질문한다:

- **데이터/속성**: 이 동작에 필요한 속성/설정값은?
- **UI/정보 표시**: 표시 가능한 정보를 먼저 항목화한 뒤, 플레이어에게 표시할 필요가 있는지 판별한다.
- **기반 작업**: 동작과 무관하게 선행되어야 할 작업은?

답이 기획서에 없는 항목 → ❓ 기획 누락으로 기록한다.

### 5-4. 업무 분류 및 분류

업무 분류 체계 문서를 참조하여, 도출된 전체 항목에 업무 분류를 부여한다:

- **구현** → Sub-task 목록에 포함
- **게임플레이 기획** (선행 필요) → 기획 누락으로 분류
- **그 외** (콘텐츠 디자인, 스케일링 디자인, 비주얼, 사운드 등) → 해당 분류의 후속 작업으로 분리

항목이 여러 업무 분류에 걸치면 분리한다.

## 산출물 원칙

산출물(동작 트리, 상호작용 매트릭스, Sub-task 항목)은 **기능/동작 수준**으로만 작성한다.

- 코드베이스 탐색은 맥락 파악을 위해 **허용**한다.
- 단, 산출물에는 클래스명, 메서드명, 코드 패턴, 파일 경로, 구현 용어(이벤트 발행, DTO, 콜백 등)를 포함하지 않는다.
- 플레이어/기획자가 읽어도 이해할 수 있는 **게임 동작** 수준으로 작성한다.
- 나쁜 예: "LifeCrystalAttacker가 DefenseLine 태그를 감지하여 OnRoundLoseEventDto를 발행한다"
- 나쁜 예: "침략자가 라인 끝에 도달하면 전투 패배 이벤트가 발생한다" (이벤트는 구현 용어)
- 좋은 예: "침략자가 라인 끝에 도달하면 전투에서 패배한다"
- **예외**: 작업 자체가 기술적 설계인 경우(데이터 스키마, 프레임워크, 데이터 구조 등)는 기술 용어가 곧 작업 내용이므로 허용한다.
