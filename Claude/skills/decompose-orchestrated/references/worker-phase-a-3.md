# 워커: Task 도출 (3단계)

이 파일은 서브에이전트가 직접 읽고 수행하는 지시서이다.

## 입력 파라미터

- **에픽 티켓 번호**: 프롬프트에서 전달받음
- **작업 파일**: 분해결과.md 경로 (1+2단계 결과가 이미 기록되어 있음)

## 사전 읽기

1. **작업 파일** (분해결과.md)을 읽어 1+2단계 결과를 확인한다.
2. **방법론**: `Document/프로젝트 관리/구현 이슈 분해 방법론.md`의 "3단계: Task 도출" 섹션을 읽는다.
3. **기획서**: 분해결과.md 1단계에 기록된 기획서 경로를 읽는다.

## 수행 절차

방법론 문서의 3단계를 따라 수행한다. 핵심 요약:

### 3-1. 상위 동작 트리 작성

- 대상의 **라이프사이클(생성 → 행동 → 소멸)**에서 출발한다.
- 1~2단계 깊이의 트리를 작성한다.
- 각 노드에 이모지 태그를 붙인다: 💻 구현 대상 / 🧩 공통 에픽 / ✅ 기존 완료 / ❓ 기획 누락
- **주의**: 기획서 목차를 그대로 트리로 변환하지 않는다. 축은 대상의 동작 흐름이다.
- **대칭성 체크**: 기획서에서 병렬/대칭으로 기술된 개념이 트리에 모두 대응되는지 확인한다.

### 3-2. 상호작용 분석 (상위)

방법론 문서의 상호작용 분석 섹션을 참조하여:
- 5개 대상 분류(인접 기능, 상위 흐름, 공유 시스템, 데이터, 플레이어)별로 상호작용 대상을 열거한다.
- 각 대상과의 주요 연결 지점을 개요 수준으로 파악한다.

### 3-3. Task 경계 판단

상위 동작 트리 + 상호작용 분석을 아래 기준으로 Task 단위로 묶는다:
- **독립성** - 다른 Task와 겹치지 않는가
- **응집성** - 내부 동작들이 하나의 기능으로 묶이는가
- **크기** - 너무 크면 분할한다

## 산출물 원칙

산출물(동작 트리, 상호작용 분석, Task 목록)은 **기능/동작 수준**으로만 작성한다.

- 코드베이스 탐색은 맥락 파악을 위해 **허용**한다.
- 단, 산출물에는 클래스명, 메서드명, 코드 패턴, 파일 경로, 구현 용어를 포함하지 않는다.
- 플레이어/기획자가 읽어도 이해할 수 있는 **게임 동작** 수준으로 작성한다.
- 나쁜 예: "침략자가 라인 끝에 도달하면 전투 패배 이벤트가 발생한다" (이벤트는 구현 용어)
- 좋은 예: "침략자가 라인 끝에 도달하면 전투에서 패배한다"
- **예외**: 작업 자체가 기술적 설계인 경우(데이터 스키마, 프레임워크, 데이터 구조 등)는 기술 용어 허용.

## 작업 파일 기록

파일 형식 참조: `Claude/skills/decompose-orchestrated/references/file-templates.md`의 "분해결과.md 형식" 섹션.

기존 1+2단계 내용을 유지하면서, 3단계 결과를 추가한다:
- 상위 동작 트리
- 상호작용 분석 (상위)
- Task 목록 (경계 판단 결과)

상태 표시를 업데이트한다:

```markdown
> 현재 Phase: A
> 다음 작업: ★검증 (Phase A 검증 + 사용자 검증)
```
