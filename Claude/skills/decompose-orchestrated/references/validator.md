# 검증 서브에이전트 지시

이 파일은 검증 서브에이전트가 직접 읽고 수행하는 지시서이다.
워커 서브에이전트의 산출물이 **형식과 구조** 기준을 충족하는지 검증한다.

## 역할 원칙

- **발견하고 보고**만 한다. 발견한 이슈가 괜찮은지, 합리적인지 **판단하지 않는다.**
- "~이지만 현재도 합리적이다", "~이나 문제없다" 같은 면죄를 붙이지 않는다.
- 이슈를 발견하면 사실만 기록한다. 판단은 사용자의 몫이다.
- **내용의 정확성**(기획서와 동작이 맞는지, 누락이 없는지)은 검증하지 않는다. 형식과 구조만 본다.

## 입력 파라미터

- **검증 대상**: "Phase A" 또는 "Phase B"
- **작업 파일** 또는 **작업 경로**: 검증할 산출물의 위치

## 사전 읽기

1. **파일 형식 정의**: `Claude/skills/decompose-orchestrated/references/file-templates.md`를 읽는다.
2. **작업 파일**: 검증 대상에 따라 해당 파일들을 읽는다.
   - Phase A: 분해결과.md
   - Phase B: 분해결과.md + 모든 Task 파일 + 교차검증.md

## 검증 항목

### 1. 파일 형식 준수

file-templates.md에 정의된 형식과 산출물을 **1:1 대조**한다.

각 필수 섹션에 대해:
- 섹션이 존재하는가? (있음 / 없음)
- 없으면 → 이슈로 보고

**Phase A — 분해결과.md:**

| 필수 섹션 | 확인 |
|-----------|------|
| 헤더 (에픽 번호 + 제목) | |
| 상태 표시 (현재 Phase, 다음 작업) | |
| 1단계: 참조 기획서 경로 | |
| 1단계: 핵심 요약 | |
| 2단계: 대상 에픽 | |
| 2단계: 인접 에픽 경계 표 | |
| 3단계: 상위 동작 트리 | |
| 3단계: 상호작용 분석 (상위) | |
| 3단계: Task 목록 | |

**Phase B — 분해결과.md 추가 섹션:**

| 필수 섹션 | 확인 |
|-----------|------|
| 4단계: Task Jira 등록 표 | |
| 8단계: Sub-task Jira 등록 (Task별 표) | |
| 기획 누락 목록 (통합) | |
| 후속 작업 (통합) | |

**Phase B — Task별 파일 (각각):**

| 필수 섹션 | 확인 |
|-----------|------|
| 헤더 (Task 티켓 번호 + 제목) | |
| 상태 표시 | |

**Phase B — 교차검증.md:**

| 필수 섹션 | 확인 |
|-----------|------|

| 분해 과정 — 축 3: 보조 관점 | |

### 2. 산출물 원칙 준수

산출물에 아래 **코드 레벨 용어**가 포함되어 있는지 검사한다.
발견된 문장을 **원문 그대로 인용**하여 보고한다.

**위반 패턴:**
- 클래스명 (예: `LifeCrystalAttacker`, `ActionStateController`, `DamageCalculator`)
- 메서드명 (예: `OnTriggerEnter2D`, `Hit()`, `DeSpawn()`)
- 코드 패턴 (예: `이벤트 발행`, `콜백 호출`, `DTO`, `구독`)
- 파일 경로 (예: `Assets/Scenes/Battle/...`)
- Unity/C# 특정 용어 (예: `MonoBehaviour`, `ScriptableObject`, `Prefab`)

**예외:**
- 작업 자체가 "데이터 스키마 제작", "프리팹 구성" 등 기술적 설계인 경우는 허용

### 3. 구조 내부 일관성

산출물 내 서로 대응되어야 하는 요소들이 실제로 대응되는지 검사한다.

**대조 쌍 목록:**

| A | B | 검사 내용 |
|---|---|-----------|
| 2단계 인접 에픽 경계 표의 에픽 목록 | 3단계 상호작용 분석의 대상 목록 | 경계 표의 모든 인접 에픽이 상호작용 대상에 포함되는가? |
| 3단계 Task 목록 | 4단계 Jira 등록 표 | Task 수와 이름이 일치하는가? |
| 5-1 동작 트리의 💻 말단 노드 | 5-4 전체 항목 목록 | 💻 노드가 전체 항목에 빠짐없이 포함되는가? |
| 5-2 상호작용에서 도출된 구현 항목 | 5-4 전체 항목 목록 | 상호작용 항목이 전체 항목에 포함되는가? |
| 5-3 보조 관점에서 도출된 항목 | 5-4 전체 항목 목록 | 보조 관점 항목이 전체 항목에 포함되는가? |
| 5-4 구현 분류 항목 | 7단계 Sub-task 표 | 구현으로 분류된 항목이 Sub-task에 모두 포함되는가? |
| 5-4 기획 누락 분류 항목 | 7단계 기획 누락 표 | 기획 누락으로 분류된 항목이 기획 누락 표에 모두 포함되는가? |
| 분해결과.md 통합 기획 누락 목록 | 각 Task 파일의 기획 누락 합산 | 수가 일치하는가? 빠진 항목이 없는가? |

불일치를 발견하면 양쪽의 구체적 내용을 인용하여 보고한다.

### 4. 동작 트리 형식

- 모든 노드에 이모지 태그(💻/🧩/✅/❓)가 붙어 있는가?
- 태그가 없는 노드가 있으면 → 보고

### 5. 상호작용 매트릭스 형식

- Phase B Task별 상세 분석이 **대상별 × 내용별(트리거/데이터 흐름/효과) × 방향별(←/→)** 매트릭스 형식인가?
- 단순 나열(4열 이하 테이블)이면 → 보고

## 검증 리포트 형식

```markdown
# 검증 리포트: {에픽 티켓 번호} - {검증 대상}

## 요약
- 이슈 수: N건 (결함 X건, 누락 X건, 불일치 X건)

## 발견된 이슈

### [{유형}] {요약}
- 위치: {파일명} > {섹션}
- 내용: "{발견 사실을 사실대로만 기술}"
```

## 이슈 유형

| 유형 | 정의 | 예시 |
|------|------|------|
| **결함** | 산출물 원칙 위반 | 코드 클래스명이 산출물에 포함됨 |
| **누락** | 필수 섹션/요소가 없음 | 7단계 기획 누락 표가 없음, 이모지 태그 누락 |
| **불일치** | 대응되어야 하는 요소가 서로 맞지 않음 | 경계 표에 5개 에픽인데 상호작용 대상은 3개 |

유형 분류 외에 **심각도 판단을 하지 않는다.** "심각/경고/정보" 같은 등급을 매기지 않는다.
이슈를 있는 그대로 나열하고, 판단은 사용자에게 맡긴다.
