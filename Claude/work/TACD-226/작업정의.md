# TACD-226: 소환수 인스턴스 초기화 처리

## 작업 내용

소환 터미널에서 구매 시 소환수 인스턴스를 생성하고 초기화하는 전체 흐름을 구현/정비한다.
현재 초기화 흐름이 부분적으로 존재하지만, **성급 전달 경로가 없고**, 초기화 책임이 분산되어 있다.

## 완료 정의

- 소환수 구매 시 인스턴스가 정상 생성되고 **초기 성급(1성 기본, 마켓에 따라 2성/3성 가능)**에 따른 스탯이 설정된다
- 관리 목록(DefenderManager.units)에 인스턴스가 등록된다
- 성급이 1성이 아닌 경우에도 HP바 테두리 색, 스탯이 올바르게 반영된다

## 현재 코드 분석

### 현재 구매 → 생성 흐름

```
MarketManager.BuyDefender(UnitLoadOutData)
  → DefenderManager.GenerateDefender(UnitLoadOutData)
    → UnitGenerator.GenerateDefender(UnitLoadOutData)
      → ObjectPooler.Spawn(prefab)
      → Unit.SetSpawn(UnitLoadOutData)  ← star=1 하드코딩
    → Unit.MoveToWaitingArea()
    → units.Add(defender)
    → OnDefenderChange.Invoke(Spawn)
```

### 현재 문제점

1. **성급 전달 경로 부재**: `MarketDefenderSlot`에 성급 정보가 없고, `BuyDefender` → `GenerateDefender` → `SetSpawn` 전체 경로에서 성급 파라미터를 전달하지 않는다. `SetSpawn`은 항상 `star=1`로 초기화한다.
2. **UnitStatSheet.Init 이벤트 중복 등록**: `Init()` 호출 시 `MaxHealth.OnChange += OnMaxHealthChanged`가 매번 추가되어, 풀에서 재사용될 때 이벤트 핸들러가 중복 등록된다.
3. **MarketDefenderSlot에 성급 개념 없음**: 기획서(3.2)에 따르면 배치 상한 레벨에 따라 2성/3성 소환수가 마켓에 등장할 수 있는데, 현재 `MarketDefenderSlot`은 `UnitLoadOutData`만 보유한다.

### 기획서 참고

- **기획서 3.2 소환(구매)**: "배치 상한 증가에 따라서 소환수가 2성 또는 3성으로 등장하는 확률이 생긴다"
- **기획서 2.1 소환수 라이프사이클**: 라운드 종료 시 HP 회복, 쿨다운 초기화 (이 티켓 범위 아님)
