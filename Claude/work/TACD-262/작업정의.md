# TACD-262: 시너지 프레임워크를 SE 프레임워크 기반으로 변경

> 상위 이슈: TACD-251 [전장 - 시너지] - 시너지 계산 및 효과 적용

## 요구사항

현재 시너지 효과 시스템(SynergyEffect, SynergyController, SynergyEffectFactory)은 독자적인 생명주기를 관리한다.
TACD-261에서 범용 StatusEffect(SE) 프레임워크가 구축되었으므로, 시너지 효과를 SE 프레임워크 위에 올려 통합한다.

변경 후에도 시너지 고유의 동작(활성화/비활성화/티어변경)은 유지되어야 한다.

## 배경지식

### 현재 시너지 효과 시스템
- `SynergyEffect`: 추상 클래스. OnSynergyOn/OnSynergyOff/OnTierChange 콜백 보유
- `SynergyController`: Defender에 붙는 MonoBehaviour. 티어 변경 이벤트를 구독하여 SynergyEffect 생명주기 관리
- `SynergyEffectFactory`: definition.Id 기반 switch로 SynergyEffect 인스턴스 생성

### SE 프레임워크 (TACD-261)
- `StatusEffect`: 추상 기본 클래스. OnApply/OnUpdate/OnRemove 생명주기
- `StatusEffectController`: MonoBehaviour. SE 컬렉션 관리, HookProvider 등록/알림
- `StatusEffectDefinitionData`: ScriptableObject. CreateEffect() 팩토리 메서드
- `HookProvider`: SE에 게임 이벤트를 전달하는 플러그인 시스템

### 설계 방향 (논의 결과)

**SynergyManager가 Apply, SSE가 자기 해제:**
1. SynergyManager가 재계산 → 시너지 활성화 감지 → 대상 유닛의 StatusEffectController에 SSE를 직접 Apply
2. SSE는 주입받은 SynergyActivation의 ActiveTier(RxValue)를 구독
3. 티어 변경 시: SSE가 OnChange 콜백에서 자체적으로 값 갱신
4. 시너지 비활성화 시: ActiveTier가 null → SSE가 RequestRemove()로 자기 만료

**SynergyController 제거:** SynergyManager가 직접 Apply하므로 중간 레이어 불필요

**비전 마법사 같은 전체 아군 대상 시너지 해결:**
SynergyManager가 "누구에게 Apply할지" 결정하므로, 시너지 보유 유닛뿐 아니라 전체 아군에게도 Apply 가능.
SSE 내부에서 보유 여부에 따라 차등 값 적용.

## 완료 정의

- SynergyActivation.ActiveTier가 RxValue로 변경되어 SSE가 구독할 수 있다
- 시너지 효과(SSE)가 StatusEffect를 상속하고, ActiveTier 구독으로 티어 변경/해제를 자체 처리한다
- SynergyManager가 시너지 활성화 시 대상 유닛의 StatusEffectController에 SSE를 직접 Apply한다
- SynergyController가 제거된다
- SynergyDefinitionData가 CreateEffect()를 통해 SSE 인스턴스를 생성한다
- 기존 테스트가 통과하고, 변경된 구조에 대한 테스트가 추가된다
