# 데이터 구조 설계 워크플로우

> 이 문서는 TACD-249 작업을 진행하며, 데이터 구조 설계 시 사용자와 AI가 협업하는 과정을 기록한다.
> 추후 데이터 설계용 스킬(skill)로 정제할 원본 자료이다.

---

## Phase 1: 컨텍스트 수집

> 설계에 필요한 정보를 모으는 단계. 1~3은 순서가 고정되지 않으며, 상황에 따라 병렬 진행하거나 순서를 바꿀 수 있다.

### 1. 요구사항 수집

**입력**: Jira 티켓 (작업 목표, 설명), 기획서 (도메인 규칙, 용어 정의)

**수행**
- Jira 티켓에서 작업 범위와 완료 정의를 파악한다
- 관련 기획서에서 도메인 규칙, 용어, 제약 조건을 수집한다
- 수집한 내용을 정리하여 사용자와 확인한다

**산출물**: 요구사항 요약 (도메인 규칙, 용어 정의, 제약 조건)

### 2. 기존 코드 분석

**입력**: CLAUDE.md (아키텍처 개요), 기존 코드베이스

**수행**
- 관련 기존 데이터 구조, 패턴, 컨벤션을 파악한다
- 새 데이터 구조가 연결될 기존 시스템의 인터페이스를 확인한다
- 활용 가능한 기존 인프라(enum, 수정자 시스템 등)를 식별한다

**산출물**: 기존 코드 참조 요약 (관련 클래스, 패턴, 확장 포인트)

### 3. 외부 리서치 (필요 시)

**수행**
- 유사 시스템의 데이터 구조를 조사한다 (예: TFT 트레이트 구조)
- 잘 알려진 게임/시스템을 구현할 때, 검증된 패턴을 참고한다
- 리서치 결과를 요약하여 설계 참고 자료로 정리한다

**산출물**: 외부 참조 요약 (패턴, 구조적 인사이트)

---

## Phase 2: 데이터 정의

> 수집한 정보를 정리하고, 구체적인 필드로 확정하는 단계.

### 4. 데이터 케이스 분류

**수행**
- 수집한 정보를 아래 3가지로 분류한다:
  - **표현 데이터**: UI에 보여줄 정보 (이름, 설명, 아이콘 등)
  - **로직 데이터**: 런타임에서 계산/판정에 사용하는 정보 (타입, 임계치, 상수 등)
  - **추정 데이터**: 기획서/이슈에 명시되지 않았지만 일반적으로 필요한 정보
- 추정 데이터는 사용자와 범위 확인 → 포함 여부를 결정한다

**산출물**: 데이터 케이스 분류표 (3분류 + 범위 결정)

### 5. 필드 목록 확정

**수행**
- 4단계의 분류 결과를 구체적인 필드(이름, 타입)로 변환한다
- 각 필드가 어느 구조체/클래스에 속할지 1차 배치한다
- 사용자와 최종 확인한다

**산출물**: 확정된 필드 목록 (필드명, 타입, 소속 구조체)

---

## Phase 3: 설계 및 구현

> 확정된 필드를 구조화하고, 코드로 만드는 단계.

### 6. 초안 설계

**수행**
- 확정된 필드를 struct/class/enum으로 구조화한다
- 각 구조체/클래스의 역할과 관계를 명시한다
- ERD 또는 코드 스케치로 시각화한다
- 설계 결정의 근거를 기록한다

**산출물**: 데이터 구조 다이어그램 또는 코드 스케치, 설계 결정 사항 목록

### 7. 리뷰 → 구현

**수행**
- 초안을 사용자에게 제시하고 피드백을 받는다
- 피드백을 반영하여 설계를 수정한다 (필요 시 6~7단계 반복)
- 확정된 설계를 코드로 구현한다
- 컴파일 확인

**산출물**: 구현된 코드

## 진행 기록

### Phase 1: 컨텍스트 수집

#### 1. 요구사항 수집

**Jira 티켓 TACD-249**
- 시너지 시스템 전체가 의존하는 기반 데이터 구조를 설계·제작

**기획서 핵심 규칙**
- 시너지 2종류: 소환술사 효과 (고정) / 소환수 특성 (전장마다 랜덤)
- 소환수 1체 = 소환술사 효과 1개 + 소환수 특성 1개
- 카운트 대상: 전장(BattleArea) 배치된 서로 다른 종류의 소환수 (중복 X)
- 임계치: 소환술사 효과 2/4/6/8, 소환수 특성 2/3/4
- 효과 범위: 해당 시너지를 가진 소환수에게만 적용
- 효과 형태: 스탯 변경 + 행동/메카닉 변경 (다양한 형태 가능)
- 갱신 타이밍: 배치/제거 시 재계산, 전투 중 다운 시 카운트 유지

#### 2. 기존 코드 분석

**활용 가능한 기존 인프라**
- `UnitDefinitionData.Organization` (string) - TODO: enum 변경 예정 → 시너지 필드 추가 지점
- `UnitStat` + `StatModifier` 시스템 - Source 기반 수정자 추가/제거
- `UnitStatKind` enum - 14종 능력치
- `StatModifierType` enum - Flat / Percent
- `DefenderManager.Defenders` - 배치된 유닛 목록
- `GlobalEventBus` - 이벤트 기반 통신

**기존 패턴**
- ScriptableObject 기반 데이터 정의
- enum + switch 기반 조회 (UnitStatKind → UnitStat 매핑)
- 컴포지트 데이터: UnitLoadOutData = UnitDefinitionData + SkillDefinitionData + UnitStatsByLevelData

#### 3. 외부 리서치

**TFT 트레이트 데이터 구조** (CommunityDragon)
- TFT는 named constants (name + value) 패턴 사용
- 데이터는 수치만 제공, 트레이트별 코드가 해석
- 복잡한 효과(소환, 부활, 헥스 생성 등)도 동일한 구조로 표현
- 상수에 속성(타입, 포맷 등) 메타데이터 없음

### Phase 2: 데이터 정의

#### 4. 데이터 케이스 분류

**표현 데이터** (UI 표시용)
- 시너지 이름, 설명 (플레이스홀더 포함), 아이콘
- 티어 임계치 표시

**로직 데이터** (런타임 계산/판정용)
- 시너지 종류 (소환술사 효과 / 소환수 특성)
- 티어별 임계치, 티어별 상수 (name + value)

**추정 데이터** (명시되지 않았지만 일반적으로 필요)
- 역할군, 티어 시각 등급 → **이번 범위 외로 결정**

#### 5. 필드 목록 확정

**SynergyType** (enum): SummonerEffect, SummonTrait
**SynergyTier** (struct): requiredCount (int), constants (SerializableDictionary<string, float>)
**SynergyDefinitionData** (ScriptableObject): id (int), displayName (string), description (string), icon (Sprite), synergyType (SynergyType), tiers (List<SynergyTier>)
**UnitDefinitionData** (수정): summonerEffect (SynergyDefinitionData) 추가

**용어 확정**
- 소환수 = Summon (Summoner가 Summon을 소환)
- SynergyType: SummonerEffect / SummonTrait

### Phase 3: 설계 및 구현

#### 6. 초안 설계

**v1 초안** (`데이터구조-초안.html`)
- SynergyStatEffect (statKind + modifierType + value) 기반
- 데이터가 "어떤 스탯에 어떻게 적용할지"까지 정의하는 방식

**v2 수정** (`데이터구조-v2.html`)
- SynergyStatEffect → SynergyConstant (name + value)로 교체 (TFT 패턴 채택)
- description에 `@Name*100@` 플레이스홀더 도입
- ISynergyEffect 인터페이스 + 팩토리 패턴 추가 (기존 ISkill 패턴과 동일)
- SynergyConstant struct → SerializableDictionary<string, float>로 교체 (O(1) 조회)

**사용자 설계 방침**
1. 시너지마다 구현 클래스를 따로 만들 예정
2. 티어별 수치 표현 가능한 것은 데이터에, 안되는 것은 코드에 분기처리
3. 시너지 설명에 플레이스홀더로 티어별 수치 자동 표시 예정
