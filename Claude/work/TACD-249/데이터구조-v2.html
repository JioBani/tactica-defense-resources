v1 → v2 비교

v1 (초안) — SynergyStatEffect
{
  "tiers": [{
    "requiredCount": 2,
    "statEffects": [
      {
        "statKind": "PhysicalAttack",
        "modifierType": "Percent",
        "value": 0.10
      }
    ]
  }]
}
// 데이터가 "어떤 스탯에 어떻게 적용"까지 정의
// 스탯 보너스만 가능, 행동 변경 불가
v2 (수정) — SynergyConstant
{
  "tiers": [{
    "requiredCount": 2,
    "constants": [
      {
        "name": "AttackBonus",
        "value": 0.10
      }
    ]
  }]
}
// 데이터는 이름 붙은 숫자만 제공
// 구현 클래스가 해석 → 스탯/행동 모두 가능

플레이스홀더 문법

표현의미예시
@Name@ 현재 티어의 상수값을 그대로 표시 @PhysicalDefenseBonus@ → 5
@Name*N@ 상수값에 N을 곱해서 표시 @AttackBonus*100@ → 10 (0.10 × 100)
향후 확장 가능: @Name:format@ (소수점 포맷), @Name+N@ (가산) 등 필요 시 추가

설계 결정 사항

결정근거
SynergyStatEffect → SynergyConstant으로 교체 시너지마다 구현 클래스를 만들 예정이므로, 데이터는 이름 붙은 수치만 제공하고 해석은 코드에 위임. 스탯 보너스 + 행동 변경 모두 동일한 구조로 표현 가능
description에 플레이스홀더 도입 티어별 수치가 달라지므로 @Name@으로 참조. UI에서 현재 티어의 상수를 자동 치환하여 표시
ISynergyEffect 인터페이스 + 팩토리 패턴 기존 스킬 시스템(ISkill + SkillFactory)과 동일한 패턴. 시너지별 구현 클래스를 팩토리가 SynergyDefinitionData.id 기반으로 매핑
SynergyConstant은 name(string) + value(float) TFT와 동일한 패턴. 의미는 구현 클래스만 알면 되므로 데이터 구조는 단순하게 유지
상위 티어는 하위 티어를 교체 (누적 아님) 달성한 최고 티어의 constants만 적용. 구현 클래스의 Apply/Remove가 이를 처리
소환수 특성은 UnitDefinitionData에 미포함 전장마다 랜덤 부여 → 런타임 매핑 (이번 작업 범위 외)
열린 질문 (v1에서 유지): 기존 organization 필드(string)를 summonerEffect로 대체할 것인지, 별도 필드로 추가할 것인지?
