## 방법론별 도출

### 1. 유스케이스 테스트

#### UC-1: Factory 분기 (DoD-S7)

- 정상 흐름:
  - **TC-001**: `SynergyActivation`(SynergyType=SummonTrait) → `Factory.Create()` 가 `SummonTraitSynergyController` 인스턴스 반환.
  - **TC-002**: `SynergyActivation`(SynergyType=SummonerEffect) → 기존 SynergyId switch 로 라우팅 (예: SynergyId.Bruiser → `BruiserSynergyController`). 회귀 방지.

#### UC-2: NoOpSynergyStatusEffect (DoD-S7)

- 정상 흐름:
  - **TC-101**: `SummonTraitSynergyController.CreateSynergyStatusEffect()` 의 reflection 호출 결과가 `NoOpSynergyStatusEffect` 타입.

#### UC-3: Distributor 분배 완료 이벤트 (DoD-S7 트리거)

- 정상 흐름:
  - **TC-201**: registry / SummonerManager / Store 모두 셋업 + `OnBattleStart` reflection 호출 → `OnSummonTraitsDistributedEventDto` 가 GlobalEventBus 로 정확히 1 회 발행됨 (외부 핸들러로 카운트).
- 예외 흐름:
  - **TC-202**: 가드 실패 케이스(registry null / SummonerManager null / Store null) 각각에서 이벤트 미발행 (LogAssert.Expect + 카운트 == 0). 동치 분할 매개변수화.

#### UC-4: SynergyManager.HandleSummonTraitsDistributed (DoD-S5/S6/S7)

- 정상 흐름:
  - **TC-301**: SummonTraitStore 에 trait 3 종(unique) → SynergyActivations 사전에 3 종 SummonTrait 모두 등록. 각 활성화의 Definition 이 해당 trait.
  - **TC-302**: 같은 입력 → SynergyMembers 사전에 trait → unit 목록 (Store 의 역참조) 등록. unit 수가 분배된 unit 수와 일치.
  - **TC-303**: 같은 입력 → `OnSynergyRecalculatedEventDto` 가 등록된 unique trait 수만큼 발행.
  - **TC-304**: SummonerEffect 활성화가 이미 등록된 상태에서 SummonTrait 분배 처리 → SynergyActivations 에 두 종류 모두 공존 (Count == SummonerEffect 수 + SummonTrait 수).
  - **TC-305**: SummonerEffect 의 SynergyMembers 항목이 이미 채워져 있는 상태에서 SummonTrait 분배 처리 → 기존 SummonerEffect 항목 보존, SummonTrait 항목 추가 (외부 readonly 뷰가 합산 표현).
- 예외 흐름:
  - **TC-306**: SummonTraitStore.Instance == null + `HandleSummonTraitsDistributed` 호출 → `Debug.LogError` + 활성화·역참조 미변경.
  - **TC-307**: SummonTraitStore.All 이 빈 상태에서 핸들러 호출 → 무로그 무동작 (Activations / Members 변화 없음).
  - **TC-308**: SynergyActivations 에 이미 동일 SummonTrait 키 존재 + 핸들러 재호출 → `Debug.LogWarning` + 기존 항목 유지 (continue 로 스킵).

#### UC-6: OQ-9 타이밍 경합 보정 (DoD-S5·S6·S7)

- **TC-501**: SummonTraitStore.All 이 비어 있는 상태에서 SynergyManager.Start() 직접 호출 → 구독은 등록되나 사후 동기 검사에서 0 카운트라 핸들러 미호출 (SynergyActivations 에 SummonTrait 등록 없음). 이후 OnSummonTraitsDistributedEventDto Publish → 핸들러 처리됨.
- **TC-502**: SummonTraitStore.All 에 trait 데이터를 미리 채운 상태에서 SynergyManager.Start() 직접 호출 → 사후 동기 검사가 핸들러를 1 회 동기 호출 → SynergyActivations 에 SummonTrait 등록됨 (이벤트 발행 전이라도 누락 없음).

### 2. 동치 분할 (Equivalence Partitioning)

#### 입력: Distributor 의 가드 변수 (registry / SummonerManager / Store)

- 유효 그룹: 셋 다 셋업 → **TC-201** (이벤트 발행)
- 무효 그룹 (단일 결손): 셋 중 하나 null → **TC-202** (이벤트 미발행, LogError)

#### 입력: SummonTraitStore.All

- 유효 그룹: 비어있지 않은 분배 결과 → **TC-301/302/303**
- 경계 그룹: 빈 (Count == 0) → **TC-307** (무동작 정상 흐름)
- 무효 의존: SummonTraitStore.Instance == null → **TC-306** (LogError)

## 미커버 / 보류 항목

- **UI 자체 표시·클릭 동작**(`SynergyListPanel` / `SynergyIndicator` / `SynergyDetailPanel`): 본 단계 운영 코드 변경 0 (impl 가 미터치). 두 시너지 종류를 분기하지 않고 `SynergyActivations` · `SynergyMembers` 단일 사전에서 받는 구조로 이미 작성되어 있으므로, 사전이 합산 형태로 채워지는 것 자체가 UI 동등 표시·노출의 충분 조건. 따라서 UI 자체의 표시·클릭 동작은 본 단위 테스트가 직접 검증하지 않으며 통합·수동 QA 로 위임. 본 명세는 *사전 합산* 을 통한 black-box 충분 조건만 검증.
- **분배된 SummonTrait 의 `Count` 활성화 (티어 활성화 검증)**: 본 통합 단계는 *활성화 등록·인디케이터 노출 메커니즘* 만 책임. 실제 카운트·티어 갱신은 Defender 가 BattleArea 에 놓일 때 `SynergyController.HandlePlacementChanged` 가 처리 — 그 동작은 기존 `SynergyControllerTests` 가 커버 (회귀 방지 영역). 본 단위에서는 *컨트롤러가 생성되고 활성화가 SynergyActivations 에 등록되는지* 까지 검증.
- **RoundManager.Start() 의 publish 1 줄**: trait-distribution 사이클과 동일 사유 — Start 메서드 의존 다수로 단위 격리 어려움. 분배 → 이벤트 → SynergyManager 처리 통합 경로는 본 명세의 TC-201 / TC-501 / TC-502 로 우회 검증.
