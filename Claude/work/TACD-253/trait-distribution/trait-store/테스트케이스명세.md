# TACD-253 — trait-store 테스트 케이스 명세

## 구현 단위 범위

- 구현 단위 식별자: `trait-store` (SR 그룹 B)
- 대응 클래스: `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitStore.cs`
- 다루는 DoD 묶음:
  - **DoD-S3**: 소환수별 배정된 특성 정보가 전장 중 런타임 데이터로 보관된다. 영속화(파일·DB 저장)되지 않는다.
  - **DoD-S4**: 전장 종료 시 소환수별 특성 데이터가 초기화된다.
- 대응 UR DoD: DoD-U2 (부여된 특성은 해당 전장에서만 유효하며, 전장이 끝나면 초기화되어 다음 전장에서 새로 분배된다.)

## 방법론별 도출

### 1. 유스케이스 테스트

#### UC-1: 분배 결과 런타임 보관 (DoD-S3)

- 정상 흐름:
  - **TC-001**: 1개 키-값을 담은 딕셔너리로 `Initialize` 호출 → `GetTrait(key)` 가 해당 값을 반환한다.
  - **TC-002**: 3개 키-값(서로 다른 UnitLoadOutData·SynergyDefinitionData)을 담은 딕셔너리로 `Initialize` 호출 → 각 키에 대해 `GetTrait` 가 올바른 값을 반환한다.
- 대안 흐름:
  - **TC-003**: 어떤 키도 Initialize 되지 않은 상태(혹은 다른 키들로만 채워진 상태)에서 미배정 키 조회 → `GetTrait` 가 `null` 반환.
  - **TC-004**: `Initialize({k1:v1})` 호출 후 `Initialize({k2:v2})` 호출 → `GetTrait(k1)` = null, `GetTrait(k2)` = v2. (이전 배정은 더 이상 유효하지 않다.)
  - **TC-005**: 빈 딕셔너리로 `Initialize` 호출 → 어떤 키 조회도 null. `All.Count == 0`.
- 예외 흐름:
  - **TC-006**: `Initialize(null)` 호출 → `Debug.LogError` 발생 + 직전 상태 유지(기존 데이터 보존).
  - **TC-007**: `GetTrait(null)` 호출 → `Debug.LogError` 발생 + `null` 반환.
- 선행 조건:
  - **TC-008**: 새 인스턴스 부착 직후(`Initialize` 미호출 상태) `All.Count == 0` 이며 임의 키 조회 시 null. (영속 데이터 없음 — DoD-S3 후반부 단서)

#### UC-2: 전장 종료 자동 초기화 (DoD-S4)

- 정상 흐름:
  - **TC-010**: `Initialize` 로 N 개 항목 적재 → `GlobalEventBus.Publish(new OnBattleWinEventDto())` → `All.Count == 0`, 임의 키 조회 null.
  - **TC-011**: `Initialize` 로 N 개 항목 적재 → `GlobalEventBus.Publish(new OnBattleLoseEventDto())` → `All.Count == 0`, 임의 키 조회 null.
- 대안 흐름:
  - **TC-012**: `Initialize` 후 `Clear()` 직접 호출 → `All.Count == 0`, 임의 키 조회 null.
- 예외 흐름:
  - **TC-013**: 빈 상태(`Initialize` 미호출)에서 `Clear()` 호출 → 예외 없이 무동작, `All.Count == 0` 유지.

### 2. 동치 분할 (Equivalence Partitioning)

#### 입력: `Initialize` 의 traitMap 인자

- 관련 DoD: DoD-S3
- 유효 그룹:
  - 비어있지 않은 딕셔너리 (1개 이상의 키-값) — 대표값: `{k1:v1, k2:v2, k3:v3}` → **TC-002**
  - 빈 딕셔너리 (`Count == 0`) — 대표값: `new Dictionary<...>()` → **TC-005**
- 무효 그룹:
  - null — **TC-006**

#### 입력: `GetTrait` 의 unit 인자

- 관련 DoD: DoD-S3
- 유효 그룹:
  - 배정된 키 — **TC-001 / TC-002**
  - 배정되지 않은 키 — **TC-003**
- 무효 그룹:
  - null — **TC-007**

### 3. 결정 테이블 (Decision Table)

#### 결정 포인트: 전장 종료 이벤트 → 자동 Clear 발동 여부

- 관련 DoD: DoD-S4
- 조건:
  - C1: 컴포넌트가 활성 상태(`enabled == true && gameObject.activeInHierarchy == true`)
  - C2: 이벤트 타입
- 결정 테이블:

| 조합 | C1 (활성)  | C2 (이벤트)             | 기대 결과                       | TC ID  |
|------|------------|-------------------------|---------------------------------|--------|
| 1    | T          | OnBattleWinEventDto     | _traitMap 비워짐                | TC-010 |
| 2    | T          | OnBattleLoseEventDto    | _traitMap 비워짐                | TC-011 |
| 3    | F          | OnBattleWinEventDto     | _traitMap 변화 없음 (구독 해제) | TC-014 |
| 4    | F          | OnBattleLoseEventDto    | _traitMap 변화 없음 (구독 해제) | TC-014 |

- 불가능한 조합 없음.

## 미커버 / 보류 항목

- **인스펙터 설정 (DontDestroyOnLoad = false)**: DD 가 명시한 *Awake 전 false 초기화* 는 인스펙터/씬 자산 수준의 설정 — 코드 단위 테스트로 검증 불가. 씬·프리팹의 인스펙터 값 검사 또는 PlayMode 씬 로드 통합 테스트 영역.
- **영속화 부재 (파일·DB 미저장)**: DoD-S3 후반부의 부정 명제. 직접 양성 단정 불가 — Store 인스턴스 부착 직후 비어 있다(TC-008)는 약한 단서로 대체. 파일·DB 호출 흔적이 없음은 코드 리뷰 / 정적 분석으로 보강.

## 참고: 도메인 컨벤션 / 환경

- 테스트 프레임워크: Unity Test Framework (NUnit 3 기반)
- 테스트 모드: EditMode
- 테스트 위치: `Assets/Tests/Editor/SummonTraitStoreTests.cs`
- 어셈블리: 기본 `Assembly-CSharp-Editor` (프로젝트 컨벤션 — 별도 asmdef 없음)
- namespace: `Tests.Editor`
- 클래스 이름: `SummonTraitStoreTests`
- 메서드 이름: `Test_TC{nnn}_<짧은의도>`
- 픽스처: `ScriptableObject.CreateInstance<UnitLoadOutData>()` / `CreateInstance<SynergyDefinitionData>()` (in-memory, 디스크 저장 금지)
- SceneSingleton·GlobalEventBus 정리: `[SetUp]` 에서 reflection 으로 정적 `SceneSingleton<T>._instance` 와 `GlobalEventBus.Handlers` 모두 리셋. `[TearDown]` 에서 `Object.DestroyImmediate(gameObject)` + 동일 리셋.
- **MonoBehaviour 라이프사이클**: Unity EditMode 테스트 환경에서는 `AddComponent` / `DestroyImmediate` 가 Awake·OnEnable·OnDisable 을 자동 발화하지 않는 것이 본 프로젝트의 실측 동작이다. 따라서 본 테스트는 reflection 으로 `OnEnable` / `OnDisable` 을 명시 호출하여 구독·해제 시점을 시뮬레이션한다 (`InvokeNonPublic` 헬퍼). 이는 프로덕션 시 Unity 가 호출하는 동일 메서드이므로 검증 의미는 보존된다.
- 실행 명령: UnityMCP `run_tests` (mode=EditMode, assembly_names=`Assembly-CSharp-Editor`)
- 로그 검증: `LogAssert.Expect(LogType.Error, regex)` 로 Debug.LogError 발생 검증.
