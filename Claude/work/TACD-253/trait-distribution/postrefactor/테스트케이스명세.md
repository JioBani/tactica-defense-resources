## 방법론별 도출

### 1. 유스케이스 테스트

#### UC-1: Start() 통합 흐름 (DoD-S1·S2·S3·S5·S6·S7)

리팩토링 후 핵심 통합 — SynergyManager.Start() 한 곳에서 SummonerEffect 초기화 + SummonTrait 분배·통합이 동시 수행됨.

- 정상 흐름:
  - **TC-001**: 4 명 소환술사(각 8 체 풀) + 3 개 SummonTrait 자산 → Start 호출 → summonTraitMap 에 32 entries 모두 배정, 각 배정 trait 가 풀의 멤버.
  - **TC-002**: 같은 셋업 → SynergyActivations 사전이 SummonerEffect (예: 1 개) + SummonTrait (분배에 등장한 unique 수) 합산 형태로 채워짐 — 시너지 종류 분기 없는 단일 사전 (UI 동등 표시·상세보기의 black-box 충족 조건).
  - **TC-003**: SynergyMembers 사전에 SummonTrait 별 보유 unit 목록이 등록되며, 기존 SummonerEffect 항목과 같은 사전에 합산.
  - **TC-004**: \_controllers 사전에 각 SummonTrait 에 대해 SummonTraitSynergyController 인스턴스가 생성·등록 (Factory 호출 검증).

#### UC-2: GetSummonTrait public API (DoD-S3)

- **TC-101**: Start 후 임의 배정 unit 으로 `GetSummonTrait` 호출 → summonTraitMap 의 할당값을 반환.
- **TC-102**: 미배정 unit 으로 호출 → null.
- **TC-103**: null 입력 → `Debug.LogError` + null.

#### UC-3: 전장 종료 초기화 (DoD-S4)

- 정상 흐름:
  - **TC-201**: Start 후 summonTraitMap 채워진 상태 → `GlobalEventBus.Publish(new OnBattleWinEventDto())` → `summonTraitMap.Count == 0`.
  - **TC-202**: 동일하게 OnBattleLoseEventDto → `summonTraitMap.Count == 0`.
- 예외 흐름(라이프사이클):
  - **TC-203**: Start 후 `OnDisable` 명시 호출 → 그 후 Battle 이벤트 Publish → summonTraitMap 변화 없음 (구독 해제 검증).

#### UC-5: 가드 (DoD-S2 예외 처리)

- **TC-401**: summonTraitRegistry == null (인스펙터 미연결) + Start 호출 → `Debug.LogError("[SynergyManager] summonTraitRegistry 미연결 — SummonTrait 분배 스킵")` + summonTraitMap 비어 있음 + SynergyActivations 사전에 SummonTrait 미등록 (SummonerEffect 만 정상 초기화).

#### UC-6: Factory routing (DoD-S7)

- **TC-501**: 신규 `SynergyActivation`(synergyType=SummonTrait) → `SynergyControllerFactory.Create` 가 `SummonTraitSynergyController` 인스턴스 반환.
- **TC-502**: 신규 `SynergyActivation`(synergyType=SummonerEffect, id=Bruiser) → `SummonTraitSynergyController` 아님 + `BruiserSynergyController` 인 것 — 회귀 방지.

#### UC-7: NoOpSynergyStatusEffect (DoD-S7)

- **TC-601**: `SummonTraitSynergyController.CreateSynergyStatusEffect` reflection 호출 → 반환 타입이 `NoOpSynergyStatusEffect`.

## 미커버 / 보류 항목

- **`SummonTraitService.Distribute` 의 무효·경계 입력 케이스** (null 풀 / 빈 풀 / null summoner / 빈 SummonPool / null unit 등): 이전 trait-distribution 사이클이 8 케이스로 커버했으나 본 통합 사이클에서는 *SynergyManager 통합* 에 초점. Service 자체의 격리 테스트는 Service 인스턴스를 `new()` 하여 직접 검증 가능하나, 본 명세에서는 *SynergyManager 가 Service 호출 → 결과를 summonTraitMap 에 통합* 의 black-box 통합 검증(TC-001)으로 충분으로 판단. Service 자체 격리 케이스는 차후 회귀 대비 필요 시 별도 백로그.
- **UI 자체 표시·클릭 동작** (`SynergyListPanel` / `SynergyIndicator` / `SynergyDetailPanel`): 본 단계 운영 코드 변경 0. UI 가 단일 사전 소비 구조이므로 `SynergyActivations` / `SynergyMembers` 합산형 채움(TC-002 / TC-003)이 black-box 충분 조건. 통합·수동 QA 영역.
- **분배된 SummonTrait 의 `Count` 활성화 / 티어 활성화** : 본 통합 단계는 *활성화 등록·인디케이터 노출 메커니즘* 만 책임. 실제 카운트·티어 갱신은 Defender 가 BattleArea 에 놓일 때 `SynergyController.HandlePlacementChanged` 가 처리 — 기존 `SynergyControllerTests` 가 커버.
- **균등 분포 통계 검증**: `UnityEngine.Random` 의 균등성 보장에 위임 (이전 사이클 명세와 동일).
