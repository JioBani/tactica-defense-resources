# TACD-253 — 전장 시작 시 소환수 풀에 특성이 랜덤 부여된다

> 이 문서는 시스템 요구사항을 LLM Plan 진입 직전까지 구체화한 설계 문서입니다.

## 개요

- Jira: TACD-253 (상위: TACD-250)
- 신규 클래스 7개 + 이벤트 DTO 2개 추가. 기존 코드 수정 3건: `RoundManager.Start()` 이벤트 발행 / `SummonTraitDistributor` 분배 후 이벤트 발행 / `SynergyManager` SummonTrait 통합 / `SynergyControllerFactory` 분기.

|----|----------|--------|-------------|
| DD-B1 | SummonTraitStore | 런타임 배정 결과 보관·초기화 | DoD-S3, DoD-S4 |
| DD-A1 | SummonTraitRegistry | SummonTrait 자산 목록 보관 SO | DoD-S1 |

## 설계

> **구현 순서**: 그룹 B(trait-store) → 그룹 A(trait-distribution) → 그룹 C(trait-synergy-integration). 그룹 C는 그룹 A·B 산출물(SummonTraitStore·분배 결과)을 SynergyManager에 통합하는 후속 단계.

### 그룹 B (`trait-store`). 분배 결과 런타임 보관 및 초기화

**대응 SR DoD**: DoD-S3, DoD-S4

**설계 접근**: SR 그룹 B가 부여한 책임 — 배정 결과를 전장 중 런타임으로 보관하고 전장 종료 시 초기화 — 을 `SummonTraitStore : SceneSingleton<SummonTraitStore>` 단일 클래스로 풀어낸다. 기존 SceneSingleton 패턴을 따르되 전장 전용(DontDestroyOnLoad 비활성) 데이터로 유지한다.

#### 클래스 분리

- **SummonTraitStore** (신규, `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitStore.cs`)
  - 책임: 소환수별 배정 특성 `Dictionary<UnitLoadOutData, SynergyDefinitionData>`를 전장 중 보관하고, 전장 종료 시 초기화
  - 사유: 여러 시스템(향후 시너지 효과 적용 등)이 런타임에 특성 조회가 필요하므로 SceneSingleton으로 전역 접근 제공. DontDestroyOnLoad는 false로 — 전장 씬에서만 의미있는 데이터.
  - namespace: `Scenes.Battle.Feature.SummonTrait`

#### 핵심 외부 노출 시그니처

```csharp
// Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitStore.cs
public class SummonTraitStore : SceneSingleton<SummonTraitStore>
{
    // 인스펙터에서 런타임 배정 결과를 확인 가능하도록 [SerializeField] 적용
    [SerializeField] private SerializableDictionary<UnitLoadOutData, SynergyDefinitionData> _traitMap = new();

    public SerializableDictionary<UnitLoadOutData, SynergyDefinitionData> All => _traitMap;
    public SynergyDefinitionData GetTrait(UnitLoadOutData unit);
    public void Initialize(Dictionary<UnitLoadOutData, SynergyDefinitionData> traitMap);
    public void Clear();
}
```

- `_traitMap`: `SerializableDictionary`를 `[SerializeField]`로 선언하여 플레이 모드 중 인스펙터에서 배정 결과 확인 가능. `SerializableDictionary`의 `keys[]`·`values[]` 배열 직렬화 구조가 ScriptableObject 레퍼런스 키를 지원함.
- `GetTrait`: 해당 UnitLoadOutData의 배정 특성을 반환. 미배정 시 null 반환.
- `Initialize(Dictionary<>)`: Distribute가 반환한 표준 Dictionary를 받아 내부 `_traitMap`을 초기화. 기존 데이터를 덮어쓴다.
- `Clear`: `_traitMap.Clear()` 호출. 전장 종료 시 호출.
- DontDestroyOnLoad: SceneSingleton의 `[SerializeField] bool _dontDestroyOnLoad`를 Awake 전에 false로 초기화.

#### 핵심 비즈니스 로직

1. `Initialize` 호출 시 전달받은 딕셔너리를 내부 필드에 보관한다.
2. `GetTrait` 호출 시 키로 조회하여 존재하면 반환, 없으면 null 반환한다.
3. `Clear` 호출 시 내부 딕셔너리를 비운다.

#### 예외 처리

| 상황 | 처리 |
|------|------|
| `Initialize(null)` 호출 | `Debug.LogError` + 기존 상태 유지 |
| `GetTrait(null)` 호출 | `Debug.LogError` + null 반환 |
| `Clear` 호출 시 데이터 없음 | 정상 흐름 — 무동작 (초기 상태) |

### 그룹 A (`trait-distribution`). 전장 시작 특성 분배 실행

**대응 SR DoD**: DoD-S1, DoD-S2

**설계 접근**: SR 그룹 A가 부여한 책임 — SummonTrait 자산 인식 + 전투 페이즈 진입 시 32체 균등 랜덤 배정 — 을 (1) `SummonTraitRegistry` SO(자산 풀 보관) + (2) `SummonTraitService` 순수 C#(분배 로직) + (3) `SummonTraitDistributor` MonoBehaviour(Combat 페이즈 훅) 세 클래스로 책임을 분리하여 풀어낸다. MonoBehaviour는 Unity 라이프사이클만 담당하고 비즈니스 로직은 서비스에 위임한다.

#### 클래스 분리

- **SummonTraitRegistry** (신규, `Assets/Common/Data/Synergies/SummonTraitRegistry.cs`)
  - 책임: 분배 대상 SummonTrait SynergyDefinitionData 자산 목록을 인스펙터에서 관리
  - 사유: `Resources.LoadAll` 방식은 자산을 Resources/ 폴더로 이동해야 하고 SynergyType 필터링이 필요해 비용이 큼. 인스펙터 참조 방식(Registry SO)이 기존 프로젝트 ScriptableObject 데이터 주도 패턴과 일치하며 빌드 분리도 보장.
  - namespace: `Common.Data.Synergies`

- **SummonTraitService** (신규, `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitService.cs`)
  - 책임: 소환수 풀 + 특성 풀을 입력받아 균등 랜덤 배정 딕셔너리를 생성
  - 사유: 분배 로직을 순수 C# 클래스로 분리하여 MonoBehaviour와 Unity 의존성 없이 단위 테스트 가능.
  - namespace: `Scenes.Battle.Feature.SummonTrait`

- **OnBattleStartEventDto** (신규, `Assets/Scenes/Battle/Feature/Events/RoundEvents/OnBattleStartEventDto.cs`)
  - 책임: "전장 시작" 신호를 GlobalEventBus로 전파하는 이벤트 DTO
  - 사유: `OnBattleWinEventDto`·`OnBattleLoseEventDto`와 대칭되는 전장 시작 이벤트. `OnStateEnter(Combat)` 은 매 라운드 발생하므로 전장 1회 분배 트리거로 부적합 — GlobalEventBus 이벤트로 전장 시작을 정확히 표현.
  - namespace: `Scenes.Battle.Feature.Events.RoundEvents`

- **RoundManager** (기존 수정, `Assets/Scenes/Battle/Feature/Round/RoundManager.cs`)
  - 변경: `Start()` 메서드 안, `StartRound()` 호출 전에 `GlobalEventBus.Publish(new OnBattleStartEventDto())` 1줄 추가
  - 사유: `StartRound()`는 public이라 매 라운드마다 호출될 수 있어 전장 시작 발행 지점으로 부적합. Unity `Start()`는 씬 로드 시 1회만 실행되므로 전장 시작 신호를 정확히 1회 발행하는 지점.

- **SummonTraitDistributor** (신규, `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitDistributor.cs`)
  - 책임: `OnBattleStartEventDto` 구독 시 분배 실행, `OnBattleWinEventDto`·`OnBattleLoseEventDto` 구독 시 스토어 초기화
  - 사유: IStateListener 대신 GlobalEventBus 구독 방식 — 전장 1회 분배와 전장 종료 초기화를 명시적 이벤트로 표현하여 라운드 반복 문제를 구조적으로 방지.
  - namespace: `Scenes.Battle.Feature.SummonTrait`

#### 핵심 외부 노출 시그니처

```csharp
// Assets/Scenes/Battle/Feature/Events/RoundEvents/OnBattleStartEventDto.cs
public class OnBattleStartEventDto { }

// Assets/Common/Data/Synergies/SummonTraitRegistry.cs
[CreateAssetMenu(menuName = "SummonTrait/SummonTraitRegistry", fileName = "SummonTraitRegistry")]
public class SummonTraitRegistry : ScriptableObject
{
    [SerializeField] private SynergyDefinitionData[] traits;
    public IReadOnlyList<SynergyDefinitionData> Traits => traits;
}

// Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitService.cs
public class SummonTraitService
{
    public Dictionary<UnitLoadOutData, SynergyDefinitionData> Distribute(
        IReadOnlyList<SummonerLoadOutData> summoners,
        IReadOnlyList<SynergyDefinitionData> traitPool);
}

// Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitDistributor.cs
public class SummonTraitDistributor : MonoBehaviour
{
    [SerializeField] private SummonTraitRegistry registry;
    // IStateListener 미사용 — GlobalEventBus 구독으로 전장 시작/종료 처리
}
```

- `OnBattleStartEventDto`: 필드 없는 마커 DTO. `OnBattleWinEventDto`·`OnBattleLoseEventDto`와 대칭.
- `Distribute` 시그니처 결정 사유: 입력·출력이 명확한 순수 함수. 랜덤 시드 관리 불필요 (`UnityEngine.Random`은 Unity가 시드 관리).
- `SummonTraitDistributor`의 `registry` 필드: ScriptableObject 참조로 씬의 인스펙터에서 연결. `SummonTraitRegistry` 에셋 1개를 생성하여 TACD-300에서 만든 특성 8개를 연결할 것.

#### 핵심 비즈니스 로직

**SummonTraitService.Distribute 흐름:**
1. 특성 풀이 비어 있으면 분배를 중단하고 빈 딕셔너리를 반환한다 (예외 처리 참조).
2. 4명의 소환술사(SummonerLoadOutData) 각각에서 `Summoner.SummonPool`(UnitLoadOutData 배열, 최대 8체)을 순회하여 대상 소환수를 수집한다.
3. 각 UnitLoadOutData에 대해 특성 풀에서 균등한 확률로 특성 1개를 무작위 선택하여 딕셔너리에 추가한다. 동일 특성이 복수 소환수에 배정될 수 있다.
4. 완성된 딕셔너리를 반환한다.

> **키 고유성 전제**: `Dictionary<UnitLoadOutData, SynergyDefinitionData>` 키는 UnitLoadOutData 인스턴스 참조. 서로 다른 소환술사가 동일한 UnitLoadOutData 에셋 인스턴스를 공유하면 마지막 배정이 덮어쓴다. 현재 게임 데이터에서는 소환술사별 풀이 독립적이므로 충돌 없음으로 전제한다.

**SummonTraitDistributor 훅 흐름:**
1. `Awake`: `SummonTraitService` 인스턴스 생성.
2. `OnEnable`: `GlobalEventBus.Subscribe<OnBattleStartEventDto>(OnBattleStart)` / `Subscribe<OnBattleWinEventDto>(OnBattleEnd)` / `Subscribe<OnBattleLoseEventDto>(OnBattleEnd)`.
3. `OnDisable`: 위 세 구독 해제.
4. `OnBattleStart`: `SummonTraitService.Distribute(SummonerManager.Instance.Summoners, registry.Traits)` 호출 → 결과를 `SummonTraitStore.Instance.Initialize()`에 전달.
5. `OnBattleEnd`: `SummonTraitStore.Instance.Clear()` 호출.

**RoundManager.Start() 수정:**
- `StartRound()` 호출 전에 `GlobalEventBus.Publish(new OnBattleStartEventDto())` 1줄 추가. Unity `Start()`는 씬 로드 시 1회만 실행 — 전장 시작 신호를 정확히 1회 발행.
- 코드 주석 필수: `// TODO: 전장 초기화 로직(편성 주입·씬 데이터 로드 등)이 추가될 경우 발행 시점을 해당 초기화 완료 이후로 이동 필요`

#### 예외 처리

| 상황 | 처리 |
|------|------|
| `registry`가 null (인스펙터 미연결) | `Debug.LogError("[SummonTraitDistributor] registry 미연결")` + 분배 스킵 |
| `registry.Traits`가 비어 있음 | `Debug.LogError("[SummonTraitService] 특성 풀이 비어 있어 분배 불가")` + 빈 딕셔너리 반환 |
| `SummonerManager.Instance`가 null | `Debug.LogError` + 분배 스킵 |
| `SummonPool`이 null 또는 빈 배열 | `Debug.LogWarning("[SummonTraitService] summonPool 비어있는 소환술사 발견")` + 해당 소환술사 스킵 |
| `UnitLoadOutData` 항목이 null | `Debug.LogWarning` + 해당 항목 스킵 |
| `SummonTraitStore.Instance`가 null | `Debug.LogError("[SummonTraitDistributor] SummonTraitStore 미존재")` + Initialize 스킵 |

### 그룹 C (`trait-synergy-integration`). 소환수 특성 인디케이터 표시 및 상세보기 상호작용

**대응 SR DoD**: DoD-S5, DoD-S6, DoD-S7

**설계 접근**: SR 그룹 C가 부여한 책임 — SummonTrait를 SummonerEffect와 동등하게 시너지 인디케이터·상세보기·집계에 통합 — 을 `SynergyManager`의 활성화 사전(`_synergyActivations`)·역참조 인덱스(`_synergyMembers`)에 SummonTrait도 합산하는 방향으로 푼다. 기존 `SynergyListPanel`·`SynergyIndicator`·`SynergyDetailPanel`은 시너지 종류를 분기하지 않으므로 SynergyManager 통합만으로 UI 표시·선택·상세보기가 자연스럽게 작동한다. SummonTrait는 SE가 비어 있는 자산(TACD-300 결정)이므로 SSE 적용 단계를 NoOp로 채워 기존 `SynergyController` 골격을 그대로 사용한다.

#### 클래스 분리

- **OnSummonTraitsDistributedEventDto** (신규, `Assets/Scenes/Battle/Feature/Events/RoundEvents/OnSummonTraitsDistributedEventDto.cs`)
  - 책임: SummonTraitDistributor가 분배·저장을 마쳤음을 알리는 도메인 이벤트
  - 사유: SynergyManager가 분배 완료 시점을 파악해 활성화 집계에 합치도록 명시적 신호 제공. `OnBattleStartEventDto`와 분리하여 "분배가 끝났다"는 의미를 정확히 표현.
  - namespace: `Scenes.Battle.Feature.Events.RoundEvents`

- **SummonTraitDistributor** (기존 수정, `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitDistributor.cs`)
  - 변경: `OnBattleStart` 핸들러에서 `SummonTraitStore.Instance.Initialize(traitMap)` 호출 직후 `GlobalEventBus.Publish(new OnSummonTraitsDistributedEventDto())` 1줄 추가
  - 사유: 분배 완료 신호를 GlobalEventBus로 전파 — SynergyManager와 직접 결합하지 않고 이벤트 버스 패턴 유지.

- **NoOpSynergyStatusEffect** (신규, `Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyEffects/NoOpSynergyStatusEffect.cs`)
  - 책임: SummonTrait 시너지에 부여되는 비효과 SE. SE 인스턴스만 존재하며 스탯/상태 변화는 일으키지 않음.
  - 사유: 기존 `SynergyController.CreateSynergyStatusEffect()`가 추상 — SummonTraitSynergyController가 반환할 SE 인스턴스 필요. TACD-300에서 SummonTrait용 `StatusEffectDefinitionData` 효과 필드가 비어 있다고 결정됐으므로 SE 효과 동작도 비워둠.
  - namespace: `Scenes.Battle.Feature.Synergy`
  - **코드 주석 요구사항 (파일 상단 역할 주석 필수)**:
    ```
    // ─────────────────────────────────────────────
    // NoOpSynergyStatusEffect: 테스트용 SummonTrait 시너지에 부여되는 비효과 SE.
    // TACD-300 결정에 따라 SummonTrait의 StatusEffectDefinitionData 효과 필드가 비어 있어
    // 카운트·티어 추적은 필요하지만 스탯/상태 변화는 일으키지 않는 placeholder 용도.
    // 실제 효과를 가진 SummonTrait 시너지가 추가되는 시점에는 본 클래스를 사용하지 말고,
    // 기존 SummonerEffect 시너지 패턴(BruiserSynergyStatusEffect 등)처럼 시너지별 전용
    // SynergyStatusEffect 서브클래스를 구현한다.
    // ─────────────────────────────────────────────
    ```

- **SummonTraitSynergyController** (신규, `Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyControllers/SummonTraitSynergyController.cs`)
  - 책임: `SynergyController`를 상속한 SummonTrait 통합 컨트롤러. Defender 배치·판매에 따른 카운트·티어 재계산 흐름은 기존 골격 그대로 사용하며, SSE 생성만 `NoOpSynergyStatusEffect`를 반환.

public struct OnSummonTraitsDistributedEventDto : IGameEvent { }

// Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyEffects/NoOpSynergyStatusEffect.cs
public class NoOpSynergyStatusEffect : SynergyStatusEffect<SynergyStatusEffectContext>
{
    public NoOpSynergyStatusEffect(StatusEffectDefinitionData definition) : base(definition) { }
    // OnSynergyActivated / OnSynergyDeactivated / OnSynergyTierChanged: 빈 본문 (override 생략 가능 — base가 virtual no-op)
}

// Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyControllers/SummonTraitSynergyController.cs
public class SummonTraitSynergyController : SynergyController
{
    public SummonTraitSynergyController(SynergyActivation activation) : base(activation) { }
    protected override SynergyStatusEffect CreateSynergyStatusEffect()
        => new NoOpSynergyStatusEffect(Definition.StatusEffectDefinition);
}

// Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyControllerFactory.cs (수정)
public SynergyController Create(SynergyActivation activation)
{
    if (activation.Definition.SynergyType == SynergyType.SummonTrait)
        return new SummonTraitSynergyController(activation);

    return activation.Definition.Id switch { /* 기존 케이스 유지 */ };
}

// Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyManager.cs (수정 부분)
private void HandleSummonTraitsDistributed(OnSummonTraitsDistributedEventDto _);
// HandleDefenderChanged: SummonerEffect + SummonTrait 둘 다 조회·AddSynergy

- `NoOpSynergyStatusEffect`: `SynergyStatusEffect<SynergyStatusEffectContext>` 기반 클래스의 가상 콜백(OnSynergyActivated 등)이 이미 빈 본문이므로 추가 override 불필요.
- `SummonTraitSynergyController`: SSE 생성 시점에만 NoOp 반환. 카운트·티어 추적은 기존 base 로직 그대로.

#### 핵심 비즈니스 로직

**SummonTraitDistributor 확장 흐름:**
1. (기존) `OnBattleStart`: 분배 실행 → `SummonTraitStore.Instance.Initialize(traitMap)`.
2. (신규) Initialize 직후 `GlobalEventBus.Publish(new OnSummonTraitsDistributedEventDto())` 호출 — 분배 완료 신호 전파.

**SynergyManager.HandleSummonTraitsDistributed 흐름:**
1. `SummonTraitStore.Instance.All`을 통해 UnitLoadOutData → SummonTrait SynergyDefinitionData 맵을 읽는다.
2. 내부 SummonTrait 전용 맵(`_unitSummonTraitMap`)을 새로 채운다 — 기존 `_unitSynergyMap`(SummonerEffect용)과 분리 유지.
3. 분배에 등장한 유니크 SummonTrait 집합을 추출하여 각각 `SynergyActivation` 생성 → `SynergyControllerFactory`로 `SummonTraitSynergyController` 생성 → `_synergyActivations`·`_controllers`에 추가.
4. 각 SummonTrait 활성화에 대해 보유 소환수 목록을 `_synergyMembers`·`_synergyMembersPublic`에 추가(기존 SummonerEffect 항목과 동일 인덱스에 합산).
5. SummonTrait 활성화 항목별로 `OnSynergyRecalculatedEventDto`를 발행 — `SynergyListPanel`이 인디케이터 가시성·정렬을 즉시 갱신.

**SynergyManager.HandleDefenderChanged 확장 흐름:**
1. (기존) `dto.Change == Spawn`이면 `_unitSynergyMap`에서 SummonerEffect 조회 → `dto.Defender.AddSynergy(summonerEffect)`.
2. (신규) 같은 분기에서 `_unitSummonTraitMap`에서 SummonTrait도 조회 → 존재하면 `dto.Defender.AddSynergy(summonTrait)`. 두 호출은 독립이며 한쪽이 미존재여도 다른 쪽은 진행.

**Defender 배치 후 카운트 갱신**: Defender가 BattleArea로 이동하면 기존 `SynergyController.HandlePlacementChanged`가 SummonerEffect 시너지와 SummonTrait 시너지 양쪽 컨트롤러에서 각각 동작 — base 골격이 시너지 종류 무관하게 카운트·티어를 재계산하고 `OnSynergyRecalculatedEventDto`를 발행하므로 인디케이터·상세보기 갱신이 자동.

2. **OQ-2. 런타임 저장 키 타입** — 결정: `Dictionary<UnitLoadOutData, SynergyDefinitionData>` (기존 패턴 준수).

3. **OQ-3. 전장 종료 시 초기화 방법** — 결정: `GlobalEventBus<OnBattleWinEventDto>` + `GlobalEventBus<OnBattleLoseEventDto>` 구독.

4. **OQ-4. 분배 트리거** — 결정: `GlobalEventBus<OnBattleStartEventDto>` 구독. `OnBattleStartEventDto`는 `RoundManager.Start()`(Unity 라이프사이클, 1회 실행)에서 발행.

   - 반영 위치: 그룹 A 클래스 분리 (OnBattleStartEventDto·RoundManager 수정), 비즈니스 로직 (Distributor 훅 흐름 2·4번).

8. **OQ-8. SummonTrait 역참조 인덱스 저장 구조 (그룹 C)** — 결정: SynergyManager에 `_unitSummonTraitMap : Dictionary<UnitLoadOutData, SynergyDefinitionData>`를 SummonerEffect용 `_unitSynergyMap`과 별도로 신설.

   - 반영 위치: 그룹 C 비즈니스 로직 (SynergyManager 변경 2·3번).

9. **OQ-9. SynergyManager.Start ↔ OnSummonTraitsDistributedEventDto 타이밍 경합 처리 (그룹 C)** — 결정: SynergyManager가 OnSummonTraitsDistributedEventDto 구독을 `Start()` 종료 시점(SummonerEffect 초기화 후)에 등록 + 등록 직후 `SummonTraitStore.Instance.All.Count > 0`이면 핸들러 1회 동기 호출로 누락 보정.
   - 사유: Unity Start() 순서가 객체 간 미보장 — RoundManager.Start가 SynergyManager.Start보다 먼저 실행되면 `OnSummonTraitsDistributedEventDto`가 SynergyManager 구독 전에 발행되어 유실 위험. 구독 등록 + 사후 상태 동기 검사 조합으로 두 실행 순서 모두 안전 처리. `[DefaultExecutionOrder]` 사용 대안은 다른 시스템과의 순서 관계까지 영향을 줄 수 있어 회피.
   - 반영 위치: 그룹 C 비즈니스 로직 (SynergyManager 변경 1번 — 구독 위치를 `Start()` 종료부로 명시, 누락 보정 검사 추가).
