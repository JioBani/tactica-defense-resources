# TACD-253 — 코드 탐색 결과 (코드 모드)

> 이 문서는 req 멤버의 위임 요청(SR 작성 단계)에 대한 코드 탐색 응답이며, 도메인 언어로 먼저 답한 뒤 필요한 경우 코드 단계까지 풀어낸 결과입니다.

## 요청 컨텍스트

- **요청자**: req 멤버 (SR 작성 단계)
- **요청 단계**: SR 작성
- **요청 내용**: 전장 시작 트리거 / 소환수 풀 접근 / 소환수 런타임 데이터 구조 / SummonTrait 로드 접근점
- **탐색 범위**:
  - `Assets/Scenes/Battle/Feature/Round/` (RoundManager, PhaseType 등)
  - `Assets/Scenes/Battle/Feature/Unit/Summoner/` (SummonerManager)
  - `Assets/Common/Data/Summoners/` (SummonerDefinitionData, SummonerLoadOutData)
  - `Assets/Scenes/Battle/Feature/Synergy/Scripts/` (SynergyManager, SynergyControllerFactory)
  - `Assets/Common/Data/Synergies/` (SynergyDefinitionData, SynergyType enum)
  - `Assets/Scenes/Battle/Feature/Unit/Unit/Scripts/` (Unit, UnitStatSheet)

## 응답

### 질문 1: 전장 시작 이벤트·트리거

**도메인 표현**: "전장 시작"은 `PhaseType.Combat` 상태 진입(Enter) 시점이다. GlobalEventBus에 전장 시작 전용 이벤트는 없으며, 기존 컴포넌트들은 `IStateListener<PhaseType>` 인터페이스를 구현해 `OnStateEnter(PhaseType.Combat)` 콜백을 받는 방식으로 훅한다.

**코드 구조**:

- **상태 머신**: `Assets/Scenes/Battle/Feature/Round/RoundManager.cs` (`StateBaseController<PhaseType>` 상속)
- **페이즈 흐름**: `Maintenance → Ready (3초 대기) → Combat → End → RoundLose / BattleWin`
- **기존 훅 예시**: `Assets/Scenes/Battle/Feature/Round/RoundAggressorManager.cs`
  ```csharp
  void IStateListener<PhaseType>.OnStateEnter(PhaseType phaseType)
  {
      if (phaseType == PhaseType.Combat)
      {
          OnRoundEnter();  // 침략자 스폰 시작
      }
  }
  ```

**핵심 식별자**:
- 인터페이스: `IStateListener<PhaseType>` — `OnStateEnter(PhaseType)` 메서드 구현
- Enum 값: `PhaseType.Combat` — 전장 시작 시점
- GlobalEventBus 현황: `OnBattleWinEventDto`, `OnBattleLoseEventDto`, `OnRoundLoseEventDto` 존재. **전장 시작 전용 이벤트는 없음**

**새 컴포넌트에서 훅하는 방법**:
- `IStateListener<PhaseType>` 구현 후 RoundManager에 리스너 등록
- `OnStateEnter(PhaseType.Combat)` 안에서 분배 로직 실행

### 질문 2: 소환수 풀(32체) 접근 방법

**도메인 표현**: 소환수 풀은 SummonerManager → 각 SummonerLoadOutData → SummonerDefinitionData.SummonPool(8체) 경로로 접근한다. 소환술사 4명 × 8체를 순회하면 32체 전체를 얻는다.

**코드 구조**:

- **파일**: `Assets/Scenes/Battle/Feature/Unit/Summoner/SummonerManager.cs`
- **파일**: `Assets/Common/Data/Summoners/SummonerDefinitions/SummonerDefinitionData.cs`

```
32체 풀 접근 흐름

SummonerManager.Instance.Summoners
    : IReadOnlyList<SummonerLoadOutData>   // 편성된 소환술사 4명

foreach SummonerLoadOutData summoner
    └─ summoner.Summoner                   // SummonerDefinitionData
           └─ .SummonPool                  // UnitLoadOutData[] (8체)
                   └─ [각 UnitLoadOutData] // 소환수 1체의 정의 데이터
```

**핵심 식별자**:
- 속성: `SummonerManager.Instance.Summoners` → `IReadOnlyList<SummonerLoadOutData>`
- 속성: `SummonerLoadOutData.Summoner` → `SummonerDefinitionData`
- 필드: `SummonerDefinitionData.SummonPool` → `UnitLoadOutData[]` (8체, SerializeField)
- 런타임 소환술사 인스턴스: `SummonerManager._summoners` (`List<Summoner>`, private)

### 질문 3: 소환수 런타임 데이터 저장 구조

**도메인 표현**: 소환수 1체의 런타임 표현은 `Unit` 클래스(MonoBehaviour)이며 `UnitLoadOutData` 정의 참조, `UnitStatSheet`(능력치), `StatusEffectController`(상태 효과) 등을 필드로 보유한다. 소환수별 추가 데이터를 붙이는 기존 패턴은 Unit 필드 직접 추가 또는 `Dictionary<UnitLoadOutData, T>` 방식이 혼용된다.

**코드 구조**:

- **런타임 클래스**: `Assets/Scenes/Battle/Feature/Unit/Unit/Scripts/Unit.cs` (MonoBehaviour)

```csharp
public class Unit : MonoBehaviour
{
    public UnitLoadOutData UnitLoadOutData;                  // 정의 데이터 참조 (key 역할 가능)
    public readonly UnitStatSheet StatSheet = new();         // 능력치 시트
    public ActionStateController ActionStateController;      // 행동 상태 머신
    public StatusEffectController StatusEffectController;    // 상태 효과
}
```

- **UnitStatSheet** (`Assets/Scenes/Battle/Feature/Unit/Unit/UnitStats/UnitStatSheet.cs`):
  - `Star(int)`, `Reinforcement(int)`, 능력치 14종(각 `UnitStat` 객체), `Health(RxValue<float>)`

- **기존 딕셔너리 패턴 예시** (SynergyManager):
  - `_unitSynergyMap : Dictionary<UnitLoadOutData, SynergyDefinitionData>` — 소환수 정의 데이터를 키로 사용

**핵심 식별자**:
- 클래스: `Unit` — 소환수 런타임 1체 표현
- 클래스: `Summoner : Unit` — 소환술사 런타임 (추가 필드: `SummonerDefinitionData SummonerDefinition`)
- 패턴: `Dictionary<UnitLoadOutData, T>` — UnitLoadOutData를 키로 런타임 데이터 보관

### 질문 4: SummonTrait 자산 로드 접근점

**도메인 표현**: 현재 `SynergyType.SummonTrait` 값은 enum에 정의되어 있지만, 이 타입을 로드하거나 처리하는 코드는 없다. SynergyManager.Start()는 SummonerEffect 타입만 로드한다. 새 로드 진입점은 SynergyManager.Start() 또는 PhaseType.Combat Enter 시점에 두는 것이 기존 구조와 자연스럽게 맞는다.

**코드 구조**:

- **파일**: `Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyManager.cs`

```csharp
// 현재 Start() — SummonerEffect만 처리
void Start()
{
    IReadOnlyList<SummonerLoadOutData> summoners = SummonerManager.Instance.Summoners;
    var uniqueEffects = new HashSet<SynergyDefinitionData>();
    foreach (SummonerLoadOutData summoner in summoners)
    {
        uniqueEffects.Add(summoner.Summoner.SummonerEffect);  // SummonerEffect만 수집
    }
    InitializeSynergyActivations(uniqueEffects);
}
```

**SummonTrait 로드가 없는 이유**: TACD-250(소환수 특성 부여 기능)이 미구현 상태. `SummonPool`의 각 `UnitLoadOutData`에 SummonTrait를 연결하는 구조 자체가 아직 없음.

**자연스러운 진입점 옵션**:

| 위치 | 방식 | 적합성 |
|------|------|--------|
| SynergyManager.Start() | 기존 로드 루프 옆에 SummonTrait 루프 추가 | 기존 패턴 일관성 ↑ |
| PhaseType.Combat Enter | IStateListener 구현 신규 컴포넌트 | 분배 타이밍과 일치 ↑ |
| 별도 TraitDistributionService | 단일 책임 분리 | 클린 아키텍처 ↑ |

**SynergyControllerFactory 확장 필요**: `SynergyId` enum에 새 값 추가 시 `SynergyControllerFactory` switch 분기도 추가해야 함 (`Assets/Scenes/Battle/Feature/Synergy/Scripts/SynergyControllerFactory.cs`).

## 데이터 흐름 (전장 시작 ~ 특성 분배 구조)

```
PhaseType.Combat Enter
    │
    ├─ [기존] RoundAggressorManager.OnRoundEnter() — 침략자 스폰
    │
    └─ [신규 필요] 특성 분배 진입점
           │
           ├─ SummonerManager.Instance.Summoners (4명)
           │       └─ foreach summoner → summoner.Summoner.SummonPool (8체) → 32체 수집
           │
           ├─ SummonTrait 자산 목록 로드 (현재 진입점 없음 — 신규 구현 필요)
           │
           └─ 균등 랜덤 분배 → 소환수별 특성 저장
                   └─ 저장 구조 미결정 (Dictionary<UnitLoadOutData, SynergyDefinitionData> 패턴 참고)
```

---

## 발견한 특이사항

- **SummonTrait 로드 진입점 없음**: `SynergyType.SummonTrait`는 enum에만 존재. `SummonPool` 내 각 UnitLoadOutData와 SynergyDefinitionData(SummonTrait)를 연결하는 구조가 전혀 없음 — TACD-300 자산 생성 후에도 시스템이 자동 인식하지 못함.
- **SummonerManager.Summoners는 LoadOutData 반환**: 런타임 Unit 인스턴스가 아닌 정의 데이터(`SummonerLoadOutData`)를 반환. 런타임 인스턴스가 필요하면 `SummonerManager._summoners(List<Summoner>)`를 사용해야 하나 현재 private.
- **SynergyId enum 확장 필요**: 새 SummonTrait 자산에 새 Id를 부여하려면 `SynergyId` enum에 값 추가 + `SynergyControllerFactory` switch 분기 추가 필요.

## 탐색 한계

- **TACD-250 구현 코드 미확인**: 소환수 특성 부여 기능 자체가 미구현이므로 SummonTrait 런타임 처리 흐름은 현재 코드에서 탐색 불가.
- **SummonerManager._summoners 접근성**: private 필드이므로 외부에서 런타임 Summoner 인스턴스 목록 접근 방법이 제한적임 — 공개 API 추가 필요 여부는 설계 결정 사항.
- **특성 저장 위치 미결정**: 소환수별 특성 정보를 어디(Unit 필드 / 별도 딕셔너리 / 신규 컴포넌트)에 보관할지는 탐색 대상이 아닌 설계 결정 사항.
