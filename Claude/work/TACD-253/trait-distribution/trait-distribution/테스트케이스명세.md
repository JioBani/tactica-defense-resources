# TACD-253 — trait-distribution 테스트 케이스 명세

## 구현 단위 범위

- 구현 단위 식별자: `trait-distribution` (SR 그룹 A)
- 대상 클래스:
  - `Assets/Common/Data/Synergies/SummonTraitRegistry.cs`
  - `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitService.cs`
  - `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitDistributor.cs`
  - `Assets/Scenes/Battle/Feature/Events/RoundEvents/OnBattleStartEventDto.cs` (marker struct — 행위 없음, 테스트 미대상)
  - `Assets/Scenes/Battle/Feature/Round/RoundManager.cs` (1 줄 추가)
- 다루는 DoD 묶음:
  - **DoD-S1**: SummonTrait 타입 SynergyDefinitionData 자산 전체가 분배 대상 특성 풀로 시스템에 인식되어 접근 가능하다.
  - **DoD-S2**: 전장 시작 시 소환술사 4명의 소환수 풀(각 8체, 총 32체) 각각에 특성 풀에서 1개의 특성이 균등한 확률로 무작위 배정된다. 동일 특성이 복수의 소환수에 배정될 수 있다.
- 대응 UR DoD: DoD-U1 / DoD-U3.

## 방법론별 도출

### 1. 유스케이스 테스트

#### UC-1: 특성 풀 인식 (DoD-S1)

- **TC-001**: 인스펙터에 trait 자산 N 개를 연결한 Registry → `Traits` 가 동일 순서·동일 참조로 노출. 시스템이 풀을 인식해 사용할 수 있는 상태.
- **TC-002**: 인스펙터에서 비어있는 Registry → `Traits.Count == 0`. (DD 의 기본값 `traits = {}` 검증)

#### UC-2: 균등 랜덤 분배 (DoD-S2)

- 정상 흐름:
  - **TC-101**: 4 명 소환술사(각 8 체 풀) + 5 개 trait 풀 → `Distribute` 결과 딕셔너리에 32 개 entry 가 빠짐없이 존재. (32체 전원 배정)
  - **TC-102**: 같은 입력 → 결과 딕셔너리의 모든 값(trait)이 입력 traitPool 의 멤버임을 검증.
- 대안 흐름:
  - **TC-105**: summoners 에 null 항목 1 개 포함 (3 valid + 1 null) → null 은 스킵, 나머지 3 명의 24 entries 정상 배정.
- 예외 흐름:
  - **TC-103**: traitPool == null → `Debug.LogError` + 빈 딕셔너리 반환.
  - **TC-104**: 빈 traitPool (`Count == 0`) → `Debug.LogError` + 빈 딕셔너리 반환.
  - **TC-106**: summoner.Summoner.SummonPool null/빈 → `Debug.LogWarning` + 해당 소환술사 스킵.
  - **TC-107**: summonPool 안에 null UnitLoadOutData → `Debug.LogWarning` + 해당 unit 만 스킵, 나머지 정상.

#### UC-3: 분배 통합 — 전장 시작 훅 (DoD-S1·DoD-S2)

- 정상 흐름:
  - **TC-201**: Distributor 활성화(OnEnable) → `GlobalEventBus.Handlers` 에 `OnBattleStartEventDto` 구독 등록.
  - **TC-202**: Distributor 비활성화(OnDisable) → 구독 해제.
  - **TC-203**: registry · SummonerManager · Store 모두 셋업 + `Publish(OnBattleStartEventDto)` → Store.All 에 모든 unit ↔ trait 매핑 저장 (Service 의 결과가 Store 에 위임됨).
- 예외 흐름:
  - **TC-204**: registry == null + 이벤트 발행 → `Debug.LogError("[SummonTraitDistributor] registry 미연결")` + Store 무변경.
  - **TC-205**: SummonerManager.Instance == null → `Debug.LogError("[SummonTraitDistributor] SummonerManager 미존재")` + Store 무변경.
  - **TC-206**: SummonTraitStore.Instance == null → `Debug.LogError("[SummonTraitDistributor] SummonTraitStore 미존재")` + 분배 결과 미저장.

### 2. 동치 분할 (Equivalence Partitioning)

#### 입력: `Service.Distribute` 의 traitPool

- 유효 그룹:
  - 비어있지 않은 다수 (>= 2) — 대표값: 5 개 → **TC-101 / TC-102**
  - 단일 멤버 (Count == 1, 경계) → **TC-108**
- 무효 / 경계 그룹:
  - null → **TC-103**
  - 빈 컬렉션 (Count == 0) → **TC-104**

#### 입력: `Service.Distribute` 의 summoners (각 항목)

- 유효 그룹:
  - 정상 SummonerLoadOutData (8 체 풀) — **TC-101**
- 무효 그룹:
  - null SummonerLoadOutData → **TC-105**
  - SummonPool == null / 빈 배열 → **TC-106**
  - SummonPool 안에 null 항목 → **TC-107**

### 3. 결정 테이블 (Decision Table)

#### 결정 포인트: Distributor.OnBattleStart 진입 가드

- 관련 DoD: DoD-S2
- 조건:
  - C1: registry 가 null 인가
  - C2: SummonerManager.Instance 가 null 인가
  - C3: SummonTraitStore.Instance 가 null 인가
- 결정 테이블:

| 조합 | C1 (reg null) | C2 (SM null) | C3 (Store null) | 기대 결과                         | TC ID  |
|------|---------------|--------------|-----------------|-----------------------------------|--------|
| 1    | F             | F            | F               | 정상 분배 → Store.Initialize 호출 | TC-203 |
| 2    | T             | *            | *               | LogError("registry 미연결") + skip | TC-204 |
| 3    | F             | T            | *               | LogError("SummonerManager 미존재") + skip | TC-205 |
| 4    | F             | F            | T               | LogError("SummonTraitStore 미존재") + skip (분배 결과 미저장) | TC-206 |

- 가드 순서가 단락 평가이므로 C1 우선, C2 차순, C3 종순. 각 조합은 첫 번째 truthy 조건에서 종료.

## 미커버 / 보류 항목

- **RoundManager.Start() 의 `GlobalEventBus.Publish(new OnBattleStartEventDto())` 1 줄 추가**: Start() 메서드 자체가 BattlefieldData (SceneDataManager 의존)·StartRound() 호출 등 다수의 의존을 가지고 있어 단위 테스트로 분리하기 어렵다. DD 의 명시적 수정(1 줄 추가) 의 정확성은 코드 리뷰·통합 테스트(씬 진입 시 분배 확인) 영역. 본 구현 단위에서는 *이벤트 발행 시 분배가 이뤄지는 검증* 으로 우회 검증 (TC-203).
- **균등 분포 통계 검증**: 다수 반복 호출로 분포가 균등에 수렴함을 검증하는 통계 테스트는 본 단위 테스트 범위에서 다루기 부적합 (시드 제어 없는 `UnityEngine.Random` 사용). 균등성은 *Random.Range 의 균등 분포* 라는 Unity 제공 인프라의 보장으로 위임.
- **인스펙터 설정 (Distributor.registry 연결 / Store.\_dontDestroyOnLoad / Distributor 의 SceneSingleton 미상속 — Distributor 는 일반 MonoBehaviour 1 개 인스턴스를 씬에 두는 패턴)**: 코드 단위 테스트 범위 외.
