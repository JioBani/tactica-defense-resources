# 데이터 정의

## 4. 데이터 케이스 분류

### 표현 데이터 (UI 표시용)

- 소환술사 이름, 아이콘, 일러스트, 설명

### 로직 데이터 (런타임 계산/판정용)

- 소환술사 ID — 식별
- 소환술사 효과 — 이 소환술사가 소유하는 시너지 (SynergyDefinitionData)
- 소환수 풀 — 이 소환술사가 소환할 수 있는 8체 (UnitLoadOutData[])
- 전투 스탯 — 14개 스탯, 승급 기반 (UnitStatsByLevelData 재사용)
- 스킬 — 빈 스킬로 둠. 향후 스킬 사용 소환술사 특성 대비

### 추정 데이터 → 범위 결정

| 항목 | 결정 | 사유 |
|------|------|------|
| 소환술사 스킬 | **포함** (빈 스킬) | 향후 스킬 사용 소환술사 특성 대비 |
| 공격 타입 (물리/마법) | 제외 | 불필요 |
| 레어리티 | 제외 | 인게임 데이터 범위 밖 (육성/컬렉션 데이터) |

## 5. 필드 목록 확정

### SummonerDefinitionData (독립 ScriptableObject) — 소환술사 정의

| 필드명 | 타입 | 용도 |
|--------|------|------|
| id | int | 소환술사 식별 |
| displayName | string | UI 표시명 |
| description | string | 설명 |
| icon | Sprite | UI 아이콘 |
| illustration | Sprite | 일러스트 |
| summonerEffect | SynergyDefinitionData | 소유 시너지 |
| summonPool | UnitLoadOutData[] | 소환수 8체 |

### SummonerUnitDefinitionData : UnitDefinitionData (빈 상속) — 타입 판별용

| 필드명 | 타입 | 용도 |
|--------|------|------|
| *(추가 필드 없음)* | | 런타임에서 `is SummonerUnitDefinitionData`로 판별 |

### SummonerLoadOutData : UnitLoadOutData — 인게임 복합 에셋

| 필드명 | 타입 | 용도 |
|--------|------|------|
| *(상속)* | | id, unit(=SummonerUnitDefinitionData), skill(빈 스킬), stats(승급 기반) |
| summoner | SummonerDefinitionData | 소환술사 정의 참조 |

> costByStar/refundByStar는 상속되지만 소환술사에서는 미사용

### 데이터 관계도

```
SummonerDefinitionData ◀──참조── SummonerLoadOutData ──상속──▶ UnitLoadOutData
                                        │
                                        └── unit = SummonerUnitDefinitionData ──상속──▶ UnitDefinitionData
```

- 각 정의 레이어(SummonerDefinitionData, UnitDefinitionData)는 서로 독립
- 로드아웃 레이어(SummonerLoadOutData)가 양쪽을 연결

## 설계 논의 사항

### 시너지 소속의 재정의

소환수의 시너지(소환술사 효과 + 소환수 특성)는 ScriptableObject의 고정 값이 아니라 **런타임에 결정되는 값**이다:
- 소환술사 효과: 어떤 소환술사가 이 소환수를 소유하느냐에서 유도
- 소환수 특성: 전장마다 랜덤 부여

따라서 시너지 정보는 UnitDefinitionData가 아닌 런타임에서 관리해야 한다.
