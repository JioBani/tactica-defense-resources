# 작업지시서

> 이 문서는 Claude Code 세션이 개발 작업 시 따라야 할 루틴이다.
> 참고 문서: `git.md`, `develop.md`, `CLAUDE.md`

---

## 작업 파일 구조

브랜치(일감)별로 작업 문서를 분리하여 관리한다.

```
Claude/work/
└── TACD-{번호}/
    └── 작업정의.md       # 완료 정의, 작업 단위
```

## 작업 루틴

### 1. 작업 내용 확인

1. 사용자가 지정한 Jira 일감의 작업 내용과 완료 목표 확인
2. 해당 일감의 브랜치 생성 및 전환 (`TACD-{번호}`)
3. 하위 작업 확인:
   - **하위 작업이 있는 경우**:
     1. 하위 작업 목록을 조회한다
     2. 사용자에게 하위 작업 목록을 보여주고, **task 전체 설계가 필요한지 확인**한다
        - 하위 작업들이 하나의 시스템을 구성하는 경우(클래스/데이터를 공유하며 맞물림) → 전체 설계를 먼저 수행
        - 하위 작업들이 서로 독립적인 경우 → 전체 설계 없이 바로 진행
     3. 전체 설계가 필요한 경우: 핵심 클래스와 그 관계, 데이터 흐름을 잡는다. 각 클래스의 내부 구현은 하위 일감의 plan에서 결정한다.
     4. 각 하위 작업에 대해 2~4단계를 반복 수행한다
   - **하위 작업이 없는 경우**:
     1. 사용자에게 하위 작업을 만들 것인지 확인한다
     2. 사용자가 직접 만들겠다고 하면: 생성 완료 후 다시 진행
     3. 사용자가 만들지 않겠다고 하면: Task와 동일한 제목·설명의 하위 작업을 자동 생성할 것임을 안내하고, 사용자 확인 후 생성한다
4. 작업 디렉토리 생성 (`Claude/work/TACD-{번호}/`, 하위 작업 시 하위 번호 사용)

### 2. 작업정의 작성

작업정의는 **완료 정의**와 **작업 단위** 두 섹션으로 구성한다.

#### 완료 정의

이 일감이 "끝났다"고 판단할 수 있는 조건을 나열한다.

- 비즈니스 동작은 반드시 **기능/동작적으로 서술**한다. 완료 정의만 보고 QA와 기획이 이해할 수 있어야 한다.
- 예외: 데이터베이스, 프레임워크, 라이브러리 제작 등의 경우 코드적인 부분을 서술할 수 있다.
- **데이터의 내용(무엇을)**에 집중하고, **데이터의 형태(어떤 타입/구조로)**는 명시하지 않는다.

예시:
- 나쁨: "시너지별 카운트(Dictionary<SynergyDefinitionData, int>)를 반환한다"
- 좋음: "시너지별 카운트를 산출한다"

#### 작업 단위

완료 정의를 한번에 작업하기 좋은 단계로 묶는다.

- 각 단계에 번호와 이름을 부여하고, 하위에 해당 완료 정의를 배치한다.
- 구현 시 **한 단계 = 한 번의 plan 사이클**로 수행하며, 해당 단계의 완료 정의를 모두 만족시킨다.
- **절대 여러 단계에 대해 한번에 plan을 세우지 않는다.**

#### 절차

1. Jira 일감의 작업 내용과 완료 목표 확인
2. 기획서(`Document/기획서/`)에서 관련 요구사항 수집
3. 코드베이스 탐색은 **꼭 필요한 경우에만** 수행한다
   - 필요한 경우 예시: 기존 데이터 구조 확인, 이슈 자체가 데이터 설계에 관한 것, 기존 동작 방식 파악이 불가피한 경우
   - 탐색 범위는 이슈가 직접 다루는 데이터 구조까지로 한정한다. 소비자 코드(Manager, 이벤트 등)는 3단계에서 탐색한다.
   - 예시 — "시너지 카운트 집계" 이슈의 경우:
     - 좋음: SynergyDefinitionData, SynergyType 등 시너지 데이터 구조만 확인
     - 나쁨: DefenderManager, Defender, Unit, 이벤트 구조까지 탐색 → 이것은 3단계에서 수행
4. `Claude/work/TACD-{번호}/작업정의.md` 작성 (템플릿: `Claude/work/작업정의-템플릿.md`)
5. 미결사항 파악 및 작업정의에 기록
   - 구현을 진행하기 전에 사용자의 결정이 필요한 사항을 식별한다
   - 예: 데이터 형식, 네이밍 규칙, 작업 범위, 기획 미확정 사항 등
   - 작업정의.md에 미결사항 섹션으로 정리한다
6. 사용자에게 작업정의와 미결사항을 함께 보여주고 확인을 받는다
7. 모든 미결사항이 해결되면 사용자에게 다음 단계로 넘어갈 지 확인을 받는다

### 3. 구현

아래 사이클을 **작업 단위마다** 반복한다.

```
3.1 작업정의에서 다음 단계 선택 → [~] 표시
3.2 plan 모드로 해당 단계의 구현 계획 작성 (해당 단계의 완료 정의를 모두 만족시키는 계획)
3.3 plan 모드 승인
3.4 코드 구현
3.5 Unity 컴파일 확인 (refresh_unity + read_console)
3.6 유닛 테스트 작성
3.7 유닛 테스트 실행
3.8 실패 시 수정 → 3.7로 복귀
3.9 통과 시 작업정의에서 [x] 표시
3.10 사용자에게 완료 확인을 받는다
3.11 사용자 승인 후 git.md 형식에 따라 커밋
3.12 다음 단계로 이동
```

**주의**: plan은 반드시 **작업 단위 1개씩만** 세운다. 절대 여러 단계에 대해 한번에 plan하지 않는다.

## 작업 규칙

- **섹션 단위(예: 2, 3) 작업 완료 시마다 사용자에게 확인을 받는다.**
- 확인 없이 다음 섹션으로 넘어가지 않는다.
- **작업 단위 초기 상태**: 작업정의.md를 작성할 때 작업 단위는 반드시 `[ ]` (미완료) 상태로 초기화한다. 실제로 완료한 단계만 `[x]`로 변경한다.
- **plan 모드 범위**: plan 모드는 현재 진행 중인 **작업 단위 1개**만 대상으로 한다. 절대 여러 단계를 하나의 plan에 묶지 않는다.
- **코드베이스 탐색 방법**:
  1. 커밋 히스토리 우선: `git log --all --grep="TACD-{번호}"`로 해당 일감의 기존 커밋 확인. sub-task인 경우 부모/형제 번호로도 검색
  2. 커밋으로 부족하면 코드베이스 탐색 (`CLAUDE.md` 참고)
