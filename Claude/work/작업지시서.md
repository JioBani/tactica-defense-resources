# 작업지시서

> 이 문서는 Claude Code 세션이 개발 작업 시 따라야 할 루틴이다.
> 참고 문서: `git.md`, `develop.md`, `CLAUDE.md`

---

## 작업 파일 구조

브랜치(일감)별로 작업 문서를 분리하여 관리한다.

```
Claude/work/
└── TACD-{번호}/
    └── 작업정의.md       # 작업 내용, 완료 정의, 작업 단위 체크리스트
```

## 작업 루틴

### 1. 작업 내용 확인

1. 사용자가 지정한 Jira 일감의 작업 내용과 완료 목표 확인
2. 해당 일감의 브랜치 생성 및 전환 (`TACD-{번호}`)
3. 하위 작업 확인:
   - **하위 작업이 있는 경우**:
     1. 하위 작업 목록을 조회한다
     2. 사용자에게 하위 작업 목록을 보여주고, **task 전체 설계가 필요한지 확인**한다
        - 하위 작업들이 하나의 시스템을 구성하는 경우(클래스/데이터를 공유하며 맞물림) → 전체 설계를 먼저 수행
        - 하위 작업들이 서로 독립적인 경우 → 전체 설계 없이 바로 진행
     3. 전체 설계가 필요한 경우: 핵심 클래스와 그 관계, 데이터 흐름을 잡는다. 각 클래스의 내부 구현은 하위 일감의 plan에서 결정한다.
     4. 각 하위 작업에 대해 2~4단계를 반복 수행한다
   - **하위 작업이 없는 경우**: 사용자에게 해당 Task 단위로 바로 작업할지 확인 후 진행한다
4. 작업 디렉토리 생성 (`Claude/work/TACD-{번호}/`, 하위 작업 시 하위 번호 사용)

### 2. 작업정의 작성

작업정의는 **사용자/기획자 관점의 동작·기능 문서**로 작성한다. 코드 수준의 구현 상세가 아니라, "무엇이 어떻게 동작해야 하는가"를 기술한다.

#### 작업정의의 역할

Jira sub-task는 대부분 스토리 수준의 크기를 가진다 (예: "전장 배치 소환수 기반 시너지 카운트 집계"). 이대로는 하나의 plan 사이클로 처리하기에 너무 크다. 작업정의의 **작업 체크리스트**가 이 gap을 메운다: sub-task를 plan 사이클 단위의 더 작은 **동작 기반 작업**으로 분해한다.

#### 작업 체크리스트를 동작/기능 기반으로 작성해야 하는 이유

체크리스트는 코드 탐색 전(2단계)에 작성된다.

- **동작 기반이면** 구현 방식과 독립적이다 → 3단계 plan 모드에서 코드를 깊이 탐색한 뒤 최적의 구현 방식을 결정할 여지가 생긴다.
- **클래스/함수 단위가 되면** 구현 방식이 사전 결정된다 → plan 모드에서 코드를 탐색해도, 이미 정해진 방향에 끌려간다. 코드를 얕게 훑고 내린 결정은 깊이 탐색 후 나오는 방향보다 질이 낮다.

예시 — "전장 시작 시 소환수 특성 랜덤 부여" 이슈의 경우:
- 나쁨 (코드 수준): "UnitDefinitionData에 역할군 필드 추가", "순수 C# 서비스 클래스 구현", "ScriptableObject로 외부화"
- 좋음 (동작 기반): "역할군 데이터 준비", "특성 분배 규칙 구현 및 테스트", "전장 시작 시 분배 트리거 연결"

#### 완료 정의 작성 원칙

완료 정의는 **데이터의 내용(무엇을)**에 집중하고, **데이터의 형태(어떤 타입/구조로)**는 명시하지 않는다.
구체적인 자료구조(Dictionary, List 등)는 plan 모드에서 코드를 탐색한 뒤 결정한다.

- 나쁨: "시너지별 카운트(Dictionary<SynergyDefinitionData, int>)를 반환한다"
- 좋음: "시너지별 카운트를 산출한다"

#### 절차

1. Jira 일감의 작업 내용과 완료 목표 확인
2. 기획서(`Document/기획서/`)에서 관련 요구사항 수집
3. 코드베이스 탐색은 **꼭 필요한 경우에만** 수행한다
   - 필요한 경우 예시: 기존 데이터 구조 확인, 이슈 자체가 데이터 설계에 관한 것, 기존 동작 방식 파악이 불가피한 경우
   - 탐색 범위는 이슈가 직접 다루는 데이터 구조까지로 한정한다. 소비자 코드(Manager, 이벤트 등)는 3단계에서 탐색한다.
   - 예시 — "시너지 카운트 집계" 이슈의 경우:
     - 좋음: SynergyDefinitionData, SynergyType 등 시너지 데이터 구조만 확인
     - 나쁨: DefenderManager, Defender, Unit, 이벤트 구조까지 탐색 → 이것은 3단계에서 수행
4. `Claude/work/TACD-{번호}/작업정의.md` 작성 (템플릿: `Claude/work/작업정의-템플릿.md`)
5. 사용자 확인 후 다음 단계로 진행

### 3. 구현

아래 사이클을 작업 단위마다 반복한다.

```
3.1 작업정의에서 다음 항목 선택 → [~] 표시
3.2 코드베이스 탐색 (관련 코드 구조 파악, 수정 대상 파일 확인)
3.3 plan 모드로 해당 항목의 구현 계획 작성
3.4 plan 모드 승인
3.5 코드 구현
3.6 Unity 컴파일 확인 (refresh_unity + read_console)
3.7 유닛 테스트 작성
3.8 유닛 테스트 실행
3.9 실패 시 수정 → 3.8로 복귀
3.10 통과 시 작업정의에서 [x] 표시 → 다음 항목으로 이동
```

### 4. 완료

1. 변경사항을 사용자에게 보여주고 확인을 받는다 (git diff 등)
2. 사용자 승인 후 `git.md` 형식에 따라 커밋
3. Jira 일감 상태 변경

## 작업 규칙

- **섹션 단위(예: 2, 3) 작업 완료 시마다 사용자에게 확인을 받는다.**
- 확인 없이 다음 섹션으로 넘어가지 않는다.
- **체크리스트 초기 상태**: 작업정의.md를 작성할 때 체크리스트는 반드시 `[ ]` (미완료) 상태로 초기화한다. 실제로 완료한 항목만 `[x]`로 변경한다.
- **plan 모드 범위**: plan 모드는 현재 진행 중인 **작업 단위 1개**만 대상으로 한다. 여러 작업 단위를 하나의 plan에 묶지 않는다.
- **코드베이스 탐색 방법**:
  1. 커밋 히스토리 우선: `git log --all --grep="TACD-{번호}"`로 해당 일감의 기존 커밋 확인. sub-task인 경우 부모/형제 번호로도 검색
  2. 커밋으로 부족하면 코드베이스 탐색 (`CLAUDE.md` 참고)
