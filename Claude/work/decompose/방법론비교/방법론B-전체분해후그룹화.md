### 상호작용 분석

#### 대상 목록

| 분류 | 대상 |
|------|------|
| **인접 기능** | TACD-80 수정자 시스템, TACD-81 소환수 배치/제거, TACD-237 소환술사 육성 |
| **상위 흐름** | RoundManager (Maintenance → Ready → Combat → End 라이프사이클) |

### 보조 관점 체크

| 관점 | 항목 |
|------|------|
| **데이터/속성** | SynergyDefinitionData에 SynergyType(소환술사 효과/소환수 특성) 이미 존재. 소환수 특성 풀 정의 데이터(8개 특성 목록) 필요. 역할군별 확률 보정 테이블 데이터 필요. UnitDefinitionData에 소환수 특성 필드 추가 필요 (현재 summonerEffect만 있음). |

## 상호작용 축이 추가로 도출한 항목

상호작용 분석이 없었다면 동작 트리만으로는 놓쳤을 항목들:

| # | 항목 | 이유 |
|---|------|------|
| #19, #20 | **시너지 상태 변경 이벤트 DTO 및 GlobalEventBus 발행** | 동작 트리는 시너지 내부 로직에 집중하므로 "외부에 알리는 이벤트 인프라"는 도출하기 어렵다. 상호작용 분석에서 GlobalEventBus와의 양방향 관계를 분석하면서 "시너지 → UI/외부 시스템으로 상태 전파"가 필요하다는 점이 드러났다. |
| #10 | **RoundManager 페이즈 전환과 특성 부여 타이밍 연동** | 동작 트리에서 "전장 시작 시 부여"는 나오지만, 구체적으로 어떤 페이즈에서 트리거해야 하는지는 RoundManager와의 상호작용 분석을 통해 명확해진다. Maintenance 진입? Ready 진입? Combat 시작? — 이 결정은 상위 흐름과의 트리거 관계를 분석해야 도출된다. |
| #14 | **SynergyManager에 소환수 특성 동적 등록** | 기존 SynergyManager는 allSynergies를 Inspector에서 고정 주입한다. 소환수 특성은 전장마다 달라지므로 동적 등록/해제가 필요한데, 이는 데이터 흐름(전장별 변경 데이터가 SynergyManager로 유입)을 분석해야 도출된다. |
| #6, #7 | **소환수 특성 런타임 바인딩 구조** | UnitDefinitionData가 SO(불변)라는 사실과 "전장마다 다른 특성"이라는 요구사항의 충돌은, 데이터 시스템과의 상호작용을 분석하면서 비로소 "런타임 매핑이 필요하다"는 기반 작업이 도출된다. |
