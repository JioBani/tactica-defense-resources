# TACD-296 — member-list 테스트 케이스 명세

## 구현 단위 범위

- 구현 단위 식별자: `member-list`
- 다루는 DoD 묶음 (SR 그룹 C — 시너지 보유 소환수 목록 표시):
  - DoD-S11: 표시 대상 시너지의 보유 소환수 목록을 시너지 매니저의 *시너지→보유 소환수 역참조 통로* 에서 조회. 종류 무관 일반 책임. 키 부재 시 빈 목록.
  - DoD-S12: 각 항목은 소환수의 아이콘과 이름을 표시.
  - DoD-S13: 각 소환수의 전장 배치 여부를 소환수 배치 상태 제공자에서 조회 → 배치되어 있으면 활성, 아니면 비활성. 다운 여부는 판정에 무관.
  - DoD-S14: 표시 상태인 동안 배치 / 제거 변경 알림(GlobalEventBus 두 종)을 구독하여 활성 / 비활성 표시를 갱신.

## 테스트 모드 결정

- **EditMode** — 검증 대상은 일반 클래스 `SynergyDetailMemberList` 의 외부 표면 + VisualElement 표시 상태 + GlobalEventBus 알림 응답. 시간 진행 / 물리 / 렌더 의존 없음.
- SynergyManager / DefenderManager 는 MonoBehaviour 이나 본 테스트에서는 비활성 GameObject 에 AddComponent 후 리플렉션으로 정적·인스턴스 필드를 직접 주입하여 라이프사이클(Awake/Start)을 우회.

## 방법론별 도출

### 1. 유스케이스 테스트

#### UC-1: 보유 소환수 목록 조회 + 항목 생성 (DoD-S11, DoD-S12)

- 정상 흐름:
  - **TC-1201**: SynergyMembers[def] = [u1, u2, u3] → Bind 후 항목 3 개.
  - **TC-1301**: 각 항목의 Label.text 가 unit.Unit.DisplayName 과 일치.
  - **TC-1302**: 각 항목의 icon 자식의 backgroundImage 가 unit.Unit.Icon Sprite 와 일치.
- 자연 빈 상태:
  - **TC-1202**: SynergyMembers 에 def 키 부재 → 항목 0 (시너지 종류 분기 코드 없음 — 빈 목록은 자연 흡수).
  - **TC-1203**: SynergyMembers[def] = [] → 항목 0, 예외 없음.

#### UC-2: 배치 상태 시각 (DoD-S13)

- 결정 테이블 형태 — 아래 §결정 테이블 참조.

#### UC-3: 알림 구독·해제 (DoD-S14)

- 정상 흐름:
  - **TC-1501**: Bind 후 defender 의 Placement 변경 + OnDefenderPlacementChangedEventDto publish → 해당 unit 항목의 활성 클래스 토글.
  - **TC-1502**: Bind 후 OnDefenderChangedEventDto publish (Despawn) → 해당 unit 항목 재판정 (디스폰된 디펜더 제외 후 활성 여부 재계산).
- 정리 책임:
  - **TC-1503**: Unbind 후 두 종 알림 publish → 시각 변화 없음 (구독 해제).

#### UC-4: 라이프사이클 안전망

- 예외 흐름:
  - **TC-1601**: Bind(null) → 무시.
- 정상 흐름:
  - **TC-1602**: Bind(actA) → Rebind(actB) → tier-list 가 actB 의 항목으로 교체됨.
  - **TC-1603**: Bind → Unbind → 항목 컨테이너 자식 0.

### 2. 동치 분할

#### 입력: SynergyMembers 의 해당 시너지 키 상태

- 관련 DoD: DoD-S11
- 유효 그룹:
  - 키 존재 + N (>0) 개 unit — TC-1201
- 무효 / 경계 그룹:
  - 키 부재 — TC-1202
  - 키 존재 + 빈 목록 — TC-1203

#### 입력: 동일 unit 의 디펜더 다중성 (DoD-S13 의 다중 인스턴스 케이스)

- 유효 그룹:
  - BattleArea 1 명 — TC-1401
  - WaitingArea 1 명 — TC-1402
  - 0 명 — TC-1403
  - BattleArea 1 명 + WaitingArea 1 명 — TC-1405 (활성으로 판정)

### 3. 결정 테이블

#### 결정 포인트: `IsUnitPlacedInBattleArea` 의 시각 판정 (DoD-S13)

- 조건:
  - C1: DefenderManager 의 Defenders 중 동일 UnitLoadOutData 가진 defender 가 존재
  - C2: 그 defender 의 Placement == BattleArea
  - C3: 그 defender 의 다운 상태 (참고용 — 판정 무관)
- 결정 테이블:

  | 조합 | C1 | C2 | C3 | 기대 결과 | TC ID |
  |---|---|---|---|---|---|
  | 1 | F | * | * | 비활성 (해당 unit 의 디펜더 없음) | TC-1403 |
  | 2 | T | F | * | 비활성 (WaitingArea) | TC-1402 |
  | 3 | T | T | F | 활성 | TC-1401 |
  | 4 | T | T | T | 활성 (다운 무관 — DoD-S13 / OQ-6 결정) | TC-1404 |
  | 5 | C1 다중 — 하나라도 BattleArea 면 활성 | * | * | 활성 | TC-1405 |

## 미커버 / 보류 항목

- **SynergyDetailPanel.cs 의 DefenderManager 직렬화 wiring (prefab 차원)** — impl 메모 §prefab wiring 안내 에 따라 사용자 후속 작업으로 분리됨. 코드 단위 테스트는 *DefenderManager 가 주입된 상태* 를 가정하고 검증하므로 본 단위 책임 범위 자체는 충족 검증 가능.
- **SynergyManager.BuildSynergyMembersMap 의 실제 구축 경로 (SummonerManager 의존)** — Start 의 구축 로직 자체는 `_synergyMembers` 인덱스가 올바르게 채워지는 결과가 SynergyDetailMemberList 의 단위 책임이 아닌 SynergyManager 의 단위 책임이므로 *외부 입력* 으로 취급. 본 단위 테스트는 SynergyMembers 가 *어떤 결과* 를 반환하든 그 결과대로 항목을 표시하는 책임만 검증. 구축 경로 자체의 검증은 별도 SynergyManager 단위 테스트의 영역.
