## 외부 표면 (요구사항 테스트 단계의 입력)

### SynergyManager (신규 노출)
```csharp
public IReadOnlyDictionary<SynergyDefinitionData, IReadOnlyList<UnitLoadOutData>> SynergyMembers { get; }
```
- `Start()` 시점에 한 번 구축되며 이후 변경 없음. 키 부재 시 사용자 측이 빈 목록으로 해석하는 비즈니스 약속.
- 시너지 종류 분기 코드 없음 — `SummonerLoadOutData.Summoner.SummonerEffect` 키 아래에 `SummonPool` 의 소환수를 누적. 본 이슈 시점에 누적되는 시너지는 자연스럽게 소환술사 효과 시너지뿐이며, 향후 소환수 특성 분배 도메인 로직이 동일 인덱스에 누적하면 자동 합산.

### SynergyDetailMemberList (stub → 본체)
```csharp
public SynergyDetailMemberList(VisualElement sectionRoot, DefenderManager defenderManager = null);
public void Bind(SynergyActivation activation);
public void Rebind(SynergyActivation activation);
public void Unbind();
```
- 생성자: `sectionRoot.Q<VisualElement>("member-list")` 캐시 + DefenderManager 의존 저장. `defenderManager` 는 *optional* — 기존 lifecycle 사이클 테스트(`SynergyDetailPanelLifecycleTests`)의 `new SynergyDetailMemberList(memberRoot)` 호출과의 호환을 위함.
- `Bind`: SynergyManager.Instance.SynergyMembers.TryGetValue 로 보유 소환수 목록 조회 → 키 부재 시 빈 목록 (소환수 특성 시너지의 자연 상태). 각 unit 마다 `member-item` 생성 (아이콘 + 이름) → 전체에 대해 `RefreshItemPlacement` (BattleArea 배치 여부로 `--active` 클래스 토글) → 두 종 알림 구독 시작.
- `Rebind`: Unbind 후 Bind 단순 위임.
- `Unbind`: 두 종 알림 구독 해제 + 항목 컨테이너 비움 + `_itemsByUnit` 인덱스 비움.

### SynergyDetailPanel (DefenderManager 추가)
- `[SerializeField] DefenderManager defenderManager` — prefab 의 Detail Panel GameObject 에서 wiring 필요.
- `EnsureInitialized` 의 `new SynergyDetailMemberList(memberRoot, defenderManager)` 한 라인 변경.

## 자율 판단 (계획 외 결정)

1. **DefenderManager 접근 방식 — 프로젝트 컨벤션 정렬** — DD §부속 외부 표면 표에 *"DefenderManager.Instance.Defenders"* 라고 적었지만 실제 코드는 `DefenderManager : MonoBehaviour` (SceneSingleton 미상속). 프로젝트의 14개 호출자가 모두 `[SerializeField] DefenderManager` 직렬화 주입 패턴. CLAUDE.md 의 "Important Singletons" 목록이 outdated. 다음 검토:
   - (A) DefenderManager 에 SceneSingleton 상속 추가 — 프로젝트 전체 컨벤션을 *역방향* 으로 바꿈. 14곳 호출자에 회귀 위험. **미채택**.
   - (B) SynergyDetailMemberList 생성자에 DefenderManager 매개변수 추가 (의존성 주입) — 프로젝트 표준 패턴과 정합. **채택**.
   - 생성자 시그니처 변경(`VisualElement sectionRoot` → `VisualElement sectionRoot, DefenderManager defenderManager = null`) 은 DD §그룹 C 의 4축 *클래스 분리·책임·비즈니스 로직* 결정은 그대로 두고 *기술적 의존 주입 방식만 보강* 한 것이므로 큰 설계 결정 변경 아님 — 자율 판단 영역.
   - DefenderManager 가 null 인 경우 (테스트 픽스처 등) `IsUnitPlacedInBattleArea` 가 false 를 반환하여 모든 항목이 비활성 상태로 표시 — 안전망.
2. **lifecycle 사이클 영역(SynergyDetailPanel.cs) 수정** — member-list 의 의존성 주입을 위해 컨트롤러에 `[SerializeField] DefenderManager` 필드와 자식 생성 시 한 라인 주입을 추가. 본 사이클 범위를 약간 벗어나지만 *member-list 사이클의 의존성을 생성하는 단일 라인의 보강* 이라 자연. lifecycle 사이클 *컨벤션 차원의 변경* 은 없음 (라이프사이클 / 진입점 통합 책임 그대로).
3. **시너지 종류 분기 회피** — `SynergyManager.BuildSynergyMembersMap` 가 `SummonerLoadOutData.Summoner.SummonerEffect` 한 키만 인덱스. `SynergyType` 의 enum 분기 / 배제 로직 없음. 본 이슈 시점에는 결과적으로 *소환술사 효과 시너지만 누적* 되지만 코드 상으로는 *일반 책임* (DD 결정 OQ-S1 그대로). 향후 소환수 특성 분배 도메인 로직이 동일 인덱스에 누적하면 자동 합산.
4. **알림 구독 두 종 모두 채택** — `OnDefenderChangedEventDto` (Spawn/Despawn) + `OnDefenderPlacementChangedEventDto` (배치 변경). Spawn 자체는 WaitingArea 라 시각 변화 없지만 Despawn 처리 일관성 + Refactoring 안전성을 위해 두 종 모두. 핸들러는 영향받는 unit 키가 `_itemsByUnit` 에 있을 때만 *해당 항목 1개* 의 시각 재판정 (전체 재구축 X).
5. **빈 목록 시 안내 메시지 없음** — UXML 의 member-list 컨테이너에 항목이 없으면 빈 영역 그대로. "보유 소환수 없음" 같은 라벨은 미추가 — TACD-298 결정 §4 (프로토타입 수준).

## 테스트 멤버에 전달하는 주의 사항

- 본 사이클의 요구사항 테스트 대상 DoD: **S11 / S12 / S13 / S14**.
  - **DoD-S11**: SynergyDetailMemberList.Bind 후 항목 개수가 SynergyManager.SynergyMembers[activation.Definition].Count 와 일치. 키 부재 시 0 (빈 목록). 시너지 종류 분기 코드 없음 — Definition 의 SynergyType 검사 로직 부재 확인.
  - **DoD-S12**: 각 항목이 unit.Unit.Icon (배경 이미지) + unit.Unit.DisplayName (Label.text) 를 보유.
  - **DoD-S13**: DefenderManager 의 Defenders 중 동일 UnitLoadOutData + Placement==BattleArea 인 디펜더가 있으면 `--active` 클래스, 없으면 제거. **다운된 디펜더가 BattleArea 에 있으면 활성** (Placement 만 기준, ActionStateController.CurrentState 무관).
  - **DoD-S14**: Bind 후 GlobalEventBus.Publish<OnDefenderChangedEventDto> / <OnDefenderPlacementChangedEventDto> 시 영향받는 unit 의 시각 클래스가 갱신됨.
- **새 외부 표면 SynergyManager.SynergyMembers** — 픽스처 권장:
  - SynergyManager 의 `_synergyMembers` / `_synergyMembersPublic` private 필드를 리플렉션으로 직접 주입하여 SynergyMembers 프로퍼티 응답을 제어 (Start 미경유 검증).
  - 또는 SummonerManager.Instance.Summoners 를 stub 으로 두고 Start 를 호출하여 정상 구축 경로 검증.
- **생성자 시그니처 변경**: `SynergyDetailMemberList(VisualElement sectionRoot, DefenderManager defenderManager = null)`. 기존 lifecycle 테스트(`SynergyDetailPanelLifecycleTests`) 의 `new SynergyDetailMemberList(memberRoot)` 호출은 optional 매개변수 덕에 호환 — *변경 없이 통과*.
- **DefenderManager 주입 픽스처**: 본 사이클 테스트는 *MonoBehaviour DefenderManager 의 인스턴스* 를 직접 생성하여 생성자에 전달해야 함. `new GameObject().AddComponent<DefenderManager>()` 패턴 + `units` private 필드(`List<Defender>`)에 리플렉션 주입.

## DD 결정 변경 사항

- **시그니처 자율 확장**: SynergyDetailMemberList 생성자 `(VisualElement sectionRoot)` → `(VisualElement sectionRoot, DefenderManager defenderManager = null)`. DD §그룹 C 의 4축 *클래스 분리·책임·비즈니스 로직·예외 처리* 결정은 그대로 — 외부 의존(DefenderManager)을 라이프사이클 컨트롤러를 통해 주입하기 위한 *기술적 의존 주입 방식 보강*. DD §부속 외부 표면 표의 "DefenderManager.Instance.Defenders" 표기는 *코드 사실과의 오해석* 으로 본 메모에서 정정 (실제 컨벤션은 [SerializeField] 주입).
