# TACD-300 — 테스트용 소환수 특성 제작

> 이 문서는 시스템 요구사항을 LLM Plan 진입 직전까지 구체화한 설계 문서입니다.

## 개요

- Jira: TACD-300 (상위: TACD-250)
- `SynergyId` enum에 값 8개 추가(코드 수정 1건) + `SynergyDefinitionData` 자산 8개 신규 생성 + `StatusEffectDefinitionData` 자산 8개 신규 생성(빈 껍데기). ScriptableObject 에셋 파일 위주 작업이며 신규 클래스 없음.

| ID | 컴포넌트 | 산출물 | 대응 SR DoD |
|----|----------|--------|-------------|
| DD-A1 | SynergyId enum | 테스트 특성 식별자 8개 추가 | DoD-S3, DoD-S4, DoD-S9 |
| DD-A2 | SynergyDefinitionData 자산 8개 | SummonTrait 타입 특성 자산 신규 생성 | DoD-S1, DoD-S2, DoD-S4, DoD-S5, DoD-S9 |
| DD-B1 | StatusEffectDefinitionData 자산 8개 | 빈 SE 정의 자산 신규 생성 | DoD-S8 |
| DD-B2 | SynergyDefinitionData 자산 갱신 | 티어 데이터 및 SE 참조 설정 | DoD-S6, DoD-S7, DoD-S8 |

## 설계

### 그룹 A (`summon-trait-assets`). 소환수 특성 자산 생성 및 식별

**대응 SR DoD**: DoD-S1, DoD-S2, DoD-S3, DoD-S4, DoD-S5, DoD-S9

**설계 접근**: SR 그룹 A가 부여한 책임 — SummonTrait 타입 SynergyDefinitionData 자산 8개가 확정된 SynergyId·displayName과 함께 프로젝트에 존재 — 을 (1) `SynergyId.cs` enum 확장(코드 수정 1건) + (2) ScriptableObject 자산 파일 8개 생성(에디터 작업)으로 풀어낸다. enum 확장이 자산 생성보다 선행되어야 에디터에서 id 필드를 올바르게 설정할 수 있다.

#### 클래스 분리

- **SynergyId** (기존 수정, `Assets/Common/Data/Synergies/SynergyId.cs`)
  - 책임: SummonTrait 전용 테스트 특성 식별자 8개 추가
  - 사유: `SynergyDefinitionData.id` 필드가 이 enum을 사용하므로 자산 생성 전 enum 확장이 선행 필요. enum 값 번호는 기존 최댓값(Gunslinger=5) 이후인 6~13 고정.

- **SynergyDefinitionData 자산 8개** (신규, `Assets/Common/Data/Synergies/`)
  - 책임: 각 테스트 특성의 표시 정보(id, displayName, synergyType) 보유
  - 사유: ScriptableObject 자산 파일이 도메인 데이터의 직접 표현. 파일 단위로 독립 관리. tiers·statusEffectDefinition 설정은 그룹 B에서 수행.

#### 핵심 외부 노출 시그니처

#### 핵심 비즈니스 로직

1. `SynergyId.cs`에 8개 값을 위 명세 번호(6~13)로 추가한다.
2. `Assets/Common/Data/Synergies/` 경로에 자산 파일 8개를 위 명세 파일명으로 생성한다.
3. 각 자산의 `id`, `displayName`, `synergyType` 필드를 명세 값으로 설정한다.

#### 예외 처리

| 상황 | 처리 |
|------|------|
| SynergyId 값이 기존 값과 충돌 | 6~13 고정 할당으로 기존 0~5와 겹치지 않음. 향후 신규 값 추가 시 14 이후에 이어서 할당 |
| displayName이 기존 자산과 중복 | 확정 8개 이름(강인함·맹습·주술·신속·불굴·집중·파쇄·맹독)은 기존 자산(난동꾼·녹서스·데마시아·빌지워터·슈리마)과 겹치지 않음 — 명세로 보장 |
| synergyType 기본값 오적용 | Unity 에디터 기본값이 SummonerEffect(0)이므로 반드시 SummonTrait(1)로 명시 설정 필요 |

#### 테스트 책임 분리

- 단위 테스트: 해당 없음 (자산 파일 존재 여부·필드 값 설정은 에디터 로드로 확인).
- 통합 확인: 그룹 B 완료 후 8개 자산 로드 및 필드 전체 육안 검증으로 대신.

### 그룹 B (`summon-trait-tier-data`). 임계치별 효과 데이터 설정

**대응 SR DoD**: DoD-S6, DoD-S7, DoD-S8

**설계 접근**: SR 그룹 B가 부여한 책임 — 각 자산이 requiredCount 2·3·4인 SynergyTier 3개 및 constants, 그리고 StatusEffectDefinitionData 참조를 보유 — 을 (1) StatusEffectDefinitionData 자산 8개 신규 생성(id·displayName만 설정, 효과 필드 비움) + (2) SynergyDefinitionData 자산에 tiers 목록과 statusEffectDefinition 참조 설정으로 풀어낸다. SE 자산 생성이 SynergyDefinitionData 참조 설정보다 선행되어야 한다.

#### 클래스 분리

- **StatusEffectDefinitionData 자산 8개** (신규, `Assets/Common/Data/StatusEffects/`)
  - 책임: 각 테스트 특성에 연결되는 SE 정의 자산. id·displayName만 설정하고 효과 필드는 비워 둠.
  - 사유: `SynergyDefinitionData.statusEffectDefinition`이 이 자산을 참조하므로 선행 생성 필요. 효과 수치·로직은 TACD-250에서 ISynergyEffect 구현 시 채울 예정이며 TACD-300 범위 밖.

- **SynergyDefinitionData 자산 8개 갱신** (기존 수정, 그룹 A에서 생성한 자산)
  - 책임: `tiers` 목록(tier 1·2·3, requiredCount 2·3·4, constants) 및 `statusEffectDefinition` 참조 설정
  - 사유: 그룹 A에서 신원 정보만 설정된 자산에 임계치·효과 연결 데이터를 추가하여 DoD-S6·S7·S8 충족.

| 맹습 SE.asset | 7 | 맹습 | 비움 | 비움 (설계 제약) |
| 주술 SE.asset | 8 | 주술 | 비움 | 비움 (설계 제약) |
| 신속 SE.asset | 9 | 신속 | 비움 | 비움 (설계 제약) |
| 불굴 SE.asset | 10 | 불굴 | 비움 | 비움 (설계 제약) |
| 집중 SE.asset | 11 | 집중 | 비움 | 비움 (설계 제약) |
| 파쇄 SE.asset | 12 | 파쇄 | 비움 | 비움 (설계 제약) |

- SE id는 대응 SynergyId 정수값과 동일하게 할당(6~13).
- 현재 `StatusEffectDefinitionData`에 존재하는 필드: `id`·`displayName`·`icon`. 효과 필드 자체가 없으므로 별도 조치 없이 설계 제약 충족.

SynergyTier 구조 — 8개 자산 공통:

| tier | requiredCount |
|------|--------------|
| 1 | 2 |
| 2 | 3 |
| 3 | 4 |

특성별 constants 키·값 명세:

| displayName | tier 1 (req=2) | tier 2 (req=3) | tier 3 (req=4) |
|------------|----------------|----------------|----------------|
| 강인함 | healthBonus=0.10, armorBonus=0.05 | healthBonus=0.20, armorBonus=0.10 | healthBonus=0.35, armorBonus=0.20 |
| 맹습 | attackDamageBonus=0.10 | attackDamageBonus=0.20 | attackDamageBonus=0.35 |
| 주술 | magicDamageBonus=0.10 | magicDamageBonus=0.20 | magicDamageBonus=0.35 |
| 신속 | attackSpeedBonus=0.10, moveSpeedBonus=0.05 | attackSpeedBonus=0.20, moveSpeedBonus=0.10 | attackSpeedBonus=0.35, moveSpeedBonus=0.20 |
| 불굴 | damageReduction=0.05, statusResistance=0.10 | damageReduction=0.10, statusResistance=0.20 | damageReduction=0.20, statusResistance=0.35 |
| 집중 | cooldownReduction=0.10 | cooldownReduction=0.20 | cooldownReduction=0.30 |
| 파쇄 | damageBonus=0.10 | damageBonus=0.20 | damageBonus=0.35 |

> **constants 키 명칭 안내**: 위 키 이름은 main에서 확정된 스탯 계열 표를 기반으로 도출한 테스트용 임시 명칭이다. TACD-250에서 ISynergyEffect 구현 시 실제 사용 키와 일치시켜야 하며, 필요 시 자산 갱신으로 정렬한다.

#### 핵심 비즈니스 로직

1. `Assets/Common/Data/StatusEffects/` 경로에 SE 자산 8개를 위 명세 파일명으로 생성한다. `id`·`displayName`만 명세 값으로 설정하고 `icon`은 비워 둔다.
2. 그룹 A에서 생성한 SynergyDefinitionData 자산 8개에 `tiers` 목록을 설정한다. 각 자산에 tier 1·2·3을 추가하며, `requiredCount`는 순서대로 2·3·4, `constants`는 위 특성별 명세 값으로 설정한다. tiers 목록은 requiredCount 오름차순으로 유지한다.
3. 각 SynergyDefinitionData 자산의 `statusEffectDefinition` 필드에 대응 SE 자산을 연결한다 (강인함 → 강인함 SE.asset, 맹습 → 맹습 SE.asset, ... 8개 일대일 연결).

#### 예외 처리

| 상황 | 처리 |
|------|------|
| SE 자산 id 중복 | SE id를 SynergyId 정수값(6~13)과 일치시켜 기존 SE 자산(id 1~5)과 충돌 없음 |
| SynergyTier requiredCount 오름차순 위반 | `SynergyTier` 코드에 자동 정렬 없음 — 자산 설정 시 tier 1→2→3 순서(req 2→3→4)를 명시적으로 유지 |
| statusEffectDefinition 참조 누락 | 8개 자산 각각에 대응 SE 자산 연결 필수. 미연결 시 DoD-S8 위반 |
| 효과 필드 불필요한 추가 | `StatusEffectDefinitionData`에 현재 효과 관련 필드 없음(id·displayName·icon만 존재). 신규 필드 추가 없이 기존 구조 그대로 사용 — 설계 제약 충족 |

#### 테스트 책임 분리

- 단위 테스트: 해당 없음.
- 통합 확인: 에디터에서 8개 자산의 tiers 수·requiredCount 값·constants 존재 여부·statusEffectDefinition 참조 연결을 육안으로 확인.
