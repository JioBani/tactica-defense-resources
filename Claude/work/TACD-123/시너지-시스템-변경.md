# 기존 시너지 시스템 변경

> SE 프레임워크 도입에 따른 시너지 시스템 변경 사항을 정리한다.

---

## SynergyController 제거

- 기존 역할(이벤트 수신, 시너지 보유 판단, 효과 생성/제거)이 모두 다른 곳으로 이동:
  - 이벤트 수신 + 대상 판단 → SynergyManager가 흡수
  - 효과 생성/제거 → SEController가 담당
- SynergyController는 더 이상 필요 없다.

## SynergyManager 역할 확장

- 티어 변경 감지 (기존 유지)
- **SE 부여/제거 대상 결정** (신규): 시너지 종류에 따라 자기만/모든 아군 등 판단
- SED에서 SE 정의를 꺼내서 대상의 SEController에 Apply/Remove 요청

## SED와 SynergyDefinitionData 연결

```
SynergyDefinitionData
├─ Tiers (티어별 Constants)
└─ StatusEffect Definition (이 시너지가 부여하는 SED)
```

## 시나리오 검증

### 난동꾼: 난동꾼이 체력을 얻습니다 (2) 25%

```
시너지 활성화
→ SynergyManager: 난동꾼 유닛 대상 결정
→ 각 대상의 SEController.Apply(난동꾼 SED, {체력비율: 25%})
→ SE OnApply: 체력 증가 적용
→ 시너지 비활성화 시: SynergyManager가 SEController.Remove() 호출
```

### 비전 마법사: 아군이 주문력을 얻습니다. 비전 마법사는 더 얻습니다 (2) 15/25

```
시너지 활성화
→ SynergyManager: 모든 아군 Defender 순회
→ 비전 마법사 여부에 따라 다른 Context(15 vs 25)로 Apply
→ 시너지 비활성화 시: SynergyManager가 전체 Remove
```

### 프렐요드: 공격 시 3초 둔화 (2) 20%

```
시너지 활성화
→ SynergyManager: 프렐요드 유닛에게 프렐요드 SE 부여

프렐요드 SE (IOnAttackReactive 구현)
→ CombatSEExtension이 공격 이벤트 감지 → OnAttack 호출
→ 프렐요드 SE가 공격 대상의 SEController.Apply(둔화 SED, {감속: 20%})

둔화 SE
→ OnApply: 이동속도 감소 적용
→ OnUpdate: 타이머 감소, 3초 경과 시 제거 요청
→ OnRemove: 이동속도 복원
```
