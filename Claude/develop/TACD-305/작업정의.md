# TACD-305 작업 정의 — 고해상도 대응 (레터박스 고정 방식)

> 작성: `research` 멤버 · 2026-06 · 근거: [해상도-조사.md](해상도-조사.md) (외부 조사 + survey 코드 조사 통합)
> 확정 방향: **기준 종횡비 20:9 고정 + 레터박스** (사용자 결정)

---

## 1. 배경 / 문제 정의

- 현재 카메라 프리뷰가 `800×360`(20:9 가로). 이를 "고해상도로 수정"하는 것이 티켓 요구.
- **[survey 코드 조사로 확정 — 경우 A]** 800×360은 **렌더 해상도를 360p로 묶는 락이 아니다.** `Screen.SetResolution`·Pixel Perfect Camera·RenderTexture·URP 다운스케일 전부 없음 → 실제 빌드는 이미 기기 물리 해상도(1080p 등)로 렌더된다.
- 따라서 실제 문제는 "낮은 렌더 해상도"가 아니라:
  1. **UI 비반응형** — `CanvasScaler` 2개가 `ConstantPixelSize` → 기기 해상도가 커지면 UI가 상대적으로 쪼그라듦.
  2. **종횡비 대응 정책 부재** — 비율 다른 기기에서 어떻게 보일지 미정의.

---

## 2. 사용자 요구사항 (UR)

- **UR-1.** 어느 기기(특히 해상도가 큰 기기)에서나 **현재 보이는 화면과 동일한 모습**으로 보여야 한다.
- **UR-2.** 기준 비율과 다른 기기에서는 **레터박스(검은 띠)** 로 비율을 유지한다. (화면을 억지로 늘려 왜곡하지 않는다)
- **UR-3.** 가능한 한 **단순한 구조**로 구현한다. (화면 하나 기준 설계, 종횡비별 재배치 최소화)

---

## 3. 채택 방식 — 레터박스 고정

| 항목 | 결정 |
|---|---|
| **기준 종횡비** | **20:9 고정** (현재 800×360과 동일 비율) |
| **기준 픽셀 도면값** | **2400×1080** (800×360의 정확히 3배 → 기존 좌표 호환) |
| **비율 다른 기기** | 20:9 영역을 중앙 유지, **남는 곳에 검은 띠** |
| **큰 기기** | 동일 20:9 화면을 그대로 확대 → **보이는 내용·배치 100% 동일** |
| **UI/월드 설계 기준** | **20:9 화면 하나** 에 대해서만 설계 (가변 영역 고민 불필요) |

### 트레이드오프 (수용됨)
- 비율이 많이 다른 기기(예: 16:9 태블릿)에서는 검은 띠가 두꺼워진다. "동일한 모습" 보장의 본질적 비용.
- 한국 주류가 19.5:9~20:9 대역이라 **대부분 기기에서 띠는 없거나 매우 얇음** → 실사용 체감 손해 거의 없음.

### 지원 종횡비 정책 (← main 확정)
- **하한을 두지 않는다.** 20:9보다 길쭉하거나 넓은 **모든 기기를 동일 화면 + 레터박스로 처리.**
- 별도의 "최소 지원 종횡비" 컷오프·분기 처리 없음 → 구현 단순 유지.

---

## 4. 시스템 요구사항 (SR) / 작업 항목

> impl 멤버 대상. 코드/씬 변경 단위. (research·survey 조사 기준, 세부 구현은 impl 판단)

### SR-1. 레터박스 카메라 처리 (← UR-1, UR-2 핵심)
- 메인 카메라에 **종횡비 보정 로직** 추가: 기기 aspect가 20:9(≈2.222)와 다르면 `Camera.rect`(뷰포트)를 조정해 20:9 영역만 렌더하고 **남는 영역은 검은 띠**로 둔다.
- 기준 종횡비 `2400/1080`(=20:9)을 데이터로 상수화(매직넘버 지양). 종횡비는 이후 변경 가능성을 고려해 한 곳에서 관리.
- 화면 회전/창 크기 변경 시 재계산 (해상도 변동 대응).

### SR-2. UI 반응형화 + 레터박스 영역 정합 (← UR-1)
- UI 캔버스 2개의 `CanvasScaler`: `ConstantPixelSize → Scale With Screen Size`.
- **Reference Resolution: `2400×1080`** (현재 `800×600`은 가로 게임 비율과 불일치 → 교체).
- **Screen Match Mode / Match: 높이(Height=1) 기준** 권장 (세로 기준 균일 스케일 → 모든 기기 동일 비율).
- UI가 **레터박스된 20:9 영역과 정합**되도록 배치 (띠 위에 UI가 어색하게 걸치지 않게). 구현 방식(Screen Space-Camera 연동 등)은 impl 판단.

### SR-3. 월드 오브젝트 배치 점검 (← UR-3)
- BattleField·유닛·배경 등을 **20:9 화면 하나 기준**으로 보이도록 점검·정렬. (레터박스로 가시 영역이 항상 20:9로 고정되므로 가변 대응은 불필요)
- 카메라 Orthographic size(현재 3.3, 가시 높이 6.6유닛)가 새 기준에서 의도대로 보이는지 확인.

### SR-4. 에셋 원본 해상도 감사 (← UR-1, **TACD-305 범위 포함 — main 확정**)
- 렌더는 이미 고해상도이므로, 큰 기기에서 흐릿함이 남는다면 원인은 카메라가 아니라 **스프라이트 원본 px 부족**.
- 주요 스프라이트 원본 해상도·텍스처 임포트 설정(Max Size/PPU/Filter Mode) 점검.
- 새 기준(2400×1080)에서 흐릿하게 보이는 에셋을 식별하고, 필요 시 **고해상도 원본 재제작/교체** 로 이어짐(아트 작업).
- 픽셀아트 의도(Filter Mode=Point로 또렷한 도트 유지)인 경우는 "깨짐"이 아니므로 제외.

## 5. 범위 밖 / 비채택
- ❌ "360p 렌더 잠금 해제" — 존재하지 않는 락 (경우 A 확정).
- ❌ 종횡비별 완전 반응형 재배치 — 단순화를 위해 레터박스 고정으로 대체.
- ❌ 종횡비 자체 변경 — 20:9 유지 (한국 주류 대역).

---

## 6. Definition of Done (제안)
- [ ] 20:9 외 종횡비 기기에서 레터박스(검은 띠)가 생기고 게임 내용이 잘리지 않는다.
- [ ] 종횡비 하한 없이 길쭉/넓은 모든 기기가 동일 화면 + 띠로 처리된다.
- [ ] 해상도가 큰 기기에서 작은 기기와 **동일한 레이아웃**으로 보인다 (UI·월드 동일 비율).
- [ ] 기존 800×360(20:9) 프리뷰와 콘텐츠 배치가 일치한다.
- [ ] 기준 종횡비 값이 매직넘버 없이 한 곳에서 관리된다.
- [ ] 주요 에셋이 2400×1080 기준에서 흐릿하지 않다(또는 흐릿한 에셋이 식별·기록됨). (SR-4)

## 7. main 결정 사항 (확정 완료)
1. **에셋 해상도 감사(SR-4)** → ✅ **TACD-305 범위 포함** 으로 확정 (정식 SR로 반영됨).
2. **지원 최소 종횡비 하한** → ✅ **명시하지 않음** 으로 확정. 20:9 고정 레터박스만 적용, 그보다 길쭉/넓은 기기는 모두 동일 화면 + 띠 허용. 하한 정책 없음.
