# TACD-305 해상도 조사

> 작성: `research` 멤버 · 조사 시점 2026-06 · 대상 작업 TACD-305 (고해상도로 수정)
> 목적: 모바일 **가로** 게임에서 (1) Unity 해상도 지정 관행, (2) 2026-05~06 한국 모바일 최다 사용 해상도 파악

---

## 1. 결론 요약 (TL;DR)

- Unity 모바일은 **물리 해상도를 고정하지 않는다.** 기기 해상도는 제각각이므로, 대신 **기준 해상도(Reference Resolution) 1개를 정해 놓고 비율로 스케일**하는 방식이 표준이다.
- 한국 모바일 실사용 해상도는 거의 전부 **세로 19.5:9 ~ 20:9 계열**(물리적으로 1080×2340 / 1080×2400 = Full HD+ 급)에 몰려 있다.
- 본 게임은 **가로** 이므로 이를 가로로 뒤집은 **종횡비 ≈ 19.5:9(2.165) ~ 20:9(2.222)** 가 타깃이다.
- 현재 카메라 `800×360` = **20:9 가로** → **종횡비는 이미 적절**하다.
- ⭐ **[survey 코드 조사로 확정 — 경우 A]** 800×360은 **렌더 해상도를 고정하는 값이 아니다.** 에디터 Game 뷰 프리뷰 종횡비일 뿐이고, **실제 빌드는 이미 기기 물리 해상도(1080p 등)로 렌더된다.** `Screen.SetResolution`·Pixel Perfect Camera·RenderTexture·URP 다운스케일 **전부 없음.** → **"360p 잠금 해제" 같은 작업은 불필요.**
- 따라서 "고해상도로 수정"의 **진짜 작업**은 해상도 숫자 변경이 아니라 **① 타깃 종횡비 확정 ② 비반응형 UI(현재 ConstantPixelSize) → Scale With Screen Size 전환 ③ 새 종횡비에 맞춘 orthographic size·오브젝트 배치 재조정** 이다.
- 권장 기준 해상도(설계 도면값): 현재 비율 유지 시 **2400×1080 (20:9)**, 표준값 선호 시 **2340×1080 (19.5:9)**.
- ⚠️ 별도 리스크: **에셋 원본 해상도 감사** — 렌더는 이미 고해상도이므로, 흐릿함이 남는다면 원인은 카메라가 아니라 **스프라이트 원본 px 부족**. 이 점검은 코드와 별개 작업.

## 2. Unity 모바일에서 해상도를 "지정"하고 개발하는가?

### 2-1. 물리 해상도는 고정하지 않는다
모바일은 기기마다 해상도·종횡비가 다르기 때문에 단일 물리 해상도로 개발하지 않는다.
대신 **기준 해상도(reference resolution) 하나를 정하고, 실제 화면이 그보다 크거나 작으면 전체를 비율로 스케일**하는 것이 Unity 공식 권장 방식이다.

### 2-2. UI — CanvasScaler (Scale With Screen Size)
- `Canvas Scaler` 컴포넌트의 **UI Scale Mode = "Scale With Screen Size"** 로 설정하고 **Reference Resolution** 을 지정한다.
- 실제 화면이 기준과 종횡비가 다르면, **Screen Match Mode** 설정에 따라 어느 축을 기준으로 맞출지 결정한다.
  - `Match Width Or Height` + **Match 슬라이더(0=너비, 1=높이)** 가 핵심. **가로 게임은 보통 Height(1)에 가깝게** 둬서 세로 기준으로 맞추고 좌우 여백을 흡수하는 패턴이 흔하다.
- 올바른 **앵커링(Anchor)** + CanvasScaler 조합으로 다양한 해상도를 지원한다 (공식 문서 권장).

### 2-3. 카메라/월드 — Orthographic Size + Aspect
- 2D 가로 게임의 **Orthographic 카메라는 세로(Orthographic Size)가 화면 높이를 고정**하고, 가로 가시 범위는 **화면 aspect(width/height)에 따라 자동으로 늘었다 줄었다** 한다.
- 따라서 기기마다 **가로로 보이는 영역이 달라진다** → 본 작업의 "반응형 오브젝트 배치"는 이 가변 가로 영역을 어떻게 채우고/앵커링할지의 문제.
- 종횡비가 더 좁은 기기(예: 16:9 = 1.78)에서 좌우가 잘리거나, 더 넓은 기기(20:9 = 2.22)에서 여백이 생기는 케이스를 정의해야 함.

### 2-4. 한 줄 정리
> "해상도를 지정한다" = 물리 픽셀 고정이 아니라, **기준 해상도 + 종횡비 대응 정책(스케일·앵커·letterbox)** 을 정하는 것.

## 3. 2026-05 한국 모바일 해상도 점유율 (Statcounter)

> 출처: Statcounter Global Stats — Mobile Screen Resolution, Republic of Korea, **2026년 5월**
> ⚠️ 아래 수치는 **논리(CSS) 해상도이며 세로(portrait) 기준**이다. 실제 물리 픽셀은 기기 픽셀 배율(@2x~@3x)을 곱해야 함.

| 순위 | 논리 해상도(세로) | 점유율 | 종횡비(h/w) | 대표 기기군 | 추정 물리 해상도 |
|---|---|---|---|---|---|
| 1 | 414×896 | 16.91% | 2.164 (≈19.5:9) | iPhone XR/11/XS Max | 828×1792 (@2x) |
| 2 | 384×832 | 9.6% | 2.167 (≈19.5:9) | Android 다수 | ~1080×2340 |
| 3 | 360×780 | 8.94% | 2.167 (≈19.5:9) | Android 다수 | ~1080×2340 |
| 4 | 412×892 | 7.2% | 2.165 (≈19.5:9) | Android(Pixel/삼성 일부) | ~1080×2340 |
| 5 | 390×844 | 4.97% | 2.164 (≈19.5:9) | iPhone 12~14 | 1170×2532 (@3x) |
| 6 | 360×800 | 3.3% | 2.222 (20:9) | Android(삼성 다수) | 1080×2400 (@3x) |

(상위 6종 합 ≈ 50.9%, 나머지는 긴 꼬리 분포)

### 3-1. 해석
- **단일 최다 해상도**는 `414×896`(아이폰 계열) 이지만, **점유율이 한 곳에 몰리지 않고 19.5:9 대역으로 분산**되어 있다.
- 6종 중 5종이 **19.5:9(2.16~2.17)**, 1종이 **20:9(2.22)**. → **실효 타깃 종횡비는 19.5:9 ~ 20:9 구간.**
- 물리 해상도로 환산하면 한국 주류는 **1080×2340 (19.5:9)** 와 **1080×2400 (20:9)** = **Full HD+ 급**.

### 3-2. 가로 게임 관점 변환 (세로→가로 스왑)
| 종횡비 | 세로 표기 | **가로 표기(본 게임)** | 비고 |
|---|---|---|---|
| 19.5:9 | 1080×2340 | **2340×1080** | iPhone·Android 최다 대역 |
| 20:9 | 1080×2400 | **2400×1080** | 현재 800×360과 동일 비율 |
| 16:9(참고) | 1080×1920 | 1920×1080 | 구형/일부 태블릿, 좌우 잘림 케이스 |

## 4. 현재 프로젝트 설정 조사 결과 (survey 멤버, read-only)

> 목적: 800×360이 **단순 프리뷰 값(경우 A)** 인지 **실제 렌더를 360p로 묶는 락(경우 B)** 인지 코드/씬에서 사실 확정.
> **판정: 경우 A.** 렌더 해상도를 고정하는 코드/컴포넌트가 **하나도 없음.** 실제 빌드는 런타임에 기기/창 크기로 렌더됨.

| 점검 대상 | 결과 | 근거 |
|---|---|---|
| 메인 카메라 | Orthographic, **size 3.3**(가시 높이 6.6유닛), 가로는 aspect 가변 | `Battle.unity:8800-8844` |
| RenderTexture 타깃 | **없음** (화면 직접 렌더) | `m_TargetTexture: {fileID:0}` |
| 뷰포트 클리핑 | 없음 (x0 y0 w1 h1) | `m_NormalizedViewPortRect` |
| `Screen.SetResolution` | **0건** | 전역 grep (.cs) |
| Pixel Perfect Camera | **비활성** | `m_PixelPerfect=0` |
| 360p RenderTexture 에셋 | **없음** (*.renderTexture 0건) | 프로젝트 전역 |
| URP 다운스케일 | 없음 (`m_RenderScale:1`, MSAA1, HDR1) | `UniversalRP.asset:29` |
| 스탠드얼론 기본 해상도 | **1920×1080** (360 아님) | `ProjectSettings.asset:45-46` |
| `.cs` 내 800/360 하드코딩 | **0건** | 전역 검색 |

### ⚠️ 핵심 발견 — 진짜 병목은 "렌더 락"이 아니라 "비반응형 UI"
- UI 캔버스 2개의 `CanvasScaler` 가 **`UiScaleMode = 0 (ConstantPixelSize)`** → **해상도가 바뀌어도 UI가 스케일되지 않음.** (현재 UI 비반응형)
- `ReferenceResolution`이 `{800, 600}`으로 적혀 있으나 ScaleMode가 0이라 **무시되는 값**이다. (카메라 800×360과도 무관)
- 근거: `Battle.unity:10703, 10872`

> 결론: **"360p 잠금 해제" 작업은 불필요(존재하지 않는 락).** 고해상도 전환의 실제 작업은 아래 시사점의 3가지로 압축된다.

## 5. TACD-305 적용 시사점

1. **렌더 해상도 변경은 작업 대상이 아님** — 이미 기기 물리 해상도로 렌더된다(경우 A 확정). 종횡비 20:9 방향성도 맞음. 실제 작업은 아래 2·3·4번.
2. **기준 해상도 후보**
   - (A) **2400×1080 (20:9)** — 현재 비율 그대로 유지, 마이그레이션 리스크 최소. (800×360 × 3배)
   - (B) **2340×1080 (19.5:9)** — 실사용 최다 대역에 정확히 일치. 단, 현재 20:9 대비 가로가 살짝 줄어 좌우 배치 재검토 필요.
3. **반드시 정의할 대응 케이스**
   - 19.5:9 ~ 20:9: 주력 대역, 무손실로 채워야 함.
   - 16:9(1.78) 같은 **더 좁은 화면**: 좌우 콘텐츠 잘림 또는 pillarbox 정책 결정 필요.
   - 더 넓은 화면/폴더블: 여백 또는 확장 영역 정책.
   - **Safe Area**(노치·펀치홀·제스처바): 상단/하단 UI는 `Screen.safeArea` 기준 앵커링 필요.
4. **SR 분해 예상축**:
   - (a) **UI 반응형화** ← 최우선·실질 작업. `CanvasScaler` 2개를 `ConstantPixelSize → Scale With Screen Size`로 변경, Reference Resolution(택한 기준값)·Match(가로 게임은 Height쪽) 설정.
   - (b) 카메라 종횡비/Orthographic size 정책 (새 종횡비에 맞춰 가시 영역 결정).
   - (c) 월드 오브젝트 앵커링·가변 가로 영역 대응.
   - (d) Safe Area 적용 (노치·펀치홀·제스처바).

## 6. 확인 필요 (기획/main 결정 사항)
- **기준 해상도 확정**: 비율 유지(2400×1080, 20:9) vs 실사용 최다(2340×1080, 19.5:9) 중 택1.
- 지원 종횡비 **최소~최대 범위**(예: 16:9 ~ 21:9)와 범위 밖 처리 정책(letterbox/crop/확장).
- 태블릿(4:3·16:10) 지원 여부.
