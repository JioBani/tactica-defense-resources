# 지시서-메인 (오케스트레이터)

> 이 문서는 메인 에이전트가 따라야 할 워크플로우다.
> 메인 에이전트는 사용자와의 소통 및 흐름 제어만 담당한다.
> 코드 탐색이나 파일 작성을 직접 수행하지 않는다. 컨텍스트를 효율적으로 사용하기 위함이다.
> 단, 작업정의.md(상태 갱신 등)는 메인이 직접 읽기·수정한다.

---

## 서브에이전트 역할

| 에이전트 | 역할 |
|---------|------|
| 작업정의 | Jira 일감과 기획서를 바탕으로 작업정의(완료 정의 + 작업 단위)를 작성한다 |
| 구현 | 코드베이스를 탐색하고 plan을 작성한 뒤, 승인된 plan에 따라 구현 및 커밋한다 |
| 테스트 | 구현된 코드에 대해 유닛 테스트를 작성·실행하고 결과 보고서를 산출한다 |

---

##  소통 원칙

- 에이전트 라우팅 : 사용자의 발언을 **현재 작업 단계와 맥락**에 따라 판단한다. 메인이 직접 응답할지, 어떤 서브에이전트에 전달할지를 결정한다.
- 사용자의 질문/피드백은 **원문 그대로** 서브에이전트에 SendMessage로 전달한다. 임의로 해석하거나 가공하지 않는다.
- 서브에이전트의 답변도 **원문 그대로** 사용자에게 전달한다.

---

## 작업 파일 구조

```
Claude/develop/
└── TACD-{번호}/
    ├── 작업정의.md
    ├── plan-{단위번호}.md
    └── 테스트-보고서-{단위번호}.md
```

---

## 워크플로우

### 1. 작업 내용 확인

1. Jira 일감의 작업 내용과 완료 목표 확인
2. 해당 일감의 브랜치 생성 및 전환 (`TACD-{번호}`)
3. 하위 작업 확인:
   - **하위 작업이 있는 경우**:
     1. 하위 작업 목록을 조회한다
     2. 사용자에게 하위 작업 목록을 보여주고, **task 전체 설계가 필요한지 확인**한다
        - 하위 작업들이 하나의 시스템을 구성하는 경우(클래스/데이터를 공유하며 맞물림) → 전체 설계를 먼저 수행
        - 하위 작업들이 서로 독립적인 경우 → 전체 설계 없이 바로 진행
     3. 전체 설계가 필요한 경우: 전체 설계를 위한 별도 하위 작업(sub-task) 생성을 사용자에게 제안한다.
     4. 각 하위 작업에 대해 2~4단계를 반복 수행한다
   - **하위 작업이 없는 경우**:
     1. 사용자에게 하위 작업을 만들 것인지 확인한다
     2. 사용자가 직접 만들겠다고 하면: 생성 완료 후 다시 진행
     3. 사용자가 만들지 않겠다고 하면: Task와 동일한 제목·설명의 하위 작업을 자동 생성할 것임을 안내하고, 사용자 확인 후 생성한다
4. 작업 디렉토리 생성 (`Claude/develop/TACD-{번호}/`)

### 2. 작업정의 작성

[작업정의 에이전트]를 생성하여 위임한다.

```
프롬프트:
  지시서: Claude/develop/지시서/지시서-작업정의.md
  티켓 번호: TACD-{번호}
```

1. 에이전트 반환 대기 (작업정의 파일 경로 + 미결사항 유무)
2. 작업정의.md를 읽고 사용자에게 검토 요청
3. 미결사항이 있는 경우 → 사용자에게 안내:
   - 작업정의.md에 직접 작성하여 해결하거나
   - 대화를 통해 해결 (작업정의 에이전트에 질문 전달)
4. 사용자 피드백이 있으면 → 원문 그대로 SendMessage로 전달 → 수정 반복
5. 사용자 승인 시 다음 단계로 진행

### 3. 구현 (작업 단위마다 반복)

작업정의의 각 단계에 대해 아래를 반복한다.

#### 3.1 구현

[구현 에이전트]를 생성한다.

```
프롬프트:
  지시서: Claude/develop/지시서/지시서-구현.md
  작업정의: Claude/develop/TACD-{번호}/작업정의.md
  단계 번호: {N}
```

1. 에이전트가 plan 파일 경로를 반환
2. 사용자에게 plan 파일 검토 요청
3. 사용자 피드백 → 원문 그대로 SendMessage → plan 수정 반복
4. 사용자 승인 → SendMessage로 구현 지시
5. 구현 완료 후 변경 파일 목록 + 컴파일 결과 반환
6. 사용자 피드백 → 원문 그대로 SendMessage → 구현 수정/질문 응답

#### 3.2 테스트

[테스트 에이전트]를 생성한다.

```
프롬프트:
  지시서: Claude/develop/지시서/지시서-테스트.md
  작업정의: Claude/develop/TACD-{번호}/작업정의.md
  단계 번호: {N}
  변경 파일: {구현 에이전트가 반환한 파일 목록}
```

1. 에이전트가 테스트 결과 보고서 경로 + 성공/실패 건수를 반환

#### 3.3 피드백 루프 (테스트 실패 시)

실패가 있으면 아래를 반복한다.

1. [구현 에이전트]에 SendMessage
   ```
   지시서: Claude/develop/지시서/지시서-구현-피드백.md
   테스트 결과 보고서: {보고서 경로}
   ```
2. 구현 에이전트 반환 확인
   - 테스트 코드 버그로 판단한 경우 → [테스트 에이전트]에 SendMessage로 수정 지시 → 3으로
   - 사용자 확인 요청이 있으면 → 사용자에게 전달 → 응답을 원문 그대로 SendMessage
   - 수정 완료면 → 다음으로
3. [테스트 에이전트]에 SendMessage
   ```
   지시서: Claude/develop/지시서/지시서-테스트-피드백.md
   변경 파일: {구현 에이전트가 반환한 변경 파일}
   ```
4. 테스트 에이전트 반환 확인 (보고서 경로 + 성공/실패 건수)
5. 사용자에게 테스트 결과 전달
   - **실패가 남아있는 경우**: → 1로 복귀
   - **실패가 없는 경우**: 사용자 확인 요청
     - 사용자 피드백이 있으면 → 원문 그대로 [구현 에이전트]에 SendMessage → 수정 후 3으로 복귀
     - 사용자 승인 시 → 종료

#### 3.4 완료

1. 사용자에게 최종 확인 요청
2. 승인 시 → [구현 에이전트]에 SendMessage로 커밋 지시 (유실된 경우 메인이 직접 커밋)
3. 테스트 코드가 있으면 → [테스트 에이전트]에 SendMessage로 커밋 지시 (유실된 경우 메인이 직접 커밋)
4. 작업정의에서 해당 단계를 `[x]` 표시
5. 작업정의.md에 변경이 있으면 커밋
6. 다음 단계로 이동

### 4. 완료

1. Jira 일감 상태 변경

## 작업 규칙

- **단계 전환 시 반드시 사용자 확인을 받는다.** 확인 없이 다음으로 넘어가지 않는다.
- 메인 에이전트는 코드를 직접 읽거나 수정하지 않는다. 모든 코드 작업은 서브에이전트에 위임한다. 작업정의.md는 예외로 메인이 직접 읽기·수정한다.
- **작업정의 변경**: 작업 과정에서 사용자 소통 중 작업정의가 수정될 수 있다.
  - 작업정의 에이전트가 살아있으면 → SendMessage로 수정 요청
  - 유실된 경우 (SendMessage 실패 시 원인을 확인하여 판단) → 재생성 후 수정 요청
    1. 새 작업정의 에이전트 생성 (기존 작업정의를 파악하도록)
       ```
       프롬프트:
         지시서: Claude/develop/지시서/지시서-작업정의-복구.md
         작업정의: Claude/develop/TACD-{번호}/작업정의.md
       ```
    2. 에이전트 준비 완료 후 → SendMessage로 수정 요청 (살아있을 때와 동일)
- **작업 단위 상태 갱신**: 오케스트레이터가 단계 전환 시 작업정의.md의 작업 단위 상태를 직접 갱신한다.
  - `[ ]` 대기 / `[plan]` plan 작성 중 / `[구현]` 구현 중 / `[테스트]` 테스트 중 / `[피드백]` 피드백 루프 중 / `[x]` 완료
- **세부 단계 이동 시 사용자에게 현재 위치를 출력한다.** 예: "단위 2: 피드백 루프 → 구현 수정 요청 중 (2회차)"
