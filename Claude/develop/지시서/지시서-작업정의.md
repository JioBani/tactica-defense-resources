# 지시서-작업정의

> 이 문서는 작업정의 에이전트가 따라야 할 절차다.
> Jira 일감과 기획서를 바탕으로 작업정의(완료 정의 + 작업 단위)를 작성한다.

---

## 입력

- 티켓 번호: `TACD-{번호}`

## 산출물

- `Claude/develop/TACD-{번호}/작업정의.md`

## 반환

- 작업정의 요약 (완료 정의 목록, 작업 단위 목록, 미결사항)

---

## 절차

### 1. 정보 수집

1. Jira에서 티켓 조회 — 작업 내용과 완료 목표 확인
2. 기획서(`Document/기획서/`)에서 관련 요구사항 수집
3. 코드베이스 탐색은 **꼭 필요한 경우에만** 수행한다
   - 필요한 경우: 기존 데이터 구조 확인, 이슈 자체가 데이터 설계에 관한 것, 기존 동작 방식 파악이 불가피한 경우
   - 탐색 범위는 이슈가 직접 다루는 데이터 구조까지로 한정한다. 소비자 코드(Manager, 이벤트 등)는 탐색하지 않는다.

### 2. 작업정의 작성

템플릿(`Claude/develop/템플릿/작업정의-템플릿.md`)에 따라 작성한다.

#### 완료 정의 작성 원칙

- 비즈니스 동작은 반드시 **기능/동작적으로 서술**한다. 완료 정의만 보고 QA와 기획이 이해할 수 있어야 한다.
- 예외: 데이터베이스, 프레임워크, 라이브러리 제작 등의 경우 코드적인 부분을 서술할 수 있다.
- **데이터의 내용(무엇을)**에 집중하고, **데이터의 형태(어떤 타입/구조로)**는 명시하지 않는다.

예시:
- 나쁨: "시너지별 카운트(Dictionary<SynergyDefinitionData, int>)를 반환한다"
- 좋음: "시너지별 카운트를 산출한다"

#### 작업 단위 작성 원칙

- 완료 정의를 한번에 작업하기 좋은 단위로 묶는다.
- 각 단위에 번호와 이름을 부여하고, 하위에 해당 완료 정의를 배치한다.
  - 구현 시 **한 단위 = 한 번의 plan 사이클**로 수행할 예정


### 3. 미결사항 파악

구현을 진행하기 전에 사용자의 결정이 필요한 사항을 식별한다.

- 예: 데이터 형식, 네이밍 규칙, 작업 범위, 기획 미확정 사항 등
- 작업정의.md에 미결사항 섹션으로 정리한다

### 4. 반환

메인 에이전트에 아래를 반환한다.

- 작업정의 파일 경로
- 미결사항 유무 (있으면 "미결사항 있음"만 전달, 내용은 파일에 기록)
- 코드 탐색을 수행한 경우: 탐색한 코드 파일 수 (기획서 등 일반 문서 제외)

반환 후, 메인 에이전트를 통해 사용자 피드백이 올 수 있다. 피드백에 따라 작업정의를 수정하고, 수정 내역과 함께 다시 반환한다.
