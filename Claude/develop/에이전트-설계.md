# 에이전트 분리 설계

## 프롬프트 설계 원칙

- 서브에이전트에게 전달하는 **프롬프트는 최소화**한다.
- 각 에이전트의 수행 절차는 **별도 지시서(md 파일)**로 분리한다.
- 프롬프트에는 **지시서 경로 + 작업마다 변하는 값(변수)**만 전달한다.
- 서브에이전트는 지시서를 읽고 수행한다.

### 메인 ↔ 서브 소통 원칙

- 사용자의 질문/피드백은 **원문 그대로** SendMessage로 전달한다. 메인이 임의로 해석하거나 가공하지 않는다.
- 서브에이전트의 답변도 **원문 그대로** 사용자에게 전달한다.

### 프롬프트 구조 (공통)

```
지시서: {지시서 경로}
티켓 번호: TACD-{번호}
{에이전트별 추가 변수}

위 지시서를 읽고 수행하라.
```

### 파일 구조

```
Claude/develop/
├── 지시서/
│   ├── 지시서-메인.md          # 메인 에이전트 (오케스트레이터)
│   ├── 지시서-작업정의.md       # 작업정의 에이전트 지시서
│   ├── 지시서-구현.md          # 구현 에이전트 지시서 (plan + 구현)
│   ├── 지시서-작업정의-복구.md   # 작업정의 에이전트 재생성 시 컨텍스트 복구 지시서
│   ├── 지시서-구현-피드백.md    # 구현 에이전트 피드백 루프 지시서
│   ├── 지시서-테스트.md         # 테스트 에이전트 지시서
│   └── 지시서-테스트-피드백.md   # 테스트 에이전트 피드백 루프 지시서
└── 템플릿/
    └── 작업정의-템플릿.md       # 작업정의 작성 시 템플릿
```

---

## 메인 에이전트 (오케스트레이터)

사용자와의 소통 및 흐름 제어만 담당한다. 코드 탐색이나 파일 작성을 직접 수행하지 않는다.

- 사용자 요청 수신 및 응답
- 워크플로우 단계 전환 판단
- Jira 조회/생성/상태 변경
- Git 브랜치 생성/전환
- 서브에이전트 생성 및 SendMessage로 피드백 전달
- 서브에이전트 산출물을 사용자에게 전달하고 승인/수정 요청 중계

## 서브에이전트

### 1. 작업정의 작성 에이전트

| 항목 | 내용 |
|------|------|
| **지시서** | `Claude/develop/지시서/지시서-작업정의.md` |
| **변수** | 티켓 번호 |
| **수행** | 기획서 읽기, 필요 시 코드베이스 탐색(데이터 구조 한정), 작업정의 작성 |
| **산출물** | `Claude/develop/TACD-{번호}/작업정의.md` |
| **반환** | 작업정의 요약 (완료 정의 목록, 작업 단위 목록, 미결사항) |
| **SendMessage** | 사용자 피드백 → 작업정의 수정 (컨텍스트 유지) |

### 2. 구현 에이전트 (작업 단위당 1회, plan + 구현 통합)

| 항목 | 내용 |
|------|------|
| **지시서** | `Claude/develop/지시서/지시서-구현.md` |
| **변수** | 작업정의 경로, 대상 단계 번호 |
| **수행** | 코드베이스 탐색 → plan 작성 → 사용자 승인 대기 → 코드 구현 → Unity 컴파일 확인 → 사용자 최종 확인 → 커밋 |
| **산출물** | `Claude/develop/TACD-{번호}/plan-{단계번호}.md`, 코드 변경, git commit |
| **반환** | plan 파일 경로 (plan 단계) / 변경된 파일 목록 + 컴파일 결과 (구현 단계) / 커밋 해시 (커밋 단계) |
| **SendMessage** | plan 수정, 구현 수정, 사용자 질문 응답, 최종 확인 후 커밋 지시 (탐색~구현 컨텍스트 전체 유지) |

### 3. 테스트 에이전트 (작업 단위당 1회)


| 항목 | 내용 |
|------|------|
| **지시서** | `Claude/develop/지시서/지시서-테스트.md` |
| **변수** | 티켓 번호, 대상 단계 번호, 변경된 파일 목록, 완료 정의 |
| **수행** | 유닛 테스트 작성 및 실행 → 사용자 최종 확인 → 커밋 |
| **산출물** | 테스트 코드, git commit |
| **반환** | 테스트 결과 보고서 경로 + 성공/실패 건수 / 커밋 해시 (커밋 단계) |
| **SendMessage** | 실패 시 수정 후 재실행, 최종 확인 후 커밋 지시 |

## 흐름 요약

```
메인(오케스트레이터)
│
├─ 1. Jira 확인, 브랜치 생성, 하위 작업 처리
│
├─ 2. [작업정의 에이전트] 생성
│   │  프롬프트: 지시서 경로 + 티켓번호
│   ├─ 산출물(작업정의.md) → 사용자 검토
│   └─ SendMessage로 수정 반복
│
└─ 3. 작업 단위마다 반복:
    ├─ [구현 에이전트] 생성
    │   │  프롬프트: 지시서 경로 + 작업정의 경로 + 단계 번호
    │   ├─ plan 작성 → 메인이 사용자에게 전달 → 검토
    │   ├─ SendMessage로 plan 수정 반복
    │   ├─ 사용자 승인 → SendMessage로 구현 지시
    │   ├─ 코드 구현 + 컴파일 확인
    │   └─ SendMessage로 구현 수정/질문 응답
    ├─ [테스트 에이전트] 생성
    │   │  프롬프트: 지시서 경로 + 작업정의 경로 + 단계 번호 + 변경 파일
    │   └─ 테스트 작성 + 실행
    ├─ 피드백 루프 (테스트 실패 시 반복):
    │   ├─ 실패 내용 → [구현 에이전트]에 SendMessage로 수정 지시
    │   ├─ 구현 수정 완료
    │   ├─ [테스트 에이전트]에 SendMessage로 재실행 지시
    │   └─ 여전히 실패 시 → 피드백 루프 반복
    ├─ 사용자 최종 확인 → 구현/테스트 에이전트에 커밋 지시
```
