# 데미지계산 — 시스템 관점

> 이 문서는 AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실이다. 이 문서는 코드를 재설명하지 않는다 — "인덱스 + α"에 한정한다.
>
> - **분류**: 기능(단순 구현) — 대응 UR 노드: `전장 > 유닛 스탯 & 데미지 > 데미지 계산`.

## 1. 한 줄 정의

공격자·피격자의 스탯 시트와 데미지 타입(물리/마법)을 받아, 단일 공격력을 기준으로 치명타 → 방어력 → 피해 증가 → 피해 감소 순의 결정적 파이프라인을 거쳐 최종 적용 피해 수치를 산출하는 순수 계산 유틸리티.

## 3. 구조 맵 (위치 중심)

진입점부터.

- **진입점**: `DamageCalculator.Calculate(attackerStats, victimStats, damageType, [isCritical], [skillCoefficient])` — `Assets/Scenes/Battle/Feature/Unit/Damage/DamageCalculator.cs` (공격력 기반 표준 경로)
- **사전 계산 진입점**: `DamageCalculator.Calculate(baseDamage, attackerStats, victimStats, damageType, [isCritical])` — 동 파일 (baseDamage를 외부에서 산출한 스킬용 파이프라인 진입)
- **스킬 계수 산출**: `SkillCoefficientCalculator.Calculate(coeff, statSheet)` — `Assets/Scenes/Battle/Feature/Unit/Damage/SkillCoefficientCalculator.cs`
- **데미지 타입**: `DamageType` (Physical / Magical) — `Assets/Common/Data/Damage/DamageType.cs`

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도 (오버로드 2계열)**: "공격력 기반" 진입점과 "baseDamage 선입력" 진입점이 나뉜 이유는 소비처 차이다. 평타는 공격자 공격력을 그대로 기준으로 삼지만, 스킬은 `SkillCoefficient`로 스탯을 스케일링해 자체 기본 데미지를 먼저 만든 뒤 치명타/방어/증감 파이프라인만 공유해야 하기 때문. 두 계열이 동일한 후반 파이프라인을 쓰도록 의도적으로 갈라 놓았다.
- **분할 의도 (`isCritical` 외부 주입 오버로드)**: 치명타를 내부 난수로 굴리는 버전과 외부에서 주입하는 버전을 함께 둔 것은 테스트 결정성과 연출/사전판정(치명타 여부를 미리 정해 두는 경우)을 위해서다 — 같은 공식을 난수 의존 없이도 검증·재현할 수 있게 한 것.
- **도메인적 의도 (단일 공격력 기반)**: 물리/마법을 별개 공식이 아니라 `DamageType` 분기만으로 동일 공식에 태운다. 공격력·방어력 채널만 바뀌고 치명타·피해증감 단계는 타입 무관 공통 — 기획서 §1.3의 "단일 공격력 기반" 설계를 그대로 인코딩.
- **비자명한 관례**: 방어력·피해감소는 0~1 비율(곱연산 차감)로 다뤄지고, 최종 수치는 소수점 버림 + 최소 0으로 클램프된다. 정확한 단계 순서·반올림 규칙은 코드가 단일 진실이며, 변경 시 `Assets/Tests/Editor/` 의 대응 테스트가 가드한다.
