# 생명크리스탈 — 시스템 관점

> 이 문서는 **AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록** 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실(single source of truth)이다. 이 문서는 코드를 재설명하지 않는다 — **"인덱스 + α"** 에 한정한다.
>
> - **분류**: 기능 — 대응 UR 노드: `전장 > 라운드 & 페이즈` (전장 승리·패배 조건 중 패배 측)
> - **이 문서가 다루는 것**: ① 이 대상이 시스템에서 무엇인가(한 줄) ② 도메인 개념 ↔ 코드(파일 경로)의 매핑 ③ 무엇이 어디에 있는가(구조 맵) ④ 다른 코드가 가져다 쓸 헬퍼 ⑤ 코드만 봐선 모르는 의도(+α).
> - **이 문서가 다루지 않는 것**: 코드의 동작을 풀어쓴 설명서, 알고리즘·자료구조 해설, 메서드 본문 요약.

> ⚠️ **빈 모듈 경고**: 대상 코드 경로(`Assets/Scenes/Battle/Feature/LifeCrystal/`)는 현재 **빈 스텁 1개만 존재**한다. `LifeCrystalManager` 는 필드·메서드가 없는 빈 `MonoBehaviour` 다. 생명크리스탈의 도메인 신호(생명 포인트 변화, 파괴)는 **대상 경로 밖**의 이벤트 DTO 로만 정의되어 있고, 실제 차감/판정/파괴 로직을 가진 구현체는 이 경로에 아직 없다. 아래 매핑은 "현재 코드에 인코딩된 표면"만 기록한다.

---

## 1. 한 줄 정의

생명크리스탈은 전장 패배 조건의 도메인 앵커로, 침략자가 도달하면 생명 포인트가 차감되고 0이 되면 전장 패배(크리스탈 파괴)를 통지할 책임을 진다 — **단, 현재 대상 경로에는 책임을 수행할 구현이 없는 빈 스텁만 존재한다.**

## 2. 도메인 ↔ 코드 매핑

| 도메인 개념 | 코드 식별자 | 위치(파일 경로) | 책임 (한 줄) |
|------------|------------|----------------|-------------|
| 생명크리스탈 (관리 주체) | `LifeCrystalManager` | `Assets/Scenes/Battle/Feature/LifeCrystal/LifeCrystalManager.cs` | 생명크리스탈을 관장할 자리(스텁) — 현재 책임 코드 없음 |
| 생명 포인트 변화(통지) | `OnLifeCrystalPointChangedEventDto` | `Assets/Scenes/Battle/Feature/Events/OnLifeCrystalPointChangedEventDto.cs` | 현재 생명 포인트(`LifePoint`)를 외부에 통지하는 이벤트 표면 |
| 크리스탈 파괴 = 전장 패배(통지) | `OnLifeCrystalDestroyEventDto` | `Assets/Scenes/Battle/Feature/Events/OnLifeCrystalDestroyEventDto.cs` | 생명크리스탈 파괴(패배 진입)를 외부에 통지하는 이벤트 표면 (페이로드 없음) |

> 두 이벤트 DTO 는 대상 경로(`LifeCrystal/`)가 아니라 인접한 `Events/` 디렉토리에 있으며 `IGameEvent` 를 구현한다. 즉 생명크리스탈의 외부 표면(도메인 신호)은 이미 정의되어 있으나, 그 신호를 발생시키는 로직(차감·판정·파괴)은 대상 경로에 아직 인코딩되지 않았다.

## 3. 구조 맵 (위치 중심)

- **진입점**: 없음 — `LifeCrystalManager` 는 빈 `MonoBehaviour` 로 진입 메서드/필드가 없다. · `Assets/Scenes/Battle/Feature/LifeCrystal/LifeCrystalManager.cs`
- **외부 표면(이벤트, 대상 경로 밖)**: `OnLifeCrystalPointChangedEventDto`, `OnLifeCrystalDestroyEventDto` — `Assets/Scenes/Battle/Feature/Events/` — `GlobalEventBus` 를 통해 흐르는 도메인 신호 정의

### 3.1 제공 헬퍼 (다른 코드가 찾아오는 곳)

제공 헬퍼 없음. (구현 스텁 상태이므로 다른 코드가 가져다 쓸 공개 표면이 `LifeCrystalManager` 에 없다. 생명크리스탈 신호를 소비하려는 코드는 헬퍼 대신 `Events/` 의 두 이벤트 DTO 를 `GlobalEventBus` 로 구독한다.)

### 3.2 의존 관계

- **밖으로 의존**: `UnityEngine` (MonoBehaviour). 관련 이벤트 DTO 는 `Common.Scripts.GlobalEventBus`(`IGameEvent`)에 의존하나, 이는 대상 경로 밖.
- **밖에서 의존**: 없음 (확인된 참조 없음). `LifeCrystalManager` 를 참조하는 코드는 코드베이스에 없다.

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도**: 생명크리스탈은 `Battle/Feature/` 의 다른 기능들(Round, Unit, Synergy 등)과 동일하게 독립 Feature 폴더로 분리될 예정인 자리다. 빈 스텁 + 인접 `Events/` 의 신호 DTO 만 선반영된 상태는, 패배 조건을 이벤트 기반으로 느슨하게 결합하려는 의도(생명 차감/파괴를 직접 호출이 아니라 `GlobalEventBus` 통지로 전파)를 시사한다.
- **도메인적 의도**: 생명크리스탈 파괴 = 전장 패배. 따라서 이 기능은 UR 트리의 `전장 > 라운드 & 페이즈` 가 정의하는 "전장 승리·패배 조건"의 **패배 측 트리거**에 해당한다.
- **비자명한 제약/관례**: `OnLifeCrystalDestroyEventDto` 는 페이로드가 없는 마커 이벤트다 — 파괴 사실 자체가 신호이며 추가 데이터 없이 패배 처리를 발화하는 관례.
