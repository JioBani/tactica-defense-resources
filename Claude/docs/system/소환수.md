# 소환수 — 시스템 관점

> 이 문서는 **AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록** 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실(single source of truth)이다. 이 문서는 코드를 재설명하지 않는다 — **"인덱스 + α"** 에 한정한다.
>
> - **분류**: 기능 — 대응 UR 노드: `전장 > 소환수`
> - **이 문서가 다루는 것**: ① 이 대상이 시스템에서 무엇인가(한 줄) ② 도메인 개념 ↔ 코드(파일 경로)의 매핑 ③ 무엇이 어디에 있는가(구조 맵) ④ 다른 코드가 가져다 쓸 헬퍼 ⑤ 코드만 봐선 모르는 의도(+α).
> - **이 문서가 다루지 않는 것**: 코드의 동작을 풀어쓴 설명서, 알고리즘·자료구조 해설, 메서드 본문 요약.

---

## 1. 한 줄 정의

`Defender` 는 소환술사가 소환하는 전투 유닛(고양이) 본체로서, 유닛 공통 능력(스탯·상태이상·행동 상태) 위에 *소환수만의 정체성* — 보유 시너지·배치 위치·드래그/배치/스폰의 전장 연동 — 을 더하는 컴포넌트다.

### 3.2 의존 관계

- **밖으로 의존**:
  - `Units.Unit` (상속) — 스탯(`StatSheet`)·행동 상태(`ActionStateController`)·상태이상(`StatusEffectController`)·스폰 초기화 기반 (`StatusEffect` 인프라 포함)
  - `GlobalEventBus` — 드래그/배치/스폰·제거 통지 발행
  - `Draggable2D` (`Common.Scripts.Draggable`) — 드래그 입력 표면
  - `IStateListener<ActionStateType>` / `ActionStateController` — 페이즈(Waiting 등) 상태 수신
  - `SynergyDefinitionData` (`Common.Data.Synergies`) — 보유 시너지 정의 타입
  - `UnitGenerator` / `UnitLoadOutData` — 소환수 인스턴스 생성·설정
  - `WaitingAreaReferences` / `ExclusiveDropZone2D` — 대기석 자리 조회·배치
- **밖에서 의존**:
  - 시너지 시스템 — `AddSynergy`/`Synergies` 로 소환수별 시너지 부여·집계 (`system/시너지.md`)
  - 합성(승급) 로직 — `DefenderManager.Defenders` 로 동일 소환수 집계
  - 전장 승패 판정 — `IsAllDefenderDowned()`

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도**: 소환수 정체성(`Defender`)을 유닛 공통 기반(`Unit`)에서 분리한 것은, 소환술사·침략자도 같은 `Unit` 능력을 공유하되 *배치·시너지 보유·드래그*는 소환수에게만 있는 도메인이기 때문이다.
- **도메인적 의도**: 시너지를 `Defender` 인스턴스에 보유시키는 구조는 "시너지가 소환수 종류(개체)에 귀속된다"는 도메인 규칙(전장마다 소환수 특성이 랜덤 배정)을 코드로 인코딩한 것이다.
- **비자명한 관례**: 정비(Waiting) 페이즈 진입 시 전체 회복은 소환수의 라운드 간 라이프사이클 규칙으로, 행동 상태 머신(`ActionStateController`)에 리스너로 올라타 트리거된다 — HP 회복 시점이 소환수 코드가 아니라 페이즈 전이에 결속되어 있다.
- **상태 통지 컨벤션**: 소환수의 생성/제거·드래그·배치 변경은 직접 호출이 아니라 `GlobalEventBus` 이벤트(`OnDefenderChangedEventDto` 등)로 외부에 알리는 단방향 통지 구조다 — 소비처(시너지·합성·UI)는 이벤트로 반응한다.
