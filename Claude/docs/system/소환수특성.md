# 소환수특성 — 시스템 관점

> 이 문서는 **AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록** 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실(single source of truth)이다. 이 문서는 코드를 재설명하지 않는다 — **"인덱스 + α"** 에 한정한다.
>
> - **분류**: 기능 — 대응 UR 노드: `전장 > 시너지 > 소환수 특성`. 전장마다 5개 효과를 소환수 종류에 랜덤 분배.
> - **이 문서가 다루는 것**: ① 이 대상이 시스템에서 무엇인가(한 줄) ② 도메인 개념 ↔ 코드(파일 경로)의 매핑 ③ 무엇이 어디에 있는가(구조 맵) ④ 다른 코드가 가져다 쓸 헬퍼 ⑤ 코드만 봐선 모르는 의도(+α).
> - **이 문서가 다루지 않는 것**: 코드 동작의 풀어쓴 설명서, 분배 알고리즘 해설, 메서드 본문 요약. 그건 코드를 보면 된다.

---

## 1. 한 줄 정의

출전 술사들의 소환수 풀 전체에, 전장 단위 특성 풀에서 뽑은 시너지를 소환수 종류별로 균등 랜덤 배정하는 분배 서비스다 (배정 결과 매핑만 산출하고, 카운트·티어·효과 부여는 시너지 시스템이 이어받는다).

## 2. 도메인 ↔ 코드 매핑

| 도메인 개념 | 코드 식별자 | 위치(파일 경로) | 책임 (한 줄) |
|------------|------------|----------------|-------------|
| 소환수 특성 분배 (전장 랜덤) | `SummonTraitService.Distribute` | `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitService.cs` | 소환술사 목록 + 특성 풀 → 소환수별 특성 배정 결과 산출 |
| 배정 결과 (소환수 → 특성) | `Dictionary<UnitLoadOutData, SynergyDefinitionData>` (반환값) | 동일 파일 | 분배 결과를 호출자(시너지 시스템)에 넘기는 매핑 |
| 전장 특성 풀 | `traitPool` (`IReadOnlyList<SynergyDefinitionData>`) | 동일 파일 (입력 인자) | 본 전장에서 분배 대상이 되는 특성 시너지 정의 집합. 본 기능은 참조만 |

> 특성 풀의 정적 정의·5개 선정 자체는 본 기능 밖이다. 호출자(`SynergyManager`)가 `SummonTraitRegistry.Traits`(`Assets/Common/Data/Synergies/SummonTraitRegistry.cs`)를 풀로 전달한다.

## 3. 구조 맵 (위치 중심)

무엇이 어디에 있는가. 진입점부터.

- **진입점**: `SummonTraitService.Distribute(summoners, traitPool)` — `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitService.cs`
- **전체 표면**: 본 feature 디렉토리에는 `SummonTraitService` 단일 public 클래스만 있다. 다른 타입·내부 헬퍼 없음.

### 3.1 제공 헬퍼 (다른 코드가 찾아오는 곳)

| 헬퍼 | 위치(파일 경로) | 이럴 때 쓴다 |
|------|----------------|-------------|
| `SummonTraitService.Distribute` | `Assets/Scenes/Battle/Feature/SummonTrait/SummonTraitService.cs` | 소환수 풀에 특성을 균등 랜덤 배정한 매핑이 필요할 때 (전장 시작 초기화) |

### 3.2 의존 관계

- **밖으로 의존**: `Common.Data.Synergies`(`SynergyDefinitionData`), `Common.Data.Summoners.SummonerLoadOuts`(`SummonerLoadOutData`·`SummonPool`), `Common.Data.Units.UnitLoadOuts`(`UnitLoadOutData`), `UnityEngine`(`Random`·`Debug`). 상태이상(`StatusEffect`)·`SynergyController`·`SynergyActivation` 등 시너지 효과 부여 계층에는 **직접 의존하지 않는다** — 본 서비스는 데이터 결정만 책임진다.
- **밖에서 의존**: 시너지 시스템([시너지](./시너지.md))의 `SynergyManager.InitializeAllSynergies` 가 본 서비스를 호출해, 반환된 unit→trait 매핑을 SummonerEffect 매핑과 합산하여 카운트·티어·SE 부여로 이어간다.

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도 (분배만 분리)**: 특성의 "데이터 결정"(누가 어떤 특성을 갖는가)만 순수 서비스로 떼어, 시너지 시스템의 카운트·티어·효과 부여 흐름과 분리했다. 덕분에 호출자는 심상 효과(편성 결정적)와 소환수 특성(랜덤)을 동일한 unit→synergy 매핑 형태로 받아 **이후 흐름을 공통화**할 수 있다 (차이는 "결정 방식"뿐).
- **도메인적 의도 (균등 랜덤)**: 분배 단위가 소환수 *종류*(`UnitLoadOutData`)라, 전장마다 어떤 종류가 어떤 특성을 갖는지가 매판 달라지는 기획 의도(전장 랜덤성)를 반영한다.
- **비자명한 제약/관례**:
  - 특성 풀이 비면 분배 불가로 보고 빈 매핑을 반환한다 (전장이 특성 없이 진행되는 fallback). UR 상 "5개 효과"이지만 풀 크기 자체는 본 서비스가 강제하지 않고 호출자가 전달한 만큼을 분배 대상으로 본다.
  - 본 서비스는 stateless — 런타임 배정 결과의 *보관·라이프사이클*은 호출자(`SynergyManager.summonTraitMap`)가 갖는다. 시너지 문서 §4에 명시된 대로 응집도가 높아지면 SummonTrait 전담 클래스로 분리 예정 (이전 Store/Distributor 분리 회귀 이력 있음).
