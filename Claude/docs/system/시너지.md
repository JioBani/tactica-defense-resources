# 시너지 — 시스템 관점

> 이 문서는 AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실이다. 이 문서는 코드를 재설명하지 않는다 — "인덱스 + α"에 한정한다.
>
> - **분류**: 기능(단순 구현) — 대응 UR 노드: `전장 > 시너지` (하위 잎 심상 효과 / 소환수 특성 / 카운트·UI 포함).

## 1. 한 줄 정의

출전 편성으로 결정되는 심상 효과와 전장마다 랜덤 분배되는 소환수 특성을 동일 흐름으로 다뤄, 배치된 소환수 카운트에 따라 티어를 산정하고 해당 효과(SE)를 부여/해제하는 2차원 시너지 시스템.

## 2. 도메인 ↔ 코드 매핑

| 도메인 개념 | 코드 식별자 | 위치(파일 경로) | 책임 (한 줄) |
|------------|------------|----------------|-------------|
| 시너지 1종의 정의(티어·임계치) | `SynergyDefinitionData` | `Assets/Common/Data/Synergies/` (외부 데이터) | 시너지 정적 정의. 본 기능은 참조 |
| 심상 효과 (소환술사 귀속) | `SynergyType.SummonerEffect` 분기 | `Scripts/SynergyManager.cs` (편성에서 결정적 매핑) | 편성 술사의 SummonerEffect를 소환수에 결정적으로 부여 |
| 소환수 특성 (전장 랜덤) | `SynergyType.SummonTrait` + `SummonTraitService` | `Scripts/SynergyManager.cs`, `Feature/SummonTrait/` | 전장 시작 시 균등 랜덤 분배 |
| 카운트·임계치(티어) | `SynergyActivation` | `Scripts/SynergyActivation.cs` | 유니크 카운트 → 활성 티어 산정, 티어 변경 이벤트 발행 |
| 시너지 효과 적용 행동 | `SynergyController` (추상) | `Scripts/SynergyController.cs` | 배치 변경 구독, 카운트 재계산, SSE 부여/해제 일원 관리 |
| 소환수에 붙는 시너지 효과 | `SynergyStatusEffect` | `Scripts/SynergyStatusEffect.cs` | 시너지 소속 SE (StatusEffect 인프라 상속) |
| 시너지 정보 UI 조회 통로 | `SynergyManager.SynergyActivations` / `SynergyMembers` | `Scripts/SynergyManager.cs` | UI가 현재 시너지 상태·소속 소환수를 조회 |

## 3. 구조 맵 (위치 중심)

진입점부터.

- **초기화 진입점**: `SynergyManager.Start` → `InitializeAllSynergies` — `Scripts/SynergyManager.cs` (편성 읽어 시너지 시스템 자체 구축)
- **시너지별 행동**: `SynergyController` 상속 → `SynergyControllers/` (Bruiser·Arcanist·Freljord·Warmonger·Gunslinger·SummonTrait)
- **생성 라우팅**: `SynergyControllerFactory.Create` (SynergyId switch) — `Scripts/SynergyControllerFactory.cs`
- **카운트·티어 상태**: `SynergyActivation` — `Scripts/SynergyActivation.cs`
- **효과 구현**: `SynergyEffects/` — 시너지별 SSE + 부수 효과(`ArcanistSpellPowerEffect`, `FreljordFrostEffect` 등)
- **SE 베이스**: `SynergyStatusEffect<TContext>` / `TierLinkedStatusEffect<TContext>` + 각 Context — `Scripts/`

| 헬퍼 | 위치(파일 경로) | 이럴 때 쓴다 |

### 3.2 의존 관계

- **밖으로 의존**: 상태이상 프레임워크(`StatusEffect` — [infra](./infra/StatusEffect.md)), `GlobalEventBus`(Defender 배치/판매·라운드 이벤트), `Common.Data.Synergies`(`SynergyDefinitionData`·`SynergyId`·`SynergyType`·`SynergyTier`), `Common.Scripts.Rxs`(`RxValue`), `Feature/Unit/Defenders`·`Summoners`, `Feature/SummonTrait`
- **밖에서 의존**: 시너지 정보 UI(시너지 상태·소속 조회), Defender(스폰 시 `AddSynergy`로 시너지 주입받음)

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도 (Manager ↔ Controller)**: `SynergyManager`는 생성·Dispose·디버그 표시만 하는 얇은 레이어이고, 시너지 비즈니스 로직(카운트·티어·SE 부여)은 `SynergyController`에 일원화돼 있다 (Manager 주석에 명시). Manager를 통한 암묵적 의존을 줄이려는 의도.
- **도메인적 의도 (2차원 통합)**: 심상 효과(편성 결정적)와 소환수 특성(전장 랜덤)이라는 기획서의 2차원 시너지 구조를, `InitializeAllSynergies`에서 unit→synergy 매핑으로 **동일 흐름**으로 합산 처리한다. 둘의 차이는 "데이터 결정 방식"(결정적 vs 랜덤 분배)뿐이고 이후 카운트·티어·효과 부여는 공통.
- **분할 의도 (SSE vs TierLinkedStatusEffect)**: 시너지 소속 정체성을 갖는 효과는 `SynergyStatusEffect`, 시너지의 부수 효과로 버프만 받는 비소속 유닛에는 `TierLinkedStatusEffect`로 분리. 둘 다 `SynergyActivation.ActiveTier`를 구독해 티어 변경/비활성에 자체 반응(자기만료).
- **비자명한 제약/관례**:
  - 카운트는 `UnitDefinitionData.ID` 기준 **유니크**(동일 소환수 중복 미카운트) — 기획서 §5.1.1 반영.
  - `SynergyControllerFactory`의 default 분기가 현재 throw가 아니라 `BruiserSynergyController` fallback (테스트 중 — 주석에 정식화 TODO).
  - SummonTrait 런타임 상태는 잠정적으로 `SynergyManager`가 보유하며, 응집도가 높아지면 전담 클래스로 분리 예정(코드 내 TODO).
