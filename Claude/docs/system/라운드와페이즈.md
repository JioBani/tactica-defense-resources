# 라운드와페이즈 — 시스템 관점

> 이 문서는 AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실이다. 이 문서는 코드를 재설명하지 않는다 — "인덱스 + α"에 한정한다.
>
> - **분류**: 기능(단순 구현) — 대응 UR 노드: `전장 > 라운드 & 페이즈` (정비↔전투 페이즈 교차, 정규/엑스트라 라운드, 전장 승리·패배 조건). 침략자 웨이브 진행(라운드별 스폰 진행·완료 판정)도 이 코드에 포함된다.

## 1. 한 줄 정의

전장 한 판을 정비/전투 페이즈가 교차하는 라운드들의 상태 머신으로 진행시키고, 각 페이즈 전이에 맞춰 라운드별 침략자 스폰·완료·실패 판정을 구동하여 전장 승리·패배 조건까지 도달시키는 인게임 루프의 오케스트레이터.

## 3. 구조 맵 (위치 중심)

진입점부터.

- **루프 진입점**: `RoundManager.StartRound` (→ 정비 페이즈로 상태 머신 시작) — `Assets/Scenes/Battle/Feature/Round/RoundManager.cs`
- **페이즈 상태 머신**: `RoundManager`는 `StateBaseController<PhaseType>` 상속 + 자기 자신을 `IStateListener<PhaseType>`로 등록 — 상태 인프라는 `Common.Scripts.StateBase`
- **페이즈별 부수 행동 구독자**: 같은 `IStateListener<PhaseType>`를 구현해 `RoundManager.Instance`에 등록하는 컴포넌트들
  - `RoundAggressorManager` — 전투/라운드실패 페이즈에 반응 (스폰·디스폰) — `Assets/Scenes/Battle/Feature/Round/RoundAggressorManager.cs`
  - `RoundInfoViewer` — 정비 페이즈에 반응 (프리뷰 표시/해제) — `Assets/Scenes/Battle/Feature/Round/RoundInfoViewer.cs`
- **전이 통지 채널**: 전장 승리/패배·라운드 실패·침략자 전멸은 `GlobalEventBus`로 발행/구독 (`OnBattleWinEventDto`, `OnBattleLoseEventDto`, `OnRoundLoseEventDto`, `RoundAggressorCompletedEventDto`) — `Scenes.Battle.Feature.Events(.RoundEvents)`
- **웨이브(미구현)**: `WaveManager` — `Assets/Scenes/Battle/Feature/Round/Wave/WaveManager.cs` (현재 빈 클래스)

### 3.2 의존 관계

- **밖으로 의존**: 상태 머신 인프라(`Common.Scripts.StateBase` — `StateBaseController`·`IStateListener`), 이벤트 버스(`Common.Scripts.GlobalEventBus`), 리액티브 값(`Common.Scripts.Rxs.RxValue`), 씬 데이터(`Common.Scripts.SceneDataManager` — 선택된 `BattlefieldData`), 라운드 데이터(`Common.Data.Battlefields` — `RoundData`·`SpawnEntry`), 오브젝트 풀(`Common.Scripts.ObjectPool` — `Poolable`), 비동기(`Cysharp UniTask`·`Common.Scripts.UniTaskHandles`), 유닛 생성·소환수 도메인(`Feature/Unit` — `UnitGenerator`·`Defenders.DefenderManager`·`Aggressors.AggressorPreview`·`Units.ActionStates`)
- **밖에서 의존**: 페이즈 전이에 반응해야 하는 모든 전장 컴포넌트(`RoundManager.RegisterListener` 구독자), 라운드 진행도·실패 상태를 표시하는 인게임 UI, 전장 승리/패배 이벤트를 받는 결과/정산 흐름

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도 (루프 vs 부수 행동)**: `RoundManager`는 페이즈 전이(상태 머신)와 전이 시점의 도메인 이벤트 발행만 책임지고, 그 페이즈에서 *무엇이 일어나는가*(침략자 스폰, 프리뷰 표시 등)는 `IStateListener<PhaseType>`를 구현한 별도 컴포넌트로 분산돼 있다. 새 페이즈 반응을 추가할 때 `RoundManager`를 건드리지 않고 리스너를 붙이는 것이 의도된 확장 경로.
- **도메인적 의도 (재도전 vs 진행)**: 라운드 실패 후 정비 페이즈로 돌아갈 때 라운드 인덱스를 증가시키지 않는 "재도전" 분기(`_isRetry`)가 있다 — 같은 라운드를 다시 치르는 도메인 규칙(엑스트라/재도전)을, 인덱스 증가를 한 번 건너뛰는 것으로 표현한다. 정상 진행에서만 `RoundIndex`가 증가한다.
- **도메인적 의도 (3개 종료 분기)**: 전투 종료(`End`)는 마지막 라운드면 전장 승리, 아니면 다음 정비로; 라운드 실패(`RoundLose`)는 임무 실패(`MaxFailCount` 도달)면 전장 패배, 아니면 정비(재도전)로 갈린다. 즉 "라운드 실패"와 "전장 패배"는 별개 도메인 개념이며 누적 실패 임계로 연결된다.
- **비자명한 제약/관례**:
  - 침략자 전멸 판정은 이벤트가 아니라 `Update` 폴링(활성 침략자 존재 여부 감시)으로 이뤄진 뒤 `RoundAggressorCompletedEventDto`로 통지된다 — 라운드 종료 조건이 풀 기반 비활성화에 묶여 있어서.
  - 스폰은 fire-and-forget 예약 + 라운드 공용 취소 토큰(`_roundContext`)으로 일괄 취소되는 구조라, 페이즈를 벗어나면 진행 중 스폰 예약이 모두 무효화된다.
  - `MaxSpawnCountPerEntry = 4` — 한 스폰 엔트리당 최대 등장 수 상한(코드 상수).
  - `WaveManager`는 자리만 잡힌 빈 클래스로, 웨이브 책임은 현재 `RoundData.spawnEntries` + `RoundAggressorManager`가 라운드 단위로 흡수하고 있다(별도 웨이브 추상은 미구현).
