# 대기석 — 시스템 관점

> 이 문서는 **AI 에이전트가 코드 레이어에 맞게 이해하고 빠르게 코드를 찾아가도록** 하는 시스템 관점 컨텍스트 문서다.
> 코드가 단일 진실(single source of truth)이다. 이 문서는 코드를 재설명하지 않는다 — **"인덱스 + α"** 에 한정한다.
>
> - **분류**: 기능 — 대응 UR 노드: `전장 > 소환수 > 배치 & 라이프사이클` (대기석/벤치 관리)
> - **이 문서가 다루는 것**: ① 이 대상이 시스템에서 무엇인가(한 줄) ② 도메인 개념 ↔ 코드(파일 경로)의 매핑 ③ 무엇이 어디에 있는가(구조 맵) ④ 다른 코드가 가져다 쓸 헬퍼 ⑤ 코드만 봐선 모르는 의도(+α).
> - **이 문서가 다루지 않는 것**: 코드의 동작을 풀어쓴 설명서, 알고리즘·자료구조 해설, 메서드 본문 요약. 그건 코드를 보면 된다.

---

## 1. 한 줄 정의

대기석(벤치)은 전장에 배치되지 않은 소환수가 머무는 드롭존 슬롯 집합으로, 소환수를 받아들일지 검사하고 점유 시 해당 유닛의 배치 상태를 대기석으로 전환할 책임을 진다.

## 4. +α — 코드만 봐선 모르는 것

- **분할 의도**: 개별 슬롯 행동(`WaitingArea`)과 전체 목록 조회(`WaitingAreaReferences`)를 분리한 것은, 슬롯은 자기 드롭 규칙만 책임지고 "빈 자리 찾기 / 소환 가능 판정" 같은 전장 단위 질의는 소비처(소환·회수 로직)가 단일 조회 지점에서 처리하도록 하기 위함이다.
- **도메인적 의도**: 대기석은 전장 그리드에 올라가지 않은 소환수의 임시 보관처(벤치)다. 드롭이 정비 페이즈에서만 허용되는 것은 전투 중 배치 변경 금지라는 도메인 규칙을 슬롯 단에서 강제하기 위함이다.
- **비자명한 제약/관례**: 슬롯의 단독 점유(슬롯당 1체) 보장은 대기석 자체 코드가 아니라 베이스 `ExclusiveDropZone2D` 가 담당하며, 점유 교체 시 기존 점유자를 이전 드롭존으로 되돌리는 스왑 동작도 그 베이스에 있다. `WaitingArea.cs` 의 `occupantUnit` 필드 및 `// TODO: 이후 Unit 으로 변경` 주석은 현재 점유 대상을 `Defender` 컴포넌트로 다루고 있으나 추후 `Unit` 추상으로 일반화할 의도를 나타낸다.
